Vue Router 提供了两种路由模式：`hash`模式和`history`模式。这两种模式的主要区别在于它们如何与浏览器的历史记录（即浏览器的后退/前进按钮）交互，以及 URL 的表现形式。

### Hash 模式

- **原理**：Hash 模式通过 URL 的 hash 来模拟一个完整的 URL，于是当 URL 改变时，页面不会重新加载。在 URL 中，hash 的改变不会导致浏览器向服务器发出请求，而是触发`hashchange`事件。Vue Router 通过监听这个事件来实现页面的跳转和组件的渲染。
- **URL 例子**：`http://example.com/#/page1`
- **优点**：由于所有的页面请求实际上都是对同一个页面的请求，因此不需要服务器的特殊配置。这种模式在所有支持 HTML5 History API 的浏览器中都能工作，包括旧浏览器。
- **缺点**：Hash 部分仅在客户端处理，不会发送到服务器，这可能会限制你的服务器端渲染(SSR)策略。此外，对于 SEO 来说，hash URL 可能不如 history 模式那么优雅。

### History 模式

- **原理**：History 模式利用了 HTML5 History Interface 中新增的`pushState()`和`replaceState()`方法。这允许你改变 URL 地址而不重新加载页面。Vue Router 会捕捉到 URL 的变化，并据此更新页面内容，而不会向服务器请求新页面。
- **URL 例子**：`http://example.com/page1`
- **优点**：URL 看起来更“干净”、更标准，对 SEO 友好。用户和搜索引擎都更容易理解。
- **缺点**：服务器需要进行特殊配置。如果用户直接访问一个非根地址（例如，直接访问`http://example.com/page1`），服务器必须配置为返回应用的入口页面（通常是`index.html`），然后由 Vue Router 渲染对应的页面，否则会出现 404 错误。

### 选择哪种模式

- 如果你的网站需要 SEO，并且你可以配置服务器以满足 History 模式的要求，推荐使用`history`模式。
- 如果你无法控制服务器配置，或者在开发过程中需要一个简单的设置，`hash`模式是一个不错的选择。

无论选择哪种模式，Vue Router 都能提供流畅的单页面应用体验。但是，从长远来看，`history`模式由于其更佳的 URL 结构和 SEO 优势，通常是更受欢迎的选择。

在浏览器记录方面，hash 模式和 history 模式处理浏览器的前进和后退按钮、以及用户手动修改 URL 地址栏时的行为有所不同，主要体现在如何与浏览器的历史记录（History API）交互上。

### Hash 模式

- **浏览器记录**：在 hash 模式下，当 URL 的 hash 部分发生变化时（即`#`后面的部分），浏览器会认为是同一个页面在进行导航。因此，每次 hash 变化都会被浏览器记录下来，形成一条独立的历史记录条目。
- **前进/后退按钮**：用户可以使用浏览器的前进和后退按钮来导航这些记录，每次导航时，Vue Router 会根据 URL 的 hash 部分变化来更新页面内容。浏览器会触发`hashchange`事件，Vue Router 侦听这个事件来响应 URL 的变化。
- **手动修改 URL**：用户可以通过直接修改地址栏中的 hash 部分来跳转到应用内的不同视图，浏览器不会重新加载页面，而是触发`hashchange`事件，Vue Router 据此更新页面。

### History 模式

- **浏览器记录**：在 history 模式下，Vue Router 利用 HTML5 的 History API（主要是`pushState`和`replaceState`方法）来改变 URL 而不重新加载页面。这种方式下的 URL 变化也会被浏览器记录为历史记录条目。
- **前进/后退按钮**：用户可以使用浏览器的前进和后退按钮来导航这些历史记录。当这样做时，浏览器不会重新请求页面，而是触发`popstate`事件。Vue Router 监听这个事件，并根据当前的 URL 更新页面内容。
- **手动修改 URL**：用户可以通过直接修改地址栏来尝试访问应用内的不同视图。在 history 模式下，这样做可能会导致服务器返回 404 错误，除非服务器被适当配置来支持单页面应用。即服务器要能够对所有的导航请求返回同一个`index.html`页面，然后让 Vue Router 根据 URL 渲染相应的组件。

### 差异总结

- **对浏览器历史记录的影响**：两种模式都会影响浏览器的历史记录，但 hash 模式通过 URL 的 hash 变化来管理记录，而 history 模式则通过 History API 来管理。
- **页面刷新**：在 hash 模式下，改变 hash 部分不会导致页面刷新；在 history 模式下，如果直接修改地址栏或刷新页面，需要服务器配置支持，否则会出现 404 错误。
- **SEO 和 URL 美观**：history 模式能提供更加友好的 URL，对 SEO 更有利。hash 模式下的 URL 包含`#`，看起来不够美观，且可能对 SEO 不友好。

选择哪种模式主要取决于应用的需求、服务器配置的可能性以及对 SEO 的考虑。
