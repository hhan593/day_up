##### 为什么技术选型要用 vitepress ？ 为什么不直接用 vue？
1、从开源角度来说，如果用 vue，那么就需要一个后台的博客录入系统，那么既然是后台管理系统，那么必然会有 管理员 这一说，这就显得我们的这个开源社区没有那么 open。
2、从运营成本来说，因为 vitepress 的定位就是一个为书写文档方便的框架，内置有路由管理，并且都是以 markdown 的形式，所以只需要运营同学在相应的文件夹下放入对应的 markdown 博客内容即可，其他配置项都不需要改动，极大的降低运营同学的成本。
3、从体验的角度来说，vitepress 打包完的内容是一个完整的 html，也就说打包后的 index.html 里面是有首屏内容的，但是传统的 Vue3 打包完是一个空的 html，只有一个节点 id 为 app 的 div 节点。因此 vitepress 就对 SEO 比较友好，但是 vue3 就没那么友好了，但是 vue 也可以做，通过 SSR 那种去做，但是由于当时没有后端团队的支撑，所以就没做


##### vitepress 的 bug ？
问题：如果在 md 中使用动态传参的组件（项目中对应的banner），当刷新的时候会导致这个 组件 渲染两次
解决方案：在这种动态传参的组件外边加 ClientOnly 组件包一层即可

clientOnly 是只会在客户端去渲染，也就说 clientOnly 不会编译成静态的 HTML，会由 js 执行脚本来生成的，跟传统的 vue3 一致的。
clientOnly 的源码也很简单也就是默认不会展示，会等到 onMounted 之后才去展示



##### 为什么 vitepress 的首屏比较快，并且刷新的时候是没有白屏闪动的,那为什么呢？
因为 vitepress 打包完之后的 index.html 是一个完整的 html，所以首屏比较快，而且刷新的时候不会有白屏闪动，用户体验比较好，而且对 seo 比较友好，但是 传统 vue3 打包完是生成一个空的 html，然后通过 js 脚本动态追加内容进去的，所以首屏没那 么快。

### 你说的对组件的一系列思考是什么？
如何封装组件、粒度和边界如何决定：
- （单一性：单一性要求一个组件具有高内聚，低耦合的特征，它只负责一件事情，不要耦合一些没必要的逻辑，并且尽量不要和其他组件有过于多的双向交互和互相依赖关系。单一性并不代表着不可以引用其他组件，当前组件可能是外层的容器组件，里面包含一些子组件，这样的设计是没问题的。）
- （复用性：在设计组件的时候，一定要考虑组件的复用性或者说是通用性。这是指，当组件封装好后，可以在类似的使用场景中直接调用。这要求我们在设计组件的时候，考虑组件功能的通用性，以及考虑组件入参的合理性。）
- （业务组件：业务组件侧重于 `数据` 和 `业务`的逻辑处理，其中 数据 一般通过接口获取。目前本团队维护的业务组件库，可以使开发人员即来即用，组件内部有完善的功能和接口 数据 处理，将组件引入到项目后可直接实现对应功能。）
- （UI 组件：UI 组件一般也可以称为基础组件，它们经常在多个地方被复用，且不耦合任何的业务功能，例如：AntD 组件库。UI 组件侧重于页面展示效果，大部分 UI 组件具有原子性，一些复杂的 UI 组件可以由基本的 UI 组件构成。一般情况下组件内部的 数据 来源于父组件传递过来的 props。）
（原子组件 ==> 分子组件  ==> 生物组件  ==> 模板组件  ==> 页面）

- 对象使用字面量
- 变量以及方法命名使用camelCase
- 约束 `var` 的使用
- 约束 `for...in` 的使用, 可以使用 `Object.keys().forEach`

- 公共 `utils` 方法请按 jsdoc 注释规范给出注释

#### 你对需求分析的理解：
首先我觉得需求非常重要，没有需求就没有开发，在做需求的时候可能会出现返工（技术评审出问题，导致实现不了了；出现理解的偏差：做远了需要返工），延期（功能太复杂了超出工期评估），伪需求（要砍掉，前期产品原因了对需求分析的不够透彻，本来就是一个为需求，因为产品设计不合理，然后导致开发留下很多坑，有违常识）

需求分部：需求从哪里来？做什么的，要用什么技术栈？如果是重构或者组件库啥子的，可能没有需求的来源方

1、需求调研（比如说扫码用车，必须知道是那部分群体用这个，对其中的那个功能点用的比较多，然后把那个点强化，把用户体验做好）

2、需求设计（有清晰的原型图，高保真，不能很随意的说怎么做怎么做，说到这就顺嘴说一句学校的很多项目就是添加个什么什么，没原型图，导致后期直接返工、还有文档都要清晰，每一块有什么功能，有什么交互都要写清楚，设计的原型图合不合理，风格统一不？以我为例就是在开发页面时就有一个页面的风格颜色不对，然后就去找 UI 去问是故意这样设计的还是疏忽了怎么样的）

3、需求评审（交互评审、技术评审），
  - 交互评审：这部分是产品和研发的同学进行沟通，比如按钮提交后要干些什么东西，如果产品同学不太清楚我们可以根据自己的经验告诉他一般怎么做怎么做，交互很重要，直接决定了我们的实现方式
  - 技术评审：有些功能就是纯技术的，我们要拉上负责人去说清楚这个怎么实现的，防止前期不做技术评审，后期实现不了，然后又打回去对吧，这就很低效了，（比如做一些模块，提前调研这个技术可不可行，做一些准备，这样可以把自己想实现的某一块的方式写进去让大家看，可以避免走坑，比如用那个插件阿，可能同事用过由什么坑什么的，）

4、工期评估（开发）：开发同学做好工期评估，排期，然后发给负责人，负责人看看你评估的合不合理；一般某个模块给开发同学，开发同学会做需求拆分，细节每个模块，比如页面很细节什么，里卖弄有页面有交互，列一个云文档，然后每一块的细节都安排到个人，时长，里面的细节开发、UI优化，逻辑调整、自测，UI走查、联调，这会给产品一个时间，然后测试给一个时间，然后进行提测，
