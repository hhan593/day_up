# 要充分理解 Rust 中数据结构的内存布局

与 Golang 不同的是，Rust 中我们可以完全自由地控制对象的分配。我们需要学会区分 Rust 中数据类型的内存布局，它们在栈上和堆上如何分配的、如何拷贝的。毕竟，堆上内存的分配开销，以及内存拷贝带来的开销，都是很大的。

Rust 常用的数据结构中，例如， `String`，`Vec`，`HashMap`和`Box<Trait>`/`Box<[T]>`的分配都在堆上。一个不好的习惯就是定义 `Vec<Box<T>>`, 这会增加不必要的堆上内存分配次数。

  


Rust 支持 `Copy` 和 `Clone` 两种拷贝方式。简单理解起来，`Copy` 就是按位复制，`Clone` 是 深拷贝。无论对象是在堆上还是栈上，都需要谨慎使用 `Clone`，尽可能地避免深拷贝。

  


# Vec 和 Map 预分配内存

每次 `Vec`、`HashMap` 的扩容都会带来内存分配和内存拷贝，可以通过 `with_capacity` 来优化。

业务中比较常见的例子：

```Rust
fn get_data(ids: &[i64]) -> HashMap<i64, Item> {
    let res = HashMap::with_capacity(ids.len());
    ...
}
```

  


# 避免 `Box<Trait>` 动态分发

动态分发是多态的一种，就是只有在运行时才能确定具体实现。 我们都知道，Golang 中是通过 ` interface{}  `实现动态分发的。那么，如何在 Rust 中实现动态分发？规范的做法是使用`Box<Trait>` ，例如：

```Rust
trait Animal {
    fn speak(&self);
}

struct Dog;
impl Animal for Dog {
    fn speak(&self) {
        println!("Dog: Woof!");
    }
}

struct Cat;
impl Animal for Cat {
    fn speak(&self) {
        println!("Cat: Meow!");
    }
}

fn invoke_speak(animal: &Box<dyn Animal>) {
    animal.speak();
}

fn main() {
    let dog = Box::new(Dog);
    let cat = Box::new(Cat);
    
    invoke_speak(&dog);
    invoke_speak(&cat);
}
```

  


但是我们并不推荐在 Rust 中使用 `Box<Trait>`。 因为动态分发虽然编码起来方便，但是需要额外的运行时开销，而且`Box` 对象是分配在堆上面的。幸运的是，我们可以将 `Box<Trait>`重构为带有泛型的函数。下面`invoke_speak`函数中，`Animal` 的具体类型在编译期可以确定，对象可以分配到栈上面，而且没有额外的运行时的开销。

```Rust

fn invoke_speak<T: Animal + ?Sized>(animal: &T) {
    animal.speak();
}

fn main() {
    let dog = Dog;
    let cat = Cat;
    
    invoke_speak(&dog);
    invoke_speak(&cat);
}
```

  


但是，使用泛型的话，这将丧失灵活性和扩展性。由于泛型约束的存在，还不能像 Golang 中的 `interface{}` 那样对任意对象进行分发。

  


为了解决这个问题，机智的同学可能已经想到了 Rust 的 `enum` 类型。利用 `enum` 枚举所有可能存在的 `Animal` 类型。为了更加优雅一点，再为 `enum` wrapper 实现一个 trait 方法。由于 `enum` 的类型是确定的，在编译期，对象可以也分配到栈上，例如：

```Rust

enum AnimalEnum {
    Dog(Dog),
    Cat(Cat),
}
impl Animal for AnimalEnum {
    fn speak(&self) {
        match self {
            AnimalEnum::Dog(dog) => dog.speak(),
            AnimalEnum::Cat(cat) => cat.speak(),
        }
    }
}

fn invoke_speak(animal: &AnimalEnum) {
    animal.speak();
}
```

  


# 借助编译器优化

Rust 为了安全，会在运行时进行边界检查等，比如下面代码：

```Plain Text
fn do_something_with_array(array: &[u8]) -> u8 {
    array[0] + array[1] + array[2] + array[3] + array[4] + array[5]
}
```

可以通过编译输出 MIR 看到，编译器会给数组索引访问插入断言检查：

```Plain Text
assert(move _9, "index out of bounds: the length is {} but the index is {}", move _8, _7) 
```

有几个数组索引访问就会被插入几次，上面的代码会被插入 6 次，这极大影响性能。

所以我们可以手工插入一次断言检查，就可以帮助消除编译器自动插入的边界检查。

```Plain Text
fn do_something_with_array(array: &[u8]) -> u8 {
    assert!(array.len >= 5);
    array[0] + array[1] + array[2] + array[3] + array[4] + array[5]
}
```

  


# 合理使用 inline 和 cold

函数 inline 的目的是减少函数调用开销，但是会增加代码二进制大小。在 release 模式下，Rust 编译器会根据一个模型来判断函数是否需要内联。但是，rust 编译器编译时无法知道哪些函数会是运行时的热点。

因此，我们可以对热点路径上面的函数，尤其是比较小的函数，手动添加`#[inline]`，来建议编译器内联。

如果函数确定是热点，甚至可以使用 `#[inline(always)]`，通常情况下，编程器一定会内联该函数。

和上面一样，`#[cold]` 则是用来提示编译器，这个函数是冷路径的。编译器就会倾向于不内联该函数，并且调整代码顺序等。例如，我们可以用来标记一些与错误处理相关的函数，或者一些 fallback 逻辑。

  


# 合理谨慎使用 unsafe API

在 rust 标准库中，有一些 API 有对应的的 ` unsafe  `版本，`unsafe` API 一般是去除了校验、边界检查或枚举匹配等操作。

比如 `String::from_utf8` 和 `String::from_utf8_unchecked`，是一对 Safe 和 Unsafe 的方法。

一般情况下，应该使用 `String::from_utf8` 将 `u8`序列转换为合法的字符串，这个方法对 `u8`序列进行了合法 utf8编码的检查。但是这个检查也会有一定开销。

  


如果开发者能确保调用环境的 `u8`序列来源是完全合法的 utf8 编码，那么这个安全检查就完全可以忽略。此时就可以使用 `String::from_utf8_unchecked` 来替换 `String::from_utf8` 用来提升性能。

```Plain Text
pub fn from_utf8(vec: Vec<u8>) -> Result<String, FromUtf8Error> {
    match str::from_utf8(&vec) {
        Ok(..) => Ok(String { vec }),
        Err(e) => Err(FromUtf8Error { bytes: vec, error: e }),
    }
}

pub unsafe fn from_utf8_unchecked(bytes: Vec<u8>) -> String {
    String { vec: bytes }
}
```

一个好的习惯是，使用 `unsafe` API 前，记得添加 assert 断言。这样能帮助我们提早发现问题。

  


# 避免使用 format

在热路径（正常路径）下，`String` 的拼接尽量不要用 `format` 来做，`format` 宏的开销是很高的。比较推荐的方式是直接使用 `String` 手动进行拼接，效率比 `format` 会高很多。

如果需要拼接字符串，用 `String::with_capacity` 分配足够大的容量，然后用 `push_str` 拼接（rust 中的`String` 类似 Golang 中的 `strings.Builder` 和 `string` 合体）。

  


除此之外，在计算不出来最大容量的情况下，还可以使用 TLS 来存储最大 `String capacity`，避免扩容导致的内存分配和拷贝。例如：

```Rust
fn format_key(
    prefix: &str,
    id: i64,
    extra: Option<Arc<HashMap<String, String>>>,
) -> String {
    // 这里在 TLS 里面再存一下最大用到的容量，之后分配的时候按最大用到容量进行分配
    thread_local! {
        static MAX_STRING_CAP: RefCell<usize> = RefCell::new(64);
    }
    MAX_STRING_CAP.with(|cap| {
        let mut s = String::with_capacity(*cap.borrow());
        let mut itoa_buffer = itoa::Buffer::new();

        s.push_str(prefix);
        s.push(':');
        s.push_str(itoa_buffer.format(id));

        format_extra(&mut s, ...);

        if s.len() > *cap.borrow() {
            cap.replace(s.len());
        }
        s
    })
}
```
