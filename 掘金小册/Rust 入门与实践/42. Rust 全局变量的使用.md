在日常编程中，往往需要存储全局的状态或数据，例如全局计数器、配置信息等，这时候就需要使用一种语法来表示全局数据。考虑到使用的全局数据，可能有以下不同使用场景：

1.  静态常量，编译期初始化好，后面一直不变，例如配置信息
1.  静态变量，编译期初始化，但是后面会改变，例如全局 ID 生成
1.  动态变量，需要运行时初始化后，而后续仍会不断变更，例如全局锁。

如果再考虑是否会在多线程环境下使用，如何合适地使用全局变量会更得更加复杂。

  


# 用 const 表示静态常量

对于编程中的静态常量，例如配置信息、默认值等，可以直接使用 `const` 关键字来表示。 使用 `const` 的好处是，常量是直接在编译时，写入了二进制中，不会有任何运行时的初始化开销。另外，`const` 常量是无法取地址和引用的。

  


另外，Rust 中常量还可以是关联的，常量可以关联某一类型，包括 `trait`。在对常量初始化时，如果遇到初始化逻辑比较复杂，但是又在编译期可以完成的，可以定义一个 `const` 函数。

```C
const MAX_ID: usize =  usize::MAX / 2;

trait UserTarit {
    const CONST_NO_DEFAULT: i32; // 不用初始化
    const CONST_WITH_DEFAULT: i32 = 99;
}

struct User { }

impl User {
    const fn get_max_id() -> usize { // const 函数，编译器会直接优化掉
        MAX_ID
    }
    const MAX_ID : usize = User::get_max_id(); // 初始化 const 常量可以用 const 函数
}

impl UserTarit for User {
    const CONST_NO_DEFAULT: i32 = 1;
}

fn main() {
   println!("free constant: {}", MAX_ID);
   println!("associated constant: {}", User::MAX_ID);
   println!("trait const no default: {}", User::CONST_NO_DEFAULT);
   println!("trait const with default: {}", User::CONST_WITH_DEFAULT);
}
```

  


# 用 static 表示静态变量

有时候，我们需要的是静态变量，并且需要在运行时初始化，甚至改变它的值。这时候就需要使用静态变量。Rust 中可以使用 static 关键字，表示一个静态变量。

  


与常量项不同，static 变量必须存储于内存中的一个精确的地址，并且是全局共享的。因此，static 变量的生成周期毫无疑问是 `'static`。同时，` static  `全局变量可以供所有线程访问。如果需要定义一个可以修改的全局变量，一个最直接的做法就是使用`static mut`。但是，在多线程下使用时，不同线程对同一变量进行读写，这时候可能会导致并发问题。

  


因此，Rust 为了尽可能限制这一问题，要求必须使用`unsafe`语句块才能修改`static mut` 变量。

```C
static mut COUNTER: usize = 0;

fn main() {
    unsafe { // 必须使用unsafe 才能通过编译，在并发下不安全
        COUNTER += 1;
        println!("Counter value: {}", COUNTER);
    }
}
```

上面的 `static` 变量使用时不得不引入了`unsafe`，在实际的多线程环境中会有并发问题。如果需要解决并发安全问题，可以将 `static` 变量定义为 `Atomic` 原子类型。 例如，我们可以使用`static` 来完成一个 id 生成器，使用`Atomic`来保证全局变量的并发安全。

  


当然，使用 `Atomic`后，每次读写操作都会有些微的性能损耗。

```C

#![allow(unused)]
fn main() {
use std::sync::atomic::{Ordering, AtomicUsize};

struct Factory{
    factory_id: usize,
}

// 编译期完成了初始化，起始id 是 0. 
static GLOBAL_ID_COUNTER: AtomicUsize = AtomicUsize::new(0);
const MAX_ID: usize = usize::MAX / 2;

fn generate_id()->usize{
    // 检查两次溢出，否则直接加一可能导致溢出
    let current_val = GLOBAL_ID_COUNTER.load(Ordering::Relaxed);
    if current_val > MAX_ID{
        panic!("Factory ids overflowed");
    }
    let next_id = GLOBAL_ID_COUNTER.fetch_add(1, Ordering::Relaxed);
    if next_id > MAX_ID{
        panic!("Factory ids overflowed");
    }
    next_id
}

impl Factory{
    fn new()->Self{
        Self{
            factory_id: generate_id()
        }
    }
}
}
```

# 使用 LazyLock 对全局变量初始化

很多时候，我们想使用全局变量，但是无法在编译期对其进行初始化。这时候只使用 `static` 关键字无法达到这一目的。而且，对于 static 全局变量，我们往往只需要初始化一次。

  


幸运的是，Rust 标准库推出了 `std``::``cell``::``LazyCell`和 `std::sync::LazyLock` 来保证对变量的单次初始化，并且将初始化逻辑延迟到第一次访问时进行，非常适合 `static`全局变量。（**不过目前的 Rust stable 版本还不支持这两个** **API**)。

```C
#![feature(lazy_cell)]

use std::collections::HashMap;

use std::sync::LazyLock;

static HASHMAP: LazyLock<HashMap<i32, String>> = LazyLock::new(|| {
    println!("initializing");
    let mut m = HashMap::new();
    m.insert(13, "Spica".to_string());
    m.insert(74, "Hoyten".to_string());
    m
});

fn main() {
    println!("ready");
    std::thread::spawn(|| {
        println!("{:?}", HASHMAP.get(&13));
    }).join().unwrap();
    println!("{:?}", HASHMAP.get(&74));

    // Prints:
    //   ready
    //   initializing
    //   Some("Spica")
    //   Some("Hoyten")
}
```


# 使用 Lazy_static

标准库中的 `LazyLock` 还没有 stable，不用担心，我们也可以使用 Rust 社区提供的 `lazy_static`，来实现对全局变量实现运行时的懒加载。

`lazy_static`是一个功能非常强大的宏，用于懒初始化静态变量。不像之前的静态变量都是在编译期初始化的，而是在运行期初始化。使用`lazy_static` 后，对于初始化逻辑比较重的全局变量，可以延迟加载，而且只用执行一次。

例如，我们可以用它实现一个全局的 hashmap 缓存，会在首次使用 hashmap 时才对其进行初始化。

```Rust
use lazy_static::lazy_static;
use std::collections::HashMap;

lazy_static! {
    static ref HASHMAP: HashMap<u32, &'static str> = {
        let mut m = HashMap::new();
        m.insert(0, "foo");
        m.insert(1, "bar");
        m.insert(2, "baz");
        m
    };
}

fn main() {
    // 首次访问`HASHMAP`的同时对其进行初始化
    println!("The entry for `0` is "{}".", HASHMAP.get(&0).unwrap());

    // 后续的访问仅仅获取值，再不会进行任何初始化操作
    println!("The entry for `1` is "{}".", HASHMAP.get(&1).unwrap());
}
```

`lazy_static`之所以能做到懒加载，是因为其内部实现用了一个底层的并发原语`std::sync::Once`，在每次访问该变量时，程序都会执行一次原子指令用于确认静态变量的初始化是否完成。对于`lazy_static`宏，匹配的是`static ref`，所以定义的静态变量都是不可变引用。

  


# 使用 `once_cell::sync::Lazy`

如果要初始化一个全局变量，也可以将全局变量定义为`once_cell``::``sync``::``Lazy` 或 `unsync::OnceCell` 。两种 cell 都能够确保变量只被赋值一次， 非常适合在 static 全局变量中使用。其中，`sync`版本是线程安全的。

```C
use std::collections::HashMap;

use once_cell::sync::Lazy;

static HASHMAP: Lazy<HashMap<i32, String>> = Lazy::new(|| {
    println!("initializing");
    let mut m = HashMap::new();
    m.insert(13, "Spica".to_string());
    m.insert(74, "Hoyten".to_string());
    m
});

fn main() {
    println!("ready");
    std::thread::spawn(|| {
        println!("{:?}", HASHMAP.get(&13));
    }).join().unwrap();
    println!("{:?}", HASHMAP.get(&74));

    // Prints:
    //   ready
    //   initializing
    //   Some("Spica")
    //   Some("Hoyten")
}
```

` once_cell  `其内部实现是基于 `lazy_static` 与lazy_cell 和 标准库的 `std::sync::Once` ，严格地来说，` once_cell  `是简化和统一了 API。因此，`once_cell`不需要宏操作，接口更加友好。

  


# 总结

对于不变的，初始化后只读的全局变量。如果是编译期常量，就使用 `const`。对于全局变量，能够在编译时初始化的，可以使用 `static`，如果必须在运行时懒加载的，甚至全局变量初始化逻辑比较重的，可以使用 rust 标准库提供的 LazyLock 接口 (暂时只有rust nightly 版本有）。此外，也可以使用现在非常流行的，社区提供的 `lazy_static`。另外还有社区的 `once_cell` 也是一个选项，使用起来会更加灵活方便，不需要宏定义。

  


如果是使用可变的全局变量，一定需要注意多线程环境下的并发问题。最好不要使用 `static mut`, 那样会引入`unsafe` 逻辑，这时候需要自行保证是线程安全的。这时候可以将全局变量类型定义为 `atomic` 或 `mutex` 等。如果确定是全局变量是可变的，但是各个线程又互不影响的，其实可以考虑使用 ` thread_local  `变量。
