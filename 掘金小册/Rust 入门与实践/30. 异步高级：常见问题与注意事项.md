# poll_like api 和 Future 互转

许多底层库提供 Poll 风格的接口，例如 Tokio 的 AsyncRead/AsyncWrite，能够提供 poll 风格接口的本质在于，它内部保存了状态，所以只需要外部去轮询是否可继续 or 完成。

而另一些库提供了 async 风格的接口，async 风格接口的本质是将状态管理任务丢给了用户。async fn 返回一个 Future，Future 同样暴露 poll 接口。

结合上述这两种风格的区别，基于 poll 接口我们可以很方便地暴露 async 接口，因为前者不需要状态，后者提供了保存状态的能力，但我们用不到。使用 `poll_fn` 可以方便地做这种转换。

但是如果将 async 接口转换为 poll_xxx 接口就比较困难了，这时我们需要引入一个中间量保存状态。而鉴于 async fn 返回的 Future 类型是匿名的，所以往往需要 BoxFuture。当这个转换处于这路径时，我们可以借助 [reusable_box_future](https://docs.rs/reusable-box-future) 优化其内存使用，提升性能。

  


# Send + Sync + 'static

Tokio 会将任务在 worker thread 之间调度，所以需要 task 是 Send + Sync 的；如果需要 spawn task，那么 task 还需要是 'static 的。如果你使用别的不做跨线程调度的 Runtime（例如 Monoio），则无 Send + Sync 的约束。

特别地，标准库的 MutexGuard 并没有实现 Send/Sync，所以将其捕获的 task 会因为不满足该约束而无法在 Tokio 中执行 —— 使用大括号限定 Guard 的作用域或异步 Mutex 可以解决这个问题。

  


# 阻塞式 syscall

在异步代码中，我们不能手动执行可能阻塞的 syscall 或其他同步调用（例如 block_on），否则可能会导致 worker 线程卡死。

  


# 手动返回 Pending 导致卡住

在手动实现 Future 的时候，当任务无法继续时就返回 Pending，看起来再自然不过了——但这并不永远正确，你可能会面临 await 永远等待的问题。

正确做法是，如果你的组件是事件源，则应当记录 Waker 并在适当的时刻唤醒；否则只有当该 Pending 是其他 poll 返回的时候才对外返回 Pending。
