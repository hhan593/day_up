Rust 可以被认为是两种编程语言的组合：Safe Rust 和 Unsafe Rust，基于 Unsafe Rust 所构建出“安全”的 Rust 是 Rust 区别于其它编程语言最重要的特性之一。如果开发者所做的只是编写 Safe Rust，那么将永远不必担心类型安全或内存安全。大多数时候基于标准库与第三方库之上的 safe Rust 已经能满足绝大部分的使用需求，但有时开发者必须使用 Unsafe Rust 以解决以下场景的问题：

1.  高效的基础数据结构与运算；
1.  与“外部”（C 的链接库、操作系统）交互；
1.  其它避开编译器的拘束提供限定情景下安全的功能；

  


Unsafe Rust 与 safe Rust 完全一样，具有所有相同的规则和语义。它只是允许你做一些绝对不安全的事情。这种分离的价值在于，开发者获得了像 C 一样的实现细节控制能力也不会丢失像其它安全语言（例如 Java）一样的安全性保证。

  


## Unsafe 与 Safe 的交互

Safe Rust 和 Unsafe Rust 之间的分离是通过 `unsafe` 关键字来控制的，它充当一个到另一个的接口。这就是为什么我们可以说 Safe Rust 是一种安全的语言：所有不安全的部分都只保留在 `unsafe` 边界后面。如果您愿意，您甚至可以将代码投入 `#![forbid(unsafe_code)]` 到您的代码库中以静态地保证您只编写 Safe Rust。

```Rust
// unsafe function
unsafe fn ufoo() {}

// unsafe block in safe function
fn foo() {
    unsafe {
        ufoo()
    }
}

// isoloate unsafe rust with safe rust
fn main() {
    foo()
}
```

  


## Unsafe 的能力与约束

相较于 Safe，Unsafe Rust 额外具备以下能力：

-   解引用原始指针；
-   调用 unsafe 函数；
-   声明实现 unsafe trait；
-   修改静态变量；
-   使用 `union` 类型；

  


这些操作被降级为不安全的原因是滥用这些东西中的任何一个都会导致可怕的未定义行为。调用未定义行为赋予编译器对你的程序做任意坏事的全部权利，所以绝对不应该产生任何未定义行为。与 C 不同，未定义行为在 Rust 中的范围非常有限。所有核心语言关心的是防止以下事情：

-   解引用悬挂指针或未内存对齐的指针；

-   突破 borrow checker 的规则；

-   使用错误的 ABI 调用函数或 unwind 一个具有错误 unwind ABI 的函数；

-   数据竞争；

-   执行当前平台（操作系统或指令集）不支持的功能；

-   产生无效值（单独或作为复合类型的字段，例如`enum`/ `struct` / `array` / `tuple`）：

    -   `bool` 不是 0 或 1；

    -   非法的 `enum` [判别式](https://doc.rust-lang.org/reference/items/enumerations.html#discriminants)；

    -   空 `fn`指针；

    -   `char` 类型超出 0x0..0xD7FF 和 0xE000..0x10FFFF 范围；

    -   `!` 类型（此类型没有任何合法的值）；

    -   指向未初始化内存的指针类型；

    -   `Box` 悬空、指向未内存对齐内容或其它非法值；

    -   指针的元信息不合法：

        -   Trait object 未实现其所宣称实现的 trait；
        -   `slice` 长度与实际长度不符；

    -   其它具有自定义 null 值类型的 null 值（如 `NonNull` 与 `MaybeUninit`）；

  


不安全的函数和特征需要以注释或文档声明该功能所适用的情景而超出情景适用则会产生未定义行为。例如，分配器 API 声明释放未分配的内存是未定义的行为。

## 使用 Unsafe

如果开发者尝试完全使用 Safe Rust 实现双向链表：

```Rust
use std::rc::Rc;
use std::cell::RefCell;

pub struct List<T> {
    head: Link<T>,
    tail: Link<T>,
}

type Link<T> = Option<Rc<RefCell<Node<T>>>>;

struct Node<T> {
    elem: T,
    next: Link<T>,
    prev: Link<T>,
}

impl<T> Node<T> {
    fn new(elem: T) -> Rc<RefCell<Self>> {
        Rc::new(RefCell::new(Node {
            elem: elem,
            prev: None,
            next: None,
        }))
    }
}

impl<T> List<T> {
    pub fn new() -> Self {
        List { head: None, tail: None }
    }

    pub fn push_front(&mut self, elem: T) {
        let new_head = Node::new(elem);
        match self.head.take() {
            Some(old_head) => {
                old_head.borrow_mut().prev = Some(new_head.clone());
                new_head.borrow_mut().next = Some(old_head);
                self.head = Some(new_head);
            }
            None => {
                self.tail = Some(new_head.clone());
                self.head = Some(new_head);
            }
        }
    }

    pub fn push_back(&mut self, elem: T) {
        let new_tail = Node::new(elem);
        match self.tail.take() {
            Some(old_tail) => {
                old_tail.borrow_mut().next = Some(new_tail.clone());
                new_tail.borrow_mut().prev = Some(old_tail);
                self.tail = Some(new_tail);
            }
            None => {
                self.head = Some(new_tail.clone());
                self.tail = Some(new_tail);
            }
        }
    }
}
```

  


开发者将面临不得不使用诸多具有运行时开销的类型例如 `Rc`、`RefCell` 以绕过 Rust 的 borrow checker 无法证明循环引用安全性的问题。对于双向列表这样的基础数据类型来说无疑这是十分低效的，此时将不得不使用 Unsafe Rust 来实现更传统的内存布局以期其能与 C 一样高效工作：

```Rust
use std::marker::PhantomData;
use std::ptr::NonNull;

pub struct LinkedList<T> {
    front: Link<T>,
    back: Link<T>,
    len: usize,
    _boo: PhantomData<T>,
}

type Link<T> = Option<NonNull<Node<T>>>;

struct Node<T> {
    front: Link<T>,
    back: Link<T>,
    elem: T,
}


impl<T> LinkedList<T> {
    pub fn new() -> Self {
        Self {
            front: None,
            back: None,
            len: 0,
            _boo: PhantomData,
        }
    }

    pub fn push_front(&mut self, elem: T) {
        // SAFETY: it's a linked-list, what do you want?
        unsafe {
            let new = NonNull::new_unchecked(Box::into_raw(Box::new(Node {
                front: None,
                back: None,
                elem,
            })));
            if let Some(old) = self.front {
                // Put the new front before the old one
                (*old.as_ptr()).front = Some(new);
                (*new.as_ptr()).back = Some(old);
            } else {
                // If there's no front, then we're the empty list and need
                // to set the back too.
                self.back = Some(new);
            }
            // These things always happen!
            self.front = Some(new);
            self.len += 1;
        }
    }

    pub fn push_back(&mut self, elem: T) {
        // SAFETY: it's a linked-list, what do you want?
        unsafe {
            let new = NonNull::new_unchecked(Box::into_raw(Box::new(Node {
                back: None,
                front: None,
                elem,
            })));
            if let Some(old) = self.back {
                // Put the new back before the old one
                (*old.as_ptr()).back = Some(new);
                (*new.as_ptr()).front = Some(old);
            } else {
                // If there's no back, then we're the empty list and need
                // to set the front too.
                self.front = Some(new);
            }
            // These things always happen!
            self.back = Some(new);
            self.len += 1;
        }
    }
}
```
