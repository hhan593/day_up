## NewType

因为 "orphan rules" 的限制，以及可能一些其他需要。我们会通过 NewType 这种形式来解决问题。

比如我们的业务中使用一个 i32 来表示 Business Id 这个语义，但是如果直接使用 i32 那么很不方便扩展。因为我们不能给i32 实现一些我们想要自定义的 method

那么可以使用`struct BusinessId(i32);` 这种方式来产生一个`BusinessId`的类型。

我们还可以根据自己的需求增加 impl

  


比如

```Rust
impl BusinessId {
    const A = Self(0);
    const B = Self(1);
    
    fn belongs_xxx(self) -> bool {
        todo!();
    }

}
```

这样写可以很方便我们进行扩展，并且也能体验到类型系统约束的便利。

使用方可以不用去关心这个BusinessId内部的数据结构是什么，而开发者以后维护时也可以放心地对内部数据结构进行更改。

  


## Deref

`Deref` 是一个非常特殊的Trait。如果使用得当可以提供非常多的便利。

如果类型 `T` 实现了 `Deref<Target = U>`。 并且x是一个类型为`T`的值，那么*x 其实等价于 *Deref::deref(&x)

除此之外 Deref 还有一些其他的便利之处

### Methods Resolution

比如我们现在有一个自己的结构`MyString` 我们想要`MyString`可以拥有所有在`str`上就有的方法。比如`find`, `spilit`。

  


```Rust
pub enum MyString {
    Static(&'static str),
    Arc(Arc<str>),
}
```

那么我们就可以通过 Deref 来实现。因为 Rust 编译器在查找类型的方法的时候，会进行 Auto Deref，在 Deref chain 上一层一层查找，知道找到同名的方法。那么只要`MyString` 实现了 `Deref<Target = str>` 那么当我们调用 `find` 的时候就可以找到`str` 的 `find` 方法了。

```Rust
impl std::ops::Deref for MyString {
    type Target = str;
    fn deref(&self) -> &Self::Target {
        match self {
            Self::Static(s) => &s,
            Self::Arc(s) => &s,
        }
    }
}
```

  


### Coerce

Deref 还会对 Coerce 产生一定影响。

如果类型 `T` 实现了`Deref<Target = U>`

那么`&T` 可以被 coerce 到 `&U`

比如存在这样的一个函数

```Rust
fn test(a: &str) {
}


test(&format!("hello world"));
```

`&format!("hello world")` 这个表达式的类型是`&String` 而因为 `String` 实现了 `Deref<Target = str>` 所以 `&String` 可以被 coerce 到 `&str`

  


## DerefMut

Deref 是仅限在不可变的上下文中使用的。

如果`T` 仅实现了 `Deref<Target = U>`，且存在`x`的类型是`&T`。我们是不能通过`*x = other` 这种形势进行赋值的。

那么为了实现赋值这种需求，我们就还需要再实现一下`DerefMut`。

并且与 Deref 一致， DerefMut 也会有一些 Methods Resolution 和 Coerce 上的一些特性，不过是适用于 Mut Refence 情况下的。

  


  


  


## 运算符重载

在 Rust 中可以通过 impl 特定的 Trait 来实现运算符重载，其实上面的 Deref 就是运算符的一种。

比如通过 PartialEq,Eq 来完成对 "==" 的重载，通过 PartialOrd,Ord 来完成对 ">" 的重载。

  


## Drop

Rust 提供了 Drop 这个 trait 可以用于实现一些在析构时执行的逻辑

当一个结构体被Drop的时候，会自动运行该结构所有字段的Drop，比如下面这个例子。

```Rust
struct D1 {}

struct D2 {
    d: D1,
}

impl std::ops::Drop for D2 {
    fn drop(&mut self) {
        println!("drop d2")
    }
}

impl std::ops::Drop for D1 {
    fn drop(&mut self) {
        println!("drop d1")
    }
}

fn main() {
    let d = D2 { d: D1 {} };
}

// output:
// drop d2
// drop d1
```

  


### Drop 的顺序

如果在一个函数中定义了多个变量，那么这些变量的 Drop 顺序是怎样的呢？

```Rust
struct D1 {}

struct D2 {}

impl std::ops::Drop for D2 {
    fn drop(&mut self) {
        println!("drop d2")
    }
}

impl std::ops::Drop for D1 {
    fn drop(&mut self) {
        println!("drop d1")
    }
}

fn main() {
    let d1 = D1 {};
    let d2 = D2 {};
}

// output:
// drop d2
// drop d1
```

与结构体中drop的顺序不同，变量的drop顺序是按照声明顺序倒序的，后声明的变量会被先被执行Drop。

  


### Drop Check

如果一个结构体中存在自引用，那么drop的时候会发生什么呢？

比如下面这种情况

```Rust
struct Inspector<'a>(&'a u8);

impl<'a> Drop for Inspector<'a> {
    fn drop(&mut self) {
        println!("I was only {} days from retirement!", self.0);
    }
}

struct World<'a> {
    inspector: Option<Inspector<'a>>,
    days: Box<u8>,
}

fn main() {
    let mut world = World {
        inspector: None,
        days: Box::new(1),
    };
    world.inspector = Some(Inspector(&world.days));
}
```

在这里 `world.inspector` 这个字段其实引用了 `world.days` 的内容。在world被drop的时候，如果`world.days` 先被 Drop 了， 那么执行 `world.inspector` 的 Drop 时，就会访问到一个 dangling pointer。

那么要如何避免掉这种危险的事情了，事实上编译器会进行 drop check。编译器会要求一个类型被 drop 时，其所有的泛型参数（包括lifetime）都需要 outlives 自身。

因此在这段代码中，会要求 `&world.days` outlives `world.inspector`，然而这段代码并不满足这条约束，因此编译器会报错。

虽然理论上这里会按照字段的顺序进行drop，即先 drop `world.inspector` 然后再 drop `world.days`。但是目前编译器似乎仅通过对实现了 drop 的类型的泛型参数进行 lifetime region 上的检查，并没有去考虑字段 drop 顺序。所以不管如何调整 `World` 这个类型的字段顺序，这段代码都不会通过编译
