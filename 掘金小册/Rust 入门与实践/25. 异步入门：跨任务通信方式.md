我们在异步编程中往往很难避免跨任务的通信，本节主要介绍常用的跨任务的通信组件。例如，对于一个日志组件，我们可能会启动后台 task，接收数据并汇总，最后定时提交到远程服务器；在需要打印日志时只需要将消息发送到 channel 中即可。

  


# Channel

类似 Golang 提供的 channel，tokio 提供了很多不同类型的 channel：

-   `mpsc`: 多生产者、单消费者的 `Channel`，能够发送多个消息
-   `oneshot` 单生产者、单消费者的 `Channel`，只能发送一次消息
-   `broadcast` 多生产者、多消费者，能够发送多个信息，每个消费者都能收到所有信息
-   `watch` 单生产者、多消费者，能够发送多个信息，但不会保存历史信息，消费者只能收到最新的信息

如果需要多生产者、多消费者的 `Channel`但希望每个信息只被一个消费者收到，可以使用 [async-channel](https://docs.rs/async-channel/) 包。

  


这种异步 channel 允许用户做异步等待，如等待接收一条消息，或等待有空余 buffer 存储发送的消息，或等待 channel 被关闭。

```Rust
use std::time::Duration;

use tokio::sync::oneshot::{
    channel as oneshot, Receiver as OneshotReceiver, Sender as OneshotSender,
};

#[tokio::main]
async fn main() {
    let guard = Guard::new();
    tokio::time::sleep(Duration::from_secs(5)).await;
    drop(guard);
    tokio::time::sleep(Duration::from_secs(5)).await;
    println!("main quit");
}

struct Guard {
    _rx: OneshotReceiver<()>,
}

impl Guard {
    fn new() -> Self {
        let (tx, _rx) = oneshot();
        tokio::spawn(background_task(tx));
        Self { _rx }
    }
}

async fn background_task(mut quit: OneshotSender<()>) {
    let mut interval = tokio::time::interval(Duration::from_secs(1));
    loop {
        tokio::select! {
            _ = quit.closed() => {
                break;
            },
            _ = interval.tick() => {
                println!("interval ticked");
            }
        }
    }
    println!("background task notified exit");
}

// Output:
// interval ticked
// interval ticked
// interval ticked
// interval ticked
// interval ticked
// interval ticked
// background task notified exit
// main quit
```

  


# Mutex

通常锁分为同步锁与异步锁两种。

标准库或第三方库（如 parking_lot）提供的是同步锁，等待时会 hang 住线程；tokio 提供的是异步锁，在等待时会让出线程控制权，不会卡住其他 task 执行。

这两种锁各有各的用途，在纯同步程序中我们使用标准库的锁即可；在异步程序中，如果我们预期竞争极弱，且不会跨 await 点等待，则可以使用同步锁；否则才使用异步锁——异步锁一定会导致 task 切换，如果竞争弱则可能不如同步锁性能好。

常见误区：同步锁的 Guard（如 MutexGuard）没有实现 Send/Sync，如果被误捕获到 Async 状态机内（即便已经在 await 之前 drop），则会因为 tokio 约束 task Send/Sync 而编译报错。这个问题的原因是 Async 状态机捕获是根据作用域的，drop 并不会提前结束变量作用域。解决方法是使用额外的大括号将 Guard 限定在 await 点之前。

```Rust
use std::sync::{Arc, Mutex};

#[tokio::main]
async fn main() {
    let global_counter = Arc::new(Mutex::new(0));
    tokio::spawn(add_counter_and_submit(global_counter));
}

async fn add_counter_and_submit(counter: Arc<Mutex<usize>>) {
 let  mut locked = counter. lock (). unwrap ();
*locked += 1 ;
 drop (locked);
    submit_something (). await ;
}

async fn submit_something() {}

// error: future cannot be sent between threads safely
```

上述例子会因为在 await 点持有了 Guard，Guard 没有实现 Send，而 spawn 要求实现 Send 而无法通过编译。

有两种解决方式，第一种是将锁换成异步锁，第二种是使用作用于限定 Guard 不被捕获到状态机内。

```Rust
use std::sync::Arc;
use tokio::sync::Mutex;

#[tokio::main]
async fn main() {
    let global_counter = Arc::new(Mutex::new(0));
    tokio::spawn(add_counter_and_submit(global_counter));
}

async fn add_counter_and_submit(counter: Arc<Mutex<usize>>) {
    let mut locked = counter.lock() . await;
    *locked += 1;
    drop(locked);
    submit_something().await;
}

async fn submit_something() {}
```

```Rust
use std::sync::{Arc, Mutex};

#[tokio::main]
async fn main() {
    let global_counter = Arc::new(Mutex::new(0));
    tokio::spawn(add_counter_and_submit(global_counter));
}

async fn add_counter_and_submit(counter: Arc<Mutex<usize>>) {
{
 let  mut locked = counter. lock (). unwrap ();
*locked += 1 ;
}
    submit_something().await;
}

async fn submit_something() {}
```

  


# 其他异步通信结构

Tokio 还提供了一些其他异步通信结构，可以参考其文档：[tokio::sync - Rust](https://docs.rs/tokio/latest/tokio/sync/index.html#structs)

其中比较常用的比如 Semaphore —— 该结构提供类似 unix 信号量的能力。

假设我们实现一个图片压缩服务，正常情况下我们接收到 Request 后即 spawn task 处理它，但这么实现可能会导致在请求量突增时 OOM，我们必须有一种手段能限定同时处理的任务。

借助 Semaphore， 我们可以在启动时配置 Semaphore 的初始总量，并在每个 Task 开始处理图片前获取 Semaphore 的一个 Permit——如果没有则该 task 会陷入等待，直到有任务 Drop 了 Permit。
