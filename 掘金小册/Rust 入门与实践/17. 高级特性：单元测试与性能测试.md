# 导读

在软件开发过程中，单元测试和性能测试是非常重要的环节。本文将讲解在Rust中进行单元测试和性能测试的方法与实践，包括如何编写，组织和运行单元测试，以及如何使用 criterion 进行性能测试。通过学习本文，你将对如何在rust项目中编写单测和性能测试有一个基本的了解，并可以开始实践。

  


# 单元测试

单元测试是软件开发中的一种测试方法，用于验证代码中单独的函数或组件的功能是否符合预期。

单元测试的目的是确保代码在每个独立的部分都按照预期工作。编写单元测试可以提高代码质量，便于后期维护和扩展。

  


## 初步了解 Rust 的单元测试

在Rust中进行单元测试非常简单。通过在函数前面加上 `#[test]` 属性，Rust编译器会将该函数识别为测试函数。

在测试函数中，可以使用 `assert!` 、`assert_eq!` 等宏来进行断言检查。以下是一个简单的单元测试示例：

```Rust
  pub fn add(a: i32, b: i32) -> i32 {
      a + b
  }

  #[cfg(test)]
  mod tests {
      use super::*;

      #[test]
      fn test_add_positive_numbers() {
          assert_eq!(add(1, 2), 3);
      }

      #[test]
      fn test_add_negative_numbers() {
          assert_eq!(add(-1, -2), -3);
      }
  }
```

通过执行 `cargo test`命令运行单元测试。`cargo test` 默认会运行你项目中所有的单元测试，如果只想运行其中特定的单元测试，可以通过相应的 `module path` 来指定需要运行的单测函数。

运行输出结果如下：

```Plain Text
➜  my-lib git:(master) ✗ cargo test
    Finished test [unoptimized + debuginfo] target(s) in 0.08s
     Running unittests src/lib.rs (target/debug/deps/my_lib-476a6fe695d7ecde)

running 2 tests
test tests::test_add_negative_numbers ... ok
test tests::test_add_positive_numbers ... ok

test result: ok. 2 passed; 0 failed; 0 ignored; 0 measured; 0 filtered out; finished in 0.00s

   Doc-tests my-lib

running 0 tests

test result: ok. 0 passed; 0 failed; 0 ignored; 0 measured; 0 filtered out; finished in 0.00s
```

  


注意，`cargo test` 在测试运行通过的情况下不会打印代码里输出到 stderr 和stdout 的内容，可以通过加 --nocapture 来让其打印这些内容，更多 `cargo test`的参数可以通过执行 `cargo help test` 获得。

  


## 单元测试的组织

Rust允许使用模块来组织测试。你可以在测试模块中创建子模块，并在子模块中编写测试函数。这有助于将相关的测试函数组织在一起，使得项目结构更加清晰。例如在上面的例子中，我们把单元测试的代码全都放到了 tests 这个模块下，通过 `#[cfg(test)]` 这个 atrribute 表示这是一个测试模块。然后在这个模块下编写了相应的单元测试函数。

  


## 使用 tokio 的单元测试

当你的项目使用 tokio作为异步运行时的时候，经常会设计到异步代码的单元测试，为了方便地测试异步代码，Tokio提供了一个`tokio::test`属性宏，用于简化异步测试函数的创建。下面是一个简单的例子：

```Rust
// src/lib.rs
pub async fn async_add(a: i32, b: i32) -> i32 {
    sleep(Duration::from_millis(50)).await;
    a + b
}

#[cfg(test)]
mod tests {
    use super::*;
    
    #[tokio::test]
    async fn test_async_add() {
        let result = async_add(1, 2).await;

        assert_eq!(result, 3, "1 + 2 should equal 3");
    }
}
```

在这个例子中，我们创建了一个简单的异步函数`async_task`，它会异步等待100毫秒。另外，我们还创建了一个`async_add`函数，它会异步等待50毫秒，然后返回两个整数的和。

在测试代码中，我们使用了`#[tokio::test]`属性宏标记异步测试函数。这将自动创建一个Tokio运行时，使得我们可以直接在测试函数中调用`.await`。

  


使用`cargo test`命令运行测试，Tokio会自动处理异步测试函数，并确保它们按预期执行。通过使用Tokio提供的测试工具，你可以方便地编写和运行异步代码的单元测试。

  


# 性能测试以及 criterion 库

性能测试是对软件系统进行评估，以了解其在特定条件下的性能。性能测试通常关注系统的响应时间、吞吐量、资源利用率等方面。性能测试可以帮助开发者找出系统的瓶颈，优化代码，提高系统性能。

  


在rust项目中进行性能测试，可以使用 criterion 库，criterion 是一个用于Rust的性能测试库，可以用于创建性能测试来衡量代码的性能。criterion 提供了易于使用的API，并生成详细的报告。

当前 Rust unstable channel 里对于 benchmark 类型的 testcase 提供了 `#[bench]` attribute，由于在本文写作时其还不支持stable channel，因此这里不做详述，详见：[cargo book](https://doc.rust-lang.org/cargo/commands/cargo-bench.html) 。

  


以下是一个使用criterion的简单示例：

假设我们有这么一个生成 fibonacci 数列的函数，我们需要对它的性能进行benchmark

```Rust
// lib.rs
pub fn fibonacci(n: u64) -> u64 {
    match n {
        0 => 1,
        1 => 1,
        n => fibonacci(n-1) + fibonacci(n-2),
    }
}
```

首先，我没需要在项目中添加 criterion依赖。

  


## 在 Cargo.toml 中添加 criterion 依赖

运行以下 cargo 命令

```Shell
cargo add criterion
```

添加完成后Cargo.toml 内会有如下的新增依赖

```TOML
  [dev-dependencies]
  criterion = "0.4"
```

  


## 创建性能测试代码

```Rust
  // benches/my_benchmark.rs
  use criterion::{criterion_group, criterion_main, Criterion};
  use my_lib::fibonacci;

  pub fn bench_fibonacci(c: &mut Criterion) {
      c.bench_function("fibonacci bench", |b| b.iter(|| fibonacci(20)));
  }

  criterion_group!(benches, bench_fibonacci);
  criterion_main!(benches);
```

在这个示例中，我们创建了一个名为 `bench_fibonacci` 的性能测试，用于衡量`fibonacci`函数的性能。

`criterion_group!`宏用于定义一个性能测试组，而 `criterion_main!` 宏用于定义一个入口函数。

  


编写完 benchmark代码之后，需要在 Cargo.toml 里指定相应的位置：

```TOML
[[bench]]
name = "my_benchmark"
harness = false
```

  


注意：cargo 在编译 benchmark 代码的时候会将其看做一个独立的 crate ，因此我们需要 use 关键字将我们的 crate 当作外部依赖引入到 benchmardk 代码中（上面测试代码的第三行），同时这也意味着我们只能对 public 的函数进行 benchmark。

  


### 运行性能测试

使用`cargo bench`命令运行性能测试。Criterion会自动执行性能测试，并生成性能报告。

运行输出结果如下：

```Plain Text
➜  my-lib git:(master) ✗ cargo bench
   Compiling my-lib v0.1.0 (/Users/bytedance/code/temp/my-lib)
    Finished bench [optimized] target(s) in 1.98s
     Running unittests src/lib.rs (target/release/deps/my_lib-77edfebf8ee4f3bb)

running 1 test
test tests::it_works ... ignored

test result: ok. 0 passed; 0 failed; 1 ignored; 0 measured; 0 filtered out; finished in 0.00s

     Running benches/my_benchmark.rs (target/release/deps/my_benchmark-35a5ab9a4d1bad5a)
fibonacci               time:   [16.505 µs 16.563 µs 16.629 µs]
Found 8 outliers among 100 measurements (8.00%)
  8 (8.00%) high mild
```

上面标绿的输出内容显示了针对该性能测试测量的每次迭代时间的置信区间。 左右值分别表示置信区间的下限和上限，而中间值显示 criterion 对基准程序每次迭代所用时间的最佳估计值。

如果多次运行`cargo bench` 的话，输出内容中还会显示每一个运行相比于上一次的性能变化结果。
