当你已经掌握了 Rust 的基本语法，接下来你可能会想要学习异步编程。Rust提供了强大的异步编程支持，使得你可以编写高效且易于维护的异步代码。

  


在这一章中，我们将深入探讨 Rust 的异步编程概念、语法和最佳实践，帮助你掌握异步编程的技能，从而在实际项目中应用它们。无论你是想学习Web开发、网络编程还是其他需要异步编程的领域，这一章都将为你提供有价值的知识和技能。让我们开始吧！

  


# Hello Async!

Rust 中写异步代码非常简单，你只需要：

1.  将含有异步逻辑的函数定义为 `async fn`
1.  在需要等待时 `.await`
1.  如果需要创建独立运行的任务，使用 `spawn`
1.  如果需要自行启动（作为 bin 而不是 lib 时），需要引入一个 Runtime

## Async fn

```Rust
// `foo()` returns a type that implements `Future<Output = u8>`.
// `foo().await` will result in a value of type `u8`.
async fn foo() -> u8 { 5 }

fn bar() -> impl Future<Output = u8> {
    // This `async` block results in a type that implements
    // `Future<Output = u8>`.
    async {
        let x: u8 = foo().await;
        x + 5
    }
}
```

Async fn 意思是一个可能会“切出去”的函数。

如果你使用 `std::thread::sleep`，那么当前线程会陷入 syscall，直到倒计时结束，在此期间线程无法完成其他任务；而当你使用 `tokio::time::sleep`，则它会生成一个 Future。对该 Future 做 await 会让这一切看起来像是 `std::thread::sleep`，但实际上线程并不会陷入等待，而是将任务挂起，并执行其他任务。

我们暂时不探究 async/await 背后的原理，只是简单使用的话，其实完全可以将其类比为线程。

在需要等待时 await，如果某个函数内用到了 await，则该 fn 也需要被定义为 async fn —— 它等价于前面的第二种写法。

  


## Spawn

类似 thread spawn，要实现同时执行的控制流，我们可以利用 runtime 提供的 spawn 方法。

与 async/await 不同的是，spawn 方法一般会绑定 runtime 实现。

以 tokio 为例，spawn 方法会返回一个 `JoinHandle`。Drop 这个结构不会导致 task 被取消，而利用这个结构可以用于某些需要等待或判断 spawn 的任务完成的需求。

  


## 启动

一个常规的程序需要有一个 main 函数作为入口点，带有异步逻辑的程序也不例外。

我们引入一个 runtime，该 runtime 会等待在我们给的 “主任务” 上。

```Rust
#[tokio::main]
async fn main() {
    let num = foo().await + bar().await;
    tokio::spawn(background()).await.unwrap();
    println!("{num}");
}

async fn foo() -> u8 {
    1
}

async fn bar() -> u8 {
    2
}

async fn background() {
    println!("background task executed");
}
```

  


## 一个真实的例子

我们以 HTTP 请求为例，这个例子请求 http://t.cn 并打印 body：

```Rust
#[tokio::main]
async fn main() {
    let client = reqwest::Client::default();
    let resp = client.get("http://t.cn").send().await.unwrap();
    let data = resp.text().await.unwrap();
    println!("data: {data}");
}
```

  


# 运行时与 IO Trait

我们通常使用 Tokio 作为我们的运行时。Tokio 是一个支持多线程任务窃取的 Runtime，在任务种类复杂，负载不均时这种类似 Golang 任务调度的能力可以降低延迟。前面代码中的 `#[tokio::main]` 是一个宏，它会将你定义 async main 转换为启动 tokio runtime 并执行 task 的同步 main 方法。

关于运行时后续章节会有更详细的讲解，当下我们只需要使用 Tokio 即可。

在 Tokio 中，我们最常见的是 `AsyncRead` 和 `AsyncWrite`，TcpStream 等结构就是通过这个 trait 暴露异步读写接口，而 TcpListener 则暴露 Stream（后续的章节中会有介绍）。

```Rust
pub trait AsyncRead {
    fn poll_read(
        self: Pin<&mut Self>,
        cx: &mut Context<'_>,
        buf: &mut ReadBuf<'_>
    ) -> Poll<Result<()>>;
}

pub trait AsyncWrite {
    fn poll_write(
        self: Pin<&mut Self>,
        cx: &mut Context<'_>,
        buf: &[u8]
    ) -> Poll<Result<usize, Error>>;
    fn poll_flush(
        self: Pin<&mut Self>,
        cx: &mut Context<'_>
    ) -> Poll<Result<(), Error>>;
    fn poll_shutdown(
        self: Pin<&mut Self>,
        cx: &mut Context<'_>
    ) -> Poll<Result<(), Error>>;

    fn poll_write_vectored(
        self: Pin<&mut Self>,
        cx: &mut Context<'_>,
        bufs: &[IoSlice<'_>]
    ) -> Poll<Result<usize, Error>> { ... }
    fn is_write_vectored(&self) -> bool { ... }
}
```

如我们所见，`AsyncRead` 和 `AsyncWrite` 提供了 poll 风格的 trait 定义。借助这两个 Trait，我们可以封装 IO 实现自定义逻辑，例如带 buffer 的 IO。

  


# 并行任务控制

前面提到了 spawn，可以创建新的并行任务（和 Golang 中的 go 关键字十分类似）。除此之外，还有一些其他的结构可以辅助做一些任务控制。

## Join

如果需要并行等待多个任务，除了 spawn 并分别等待外，我们也可以直接在原 task 内同时等待多个任务（这得益于 Rust 的异步 task 本质是可以方便组合的状态机，与写 golang 的经验不太一样）。

```Rust
#[tokio::main]
async fn main() {
    let now = std::time::Instant::now();
    tokio::join!(wait_sec(2), wait_sec(2), wait_sec(2));
    println!("elapsed: {} secs", now.elapsed().as_secs());
}

async fn wait_sec(secs: u64) {
    tokio::time::sleep(std::time::Duration::from_secs(secs)).await
}

// Output:
// elapsed: 2 secs
```

## Select

Select 是并行等待多个任务，并在首先完成的任务结束后立刻返回。

```Rust
#[tokio::main]
async fn main() {
    let now = std::time::Instant::now();
    tokio::select!{
        _ = wait_sec(1) => {
            println!("1 sec pass");
        },
        _ = wait_sec(2) => {
            unreachable!();
        },
        _ = wait_sec(3) => {
            unreachable!();
        },
    }
    println!("elapsed: {} secs", now.elapsed().as_secs());
}

async fn wait_sec(secs: u64) {
    tokio::time::sleep(std::time::Duration::from_secs(secs)).await
}

// Output:
// 1 sec pass
// elapsed: 1 secs
```

Select 常常被用于：超时返回、控制后台任务退出 等场景。

## FuturesUnordered

当我们需要等待多个 Task 结束，但 Task 并不能像前面的 Join 里例子一样可列举时，我们可以用 `FuturesUnordered` 这个辅助结构。

```Rust
use futures::StreamExt;

#[tokio::main]
async fn main() {
    let now = std::time::Instant::now();
    let task_count = 100;
    let mut futs = futures::stream::FuturesUnordered::default();
    (0..task_count).for_each(|num| futs.push(wait_ms(num)));
    while futs.next().await.is_some() {}
    println!("elapsed: {} ms", now.elapsed().as_millis());
}

async fn wait_ms(secs: u64) {
    tokio::time::sleep(std::time::Duration::from_millis(secs)).await
}

// Output:
// elapsed: 100 ms
```

除了这几种外，你也可以通过手动实现 Future 的形式实现自定义的控制逻辑。

  


# 异步代码的局限

## 同步逻辑混合问题

如果我们在异步逻辑内混入了某个同步逻辑，则我们再无办法将其拆分出来。

```Rust
async fn demo() {
    call1().await;
    cpu_heavy_calc();
    call2().await;
}
```

例如上述代码，会导致这个 task 无论怎么被调度，都会在其执行到 `cpu_heavy_calc` 时较长地占用当前线程，导致当前线程上的其他任务无法及时调度，影响服务延迟。

解决办法是，在将较重的同步逻辑混入异步代码的时候将其 `spawn_blocking`。其原理是将逻辑发送到独立线程池执行，我们在当前线程只需要等待其执行解决即可。等待期间，本线程是可以处理其他任务的。

注意事项：在 async 逻辑中，我们不能使用可能会卡住线程的同步调用，包括但不限于 block_on 等函数。

  


## Async Trait

前面介绍的 AsyncRead 和 AsyncWrite 通过 poll 风格定义，看起来一点也不直观。那么能不能直接用 async/await 定义并实现异步 Trait 呢？

由于 `async fn xx() -> Out` 的本质是 `fn xx() -> impl Future<Output=Out>`，而这个返回值类型是生成结构，我们无法拿到，所以当我们定义 Trait 时就无法写出返回值类型，导致无法为异步逻辑定义 Trait。

使用 async/await 定义并实现 trait 一直是 Rust 语言的一个目标，但当下仍存在一些问题，需要一些变通的做法才能完成。

-   一个解决方案是使用 `async-trait` 包提供的宏，这个宏会将匿名 Future 结构包装进 Box（有运行时开销），则此时可以写出类型，完成 Trait 定义。
-   另一个解决方案是使用 TAIT，这部分请参考异步高级篇中的对应章节。

  


## Async Drop

由于语言本身的限制，我们无法在 Drop 方法中 await。

  


## 传染性

由于等待是递归的，所以 async 是有传染性的。如果我们要在一个函数中调用另一个 async 函数，则这个函数也要声明为 async fn。这很容易导致 async 被大范围传染——必要的传染是合理的，我们在编程时也要注意尽量保持同步与异步逻辑的分离。
