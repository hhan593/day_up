## Trait 是什么

其实对于很多 Go 开发者来说，可以简单地把 Trait 理解成 Interface，语义上都是指一种类型约束。

但是 Rust 里面的 Trait 又要比 Go 里面的 Interface 表达能力强大得多。

  


## 定义一个 Trait

```Rust
trait Echo {
    type Content: std::fmt::Display; // trait 中的关联类型

    fn echo(&self) {
        // 拥有一个默认实现，可以在为其他类型实现时不实现这个方法
        println!(
            "echo content is {}",
            self.echo_content()
        );
    }
    fn echo_content(&self) -> Self::Content;
}
```

Rust 中可以在 Trait 中指定关联类型，Const， 并且还可以添加默认实现。

在 Trait 的关联类型中，可以指定一些Trait Bound 约束，比如 `type Content: std::fmt::Display;` 要求 `Content` 比如满足`Display` 这样可以被用做 `println!` 输出

## 实现一个 Trait

```Rust
struct A {}

impl Echo for A {
    type Content = i64;

    fn echo_content(&self) -> Self::Content {
        666
    }
}
```

我们只需要通过`impl` 就可以给一个类型实现 Trait。

## Trait Object

我们可以使用一个 trait object 来作为参数传入，这样可以方便我们进行函数抽象。

比如

```Rust
fn test(e: &dyn Echo<Content = i64>) {
    e.echo();
}
```

这种使用方法其实就是“动态分发” (区别于泛型的“静态分发”，下一章节会讲到)。`&dyn Echo` 其实是一个 "fat pointer"。由两个指针组成，一个是数据指针，一个是vtable指针。

```Rust
println!(
    "{}",
    std::mem::size_of::<&'static dyn Echo<Content = i64>>()
); // 我们通过这种方式输出一下，在64位机器上得到的结果是16. 
```

使用 Trait Object 进行方法调用的时候，会通过 vtable 获取函数指针，然后通过函数指针调用方法，可能会对编译器的inline产生影响，因此可能对性能有一定的损耗。

  


## Orphan Rule

在实现 Trait 的时候要注意下 "Orphan rule" 孤儿原则。

所有 impl Trait 都需要满足一下两个要求之一：

比如针对 `impl<P1..=Pn> Trait<T1..=Tn> for T0`

1.  Trait 是在当前 crate 定义的
1.  T0..Tn 至少有一个是在当前 crate 定义的（Ti是第一个）并且 `T0..Ti` 中不存在"[uncovered type](https://doc.rust-lang.org/reference/glossary.html#uncovered-type)" 类型参数。

但是在这里还存在一个特殊判断，T 对于 Box<T> 仍然是 uncoverd type

  


比如在 crate foo 中定义了

```Rust
pub trait Foo<T1, T2, T3> {}
```

我们在其他crate 中 定义了 `struct A`

```Rust
impl<T1, T2> Foo<A, T1, T2> for String {} // 成功编译。
impl<T1, T2> Foo<T1, A, T2> for String {} // 编译失败，因为在A的前面存在 T1 是 uncoverd type
impl<T1, T2> Foo<Vec<T1>, A, T2> for String {} // 成功编译，因为 T1 对于 Vec<T1> 来说是 covered type
impl<T1, T2> Foo<Box<T1>, A, T2> for String {} // 编译失败，因为在这里特殊判断 Box<T1> 并不认为是 covered type
```

  


## Object Safety

在使用 Trait Object的时候，我们需要遵守 Object Safety 原则。

Object Safety的原则请见：https://doc.rust-lang.org/reference/items/traits.html#object-safety

不妨来思考一下为什么对于 Trait Object 会存在这一堆使用限制呢？

因为 Trait Object 在使用的时候是一个"fat pointer", vtable 中的 methods 都是函数指针的形式存在。而如果这些 method 中存在泛型，泛型的 monomorphize 是需要在编译期确定的。运行时的函数指针无法针对泛型参数进行 instantiate。所以泛型 method 无法存在于 trait object中。
