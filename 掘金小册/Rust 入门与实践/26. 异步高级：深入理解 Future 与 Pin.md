## Future 简介

Future 接口是 Rust 异步编程的核心。 Future是一个可以输出值（可能为空值，比如()）的异步计算过程。简化版的 Future 如下：

```Go
trait SimpleFuture {
    type Output;
    fn poll(&mut self, waker: fn) -> Poll<Self::Output>;
}

enum Poll<T> {
    Ready(T),
    Pending,
}
```

  


Future 通过调用其 poll 函数来进入下个状态，让Future尽可能接近完成。Future完成返回 Poll::Ready(result)。

如果 Future 没有完成则返回 Poll::Pending，将waker注册进异步接口，等io事件ready调用wake，重新将此future放回scheduler进行poll。

  


## Future 与 Async 的关系

在讲Future之前我们先讲异步的两种实现，`stackful`和`stackless`

**stackful:** 又被称为 fiber 调度， Go为代表

**stackless:** 状态机，变量保存在struct，特点是编译期状态大小确定， rust，C#为代表

  


### stackful的问题

#### M-N model 随机调度导致 crash

如果stackful 模型的用户代码使用了 TLS(thread local storage)。fiber 从线程A被调度到B。此fiber在B执行时有两种情况：

1.  **如果 A 依旧存在：** compiler 缓存了原来A的TLS地址，则会读取、修改原线程A的TLS的内容
1.  **A 已经退出：** cache的地址早就被释放，读取会引起memory corrupt。

  


总之， stackful 里使用TLS，这时task被调度到另一个线程，task会访问到原来的TLS slot。一不小心就会出现上述两个问题。

  


**解决办法：** 使用 1:N 的 fiber，task永远在当前线程的调度队列执行，相当于Rust的`!Send`

tokio 提供work stealing的scheduler，task会在不同thread转移，所以tokio的future里不能用TLS，官方提供了近似于TLS的 [task#LocalKey](https://docs.rs/tokio/1.7.1/tokio/task/struct.LocalKey.html)

  


#### rewrite all blocking api call

必须重写全部blocking api call并携带 userspace scheduler（就像Golang），确保当调用到这些 blocking api时，可以进行fiber的切换。

## Split stacks/segmented stacks

stack 固定后无法动态扩容，go将stack以类似链表的形式连接。

![](https://p3-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/26c795153b564773b5c728c4d28723b5~tplv-k3u1fbpfcp-jj-mark:0:0:0:0:q75.image#?w=891&h=177&e=png&b=fefefe)

考虑下面这种情况：

在stack分配很少的内容导致新的segmented stack被创建，函数调用结束segmented stack又被释放（其实是个链表）。如果这个函数在loop中持续调用就会产生极多 allocation overhead。这在Go里叫做 hot split 问题。Rust中RustNoSeg（之后弃用了 stackful 异步模型）

Golang 之后改为 growable stack：当stack容量不足时分配新的stack，并将原来stack上的内容复制到新的stack，将指向原来stack的指针都指向新的stack。

这在Go这种自带GC的语言里很容易做到（调整指针的指向），但Rust由于没有GC，做不到动态调整指针，会引用原来stack的内容，导致dangle pointer。

这也是为什么Rust必须引入Pin（Rust没有GC，无法追踪指针指向的内存）防止指针被误move

Rust采用stackless的异步实现方式，`async await`语法会被编译成匿名结构体，结构体保存着整个future执行的全生命周期跨过`await`点的变量，用来在不同状态进行切换

  


## Pin简单科普

关于Pin的介绍已经有很多了，想提供一个不同的视角。

***不依赖Pin能否做到希望的不被move。Pin 的作用是防止 move，但如果程序员小心处理，那就不会出错。为什么还需要 Pin 呢？***

既然在某些场景下move是错的，那作为安全的编程语言，就需要显式限制这些场景。**不能将安全交给人的直觉**保证，**这是** **Rust** **编译器的责任，不是人的**。

  


值得注意的是，即使有Pin如果想制造 UB 行为，通过 unsafe 也**可以** 做到。我只是想说明，Pin没有魔法

即使别人眼中UB是错的，但目的不同，对正确的理解不同。如果想要的就是UB的结果，那这UB对你而言是正确的，但这种超出语言的范畴，Pin也不是为这种场景（人的主观）而生的。

在代码整个链路不使用任何unsafe情况下，哪怕没有Pin，你也不会写出UB的代码。编译器的静态检查会教你做人。

unsafe 作为 rust 的一部分，其存在是有意义的，Pin也主要针对这场景。

  


Pin的意义在于，当你不想出错时，通过 Pin 隐藏 `&mut T`，从语法上一定程度限制 unsafe 的危险性。

为什么限定 unsafe，在safe rust 中，构造自引用并且用起来都十分困难，UB受到 borrow checker的严格限制。不会由于move特性导致错误。
