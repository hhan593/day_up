# 别具一格的 Rust 错误处理

绝大多数主流编程语言的错误处理都是采用 `try...catch`,`finally` 之类的语法，并且还有异常 `Exception` 之类的概念，而 Rust 的错误处理却抛弃了这些方式，完全依靠 Rust 的类型系统来做错误的处理，所以这让 Rust 这种错误处理显得别具一格。

具体来说，Rust 的错误处理主要有如下三个概念：

## Option

Rust 具有完备的类型系统，不包含空类型 null 这样的概念，所以 Rust 也没有空指针异常这种错误。对于可能为空的类型，Rust 提供了 `Option` 这个枚举类型来表达，其中`Some()`代表值存在的情况，`None` 则代表值为空的情况。

```
pub enum Option<T> {
    None,
    Some(T),
}
```

下面看一个简单的例子：

```
struct User {
  id: u64,
  name: Option<String>,
}
```

`Option` 通常用在结构体的字段上用来表示这个字段值不是必须的，比如 `User` 可能没有 `name`。

```
// 除法函数，如果被除数为 0 则返回 None
fn divide(numerator: f64, denominator: f64) -> Option<f64> {
    if denominator == 0.0 {
        None
    } else {
        Some(numerator / denominator)
    }
}
​
​
assert_eq!(divide(6.0, 3.0), Some(2.0));
assert_eq!(divide(6.0, 0.0), None);
```

上面的除法函数对于可以正常相除的情况返回 `Some`，对于除零这种无意义的情况直接返回 `None`。`Option` 作为返回值是一个非常常用的操作，标准库有大量 `get` 类方法返回值都是 `Option`，比如 `HashMap::get()`。

## Result

Rust 中对于成功或失败的操作结果使用 `Result` 这个类型来表达，成功返回 `Ok(T)`，否则返回 `Err(E)`，其中的 `T` 和 `E` 分别是结果的类型和具体错误的类型。

```
pub enum Result<T, E> {
    Ok(T),
    Err(E),
}
```

从 `Result` 的定义中，我们可以看到 `E` 没有任何类型约束，它可以是 `String`, `&'static str` 等任何类型。

继续看一个例子：

```
#[derive(Debug)]
enum Version { Version1, Version2 }
​
fn parse_version(header: &[u8]) -> Result<Version, &'static str> {
    match header.get(0) {
        None => Err("invalid header length"),
        Some(&1) => Ok(Version::Version1),
        Some(&2) => Ok(Version::Version2),
        Some(_) => Err("invalid version"),
    }
}
​
let version = parse_version(&[1, 2, 3, 4]);
match version {
    Ok(v) => println!("working with version: {v:?}"),
    Err(e) => println!("error parsing header: {e:?}"),
}
```

`parse_version()` 是一个解析版本号的函数，它的返回值是 `Result<Version, &'static str>`，这里的泛型 `E` 就是一个 `&'static str`。

当然 `Result<T, E>` 里面的 `E` 通常是一个实现了 `Error` trait 的具体错误类型，接下来我们介绍一下 `Error` 这个 trait。

## Error

实现`Error` 这个 trait 的时候需要先实现它的两个 super trait `Debug` 和 `Display` 用来提供错误描述信息，`Error` trait 的方法都有默认实现，通常不需要太关心，只有 `source()` 方法在需要错误溯源的时候需要实现一下。

```
pub trait Error: Debug + Display {
    fn source(&self) -> Option<&(dyn Error + 'static)> { ... }
}
```

下面看一个简单的例子：

```
use std::error::Error;
use std::fmt;
​
#[derive(Debug)]
struct SuperError {
    source: SuperErrorSideKick,
}
​
impl fmt::Display for SuperError {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        write!(f, "SuperError is here!")
    }
}
​
impl Error for SuperError {
    fn source(&self) -> Option<&(dyn Error + 'static)> {
        Some(&self.source)
    }
}
​
#[derive(Debug)]
struct SuperErrorSideKick;
​
impl fmt::Display for SuperErrorSideKick {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        write!(f, "SuperErrorSideKick is here!")
    }
}
​
impl Error for SuperErrorSideKick {}
​
fn get_super_error() -> Result<(), SuperError> {
    Err(SuperError { source: SuperErrorSideKick })
}
​
fn main() {
    match get_super_error() {
        Err(e) => {
            println!("Error: {e}");
            println!("Caused by: {}", e.source().unwrap());
        }
        _ => println!("No error"),
    }
}
```

`SuperError` 有一个字段 `source` 记录它的错误来源，如果不需要关心错误来源，实现 `Error` trait 的时候不需要重写 `soure()` 方法，比如 `impl Error for SuperErrorSideKick {}`。

上面的代码运行后输出如下：

```
Error: SuperError is here!
Caused by: SuperErrorSideKick is here!
```

# 错误处理

了解了上面几个概念之后，我们来看一下 Rust 常用的错误处理方式。

## Panic 与 Unwrap

Rust 提供了一个 `panic!()` 宏，在程序发生不可恢复的严重错误时，可以直接使用 `panic!()` 退出进程。

`Option` 和 `Result` 都提供了各自的 `unwrap()` 方法，里面也是调用了 `panic!()` 宏。

Rust panic 的实现机制有两种：**unwind** 和 **abort**。

-   **unwind**: 方式会一层一层退出函数调用栈，正常析构栈上的资源。
-   **abort**: 方式会直接退出程序。

**unwind** 会有比较大的性能开销，嵌入式或其他资源紧张的环境不支持 **unwind**，这时候都是 **abort** 模式。

可以在 Cargo.toml 文件指定 panic 的具体模式，**abort** 的额外好处是能够降低程序二进制文件大小。

```toml
// Cargo.toml
[profile.release]
panic = 'abort' // Default is unwind
```

## catch_unwind 与 UnwindSafe

-   [std::panic::catch_unwind](https://doc.rust-lang.org/std/panic/fn.catch_unwind.html) 可以用来捕获可能会发生 panic 的代码并阻止栈回退(**unwind**)，但是**不要把它当做其他语言的 try/catch 使用**。

> abort 模式下永远都是直接退出，catch_unwind() 在这种模式下无效。

```
use std::panic;

let r = panic::catch_unwind(|| {
    panic!("Panic in catch_unwind()...");
});

assert!(r.is_err());

let r = panic::catch_unwind(|| {
    println!("No panic...");
});

assert!(r.is_ok());
```

-   `std::panic::catch_unwind` 主要适用如下几种情况：

- **FFI**：其他语言通过 FFI 调用 Rust 函数时，可以使用 `catch_unwind()` 捕获可能发生的 panic 并手动处理好，避免将 `unwind` 抛到其他语言那端，造成 `undefined behavior`。

- **线程池**：如果某个线程发生 panic 了，可以使用 `catch_unwind()` 捕获，阻止栈展开，然后单独退出该线程，避免将整个线程池"拖下水"。

- **Future**：futures-rs 的 `FutureExt` 和 `StreamExt` 都提供了 [catch_unwind](https://docs.rs/futures/0.3/futures/future/trait.FutureExt.html#method.catch_unwind) 机制，可以捕获 `Future` poll 的时候可能发生的 panic。详细可以看相应的文档。

-   使用 `std::panic::catch_unwind` 的时候，闭包需要满足 [UnwindSafe](https://doc.rust-lang.org/std/panic/trait.UnwindSafe.html) 约束。`UnwindSafe` 是一个 `auto trait`，绝大部分类型都满足这个约束，只有少数类型不满足 `UnwindSafe`，其中最值得注意的是 `&mut T`，`Ref`，`RefMut`。

```
  use std::panic;

  let mut variable = 4;

  // This code will not compile because the closure captures &mut variable
  // which is not considered unwind safe by default.
  panic::catch_unwind(|| {
      variable += 3;
  });
```

-   可以使用 `AssertUnwindSafe` 包裹不满足 `UnwindSafe` 约束的类型绕开这种限制，这种情况下需要由开发者自己保证你的代码不会发生 `unwind safety` 的问题。

```
  use std::panic::{self, AssertUnwindSafe};

  let mut variable = 4;

  // This, however, will compile due to the AssertUnwindSafe wrapper
  let result = panic::catch_unwind(AssertUnwindSafe(|| {
      variable += 3;
  }));
```

-   使用 `AssertUnwindSafe` 时尽量应用到最小作用域，以便在未来新增一个不满足 `UnwindSafe` 的类型时能够及时在编译阶段发现。

```rust
  use std::panic::{self, AssertUnwindSafe};
  // Bad
  let mut variable = 4;
  let other_capture = 3;

  let result = {
      panic::catch_unwind(AssertUnwindSafe(move || {
          let mut wrapper = &mut variable;
          *wrapper += other_capture;
      }))
  };

  // Good
  let mut variable = 4;
  let other_capture = 3;

  let result = {
      let mut wrapper = AssertUnwindSafe(&mut variable);
      panic::catch_unwind(move || {
          **wrapper += other_capture;
      })
  };
```

## 错误传递与问号表达式

Rust 的错误通常是一层一层往上传递，最后交给消费错误的业务方处理错误，在错误传递过程路径中通常会包含多种不同类型的错误，往往通过一个最顶层的统一错误类型来包裹这些错误类型。比如：

```
enum MyError {
    IOError(io::Error),
    RequestError(reqwest::Error),
    OtherError,
}
​
impl From<io::Error> for MyError {
    fn from(err: io::Error) -> MyError {
        MyError::IOError(err)
    }
}
​
impl From<reqwest::Error> for MyError {
    fn from(err: reqwest::Error) -> MyError {
        MyError::RequestError(err)
    }
}
​
async fn foo(url: &str) -> Result<(), MyError>{
    let text = reqwest::get(url)
        .await?
        .text()
        .await?;
​
    let mut file = File::create("file.txt")?;
    file.write_all(text.as_bytes())?;
​
    Ok(())
}
```

通过给最顶层的错误类型（比如这里的 `MyError`) 实现各种 `From` trait，来处理错误类型的转换。另外 Rust 提供了一个 `? 表达式`来便捷的处理这种错误转换。

# 借助第三方库来做错误处理

Rust 错误处理机制目前还不够完善，比较大的原因在于 `Error` trait 有不少不合理之处，比如 `cause()`(已弃用)不支持 `'static` 约束，以至于不能做向下转换 (`Error::downcast_ref()`)，不支持 `backtrace` 等(目前 nightly 开始支持了)，详细可以查看 [RFC 2504](https://github.com/rust-lang/rfcs/blob/master/text/2504-fix-error.md)。

经过 Rust 社区多年的实践和经验总结，目前按照 library 和 application 有不同的错误处理需求，将错误处理库分成 [thiserror](https://docs.rs/thiserror) 和 [anyhow](https://docs.rs/anyhow)。

## library 使用 thiserror 来做错误处理

通常来说，library 专注于产生 (producing)、组合 (composing) 和上报 (reporting) 结构化的 Error，将 Error 的消费权交给最终的 application，避免在 library 层消费 Error。

```
use thiserror::Error;
​
#[derive(Error, Debug)]
pub enum DataStoreError {
    #[error("data store disconnected")]
    Disconnect(#[from] io::Error),
    #[error("the data for key `{0}` is not available")]
    Redaction(String),
    #[error("invalid header (expected {expected:?}, found {found:?})")]
    InvalidHeader {
        expected: String,
        found: String,
    },
    #[error("unknown data store error")]
    Unknown,
}
```

## application 使用 anyhow 来做错误处理

application 通常专注于统一消费 (consuming) 自身项目或各种其他 library 报上来的错误。

```
use anyhow::Result;
​
fn get_cluster_info() -> Result<ClusterMap> {
    let config = std::fs::read_to_string("cluster.json")?;
    let map: ClusterMap = serde_json::from_str(&config)?;
    Ok(map)
}
```

# 总结

Rust 的错误处理是通过分层的方式来实现：

-   如果某个类型的值可能为空，则使用 `Option`;
-   如果可能发生错误但是还可以被处理，则使用 `Result` 和 `Error` trait;
-   如果发生了错误而且不能被处理，则使用 `panic!()`;
-   `Result` 和 `panic!` 分别被称为可恢复错误(Recoverable Error)和不可恢复错误(Non-recorable Error);

|                     | Recoverable               | Non-recoverable |
| ------------------- | ------------------------- | --------------- |
| Defining            | types 和 traits            | panic!          |
| Propagating         | ?表达式                      | builtin         |
| Matching 和 Reacting | match, if let 或者 downcast | 不支持             |
| Discarding          | drop 或 unwrap             | catch_unwind    |
| Reporting           | Error trait               | panic hook      |

-   `Result` 与 `Option` 的所有方法，它们大都是可以组合的，让编写的代码更符合 Rust 风格;
