# 使用 itoa 和 ryu

对数据类型进行序列化时，可以使用[ itoa](https://docs.rs/itoa/latest/itoa/) 和 [ryu](https://docs.rs/ryu/latest/ryu/) 这两个基础库，比标准库的实现性能更好。这两个库已经应用在 serde-json 中，稳定性有保障。下面是使用 itoa 序列化 u64 的一个例子：

```Rust
fn main() {
    let mut buffer = itoa::Buffer::new();
    let printed = buffer.format(128u64);
    assert_eq!(printed, "128");
}
```

 

# 字符串 / bytes 相关处理库

使用高效的字符串处理，能够避免不必要的拷贝。下面推荐使用以下几个 字符串/bytes 相关的库：

-   **[Bytes/BytesMut](https://docs.rust-embedded.org/rust-sysfs-gpio/bytes/struct.Bytes.html)**: 底层内存空间是连续的，避免了对 `bytes([u8])` 的拷贝。多个 `Bytes` 对象可以共享一段连续的内存空间 `[u8]`，底层通过 `Arc` 引用计数机制来确保对象的安全释放。
-   **[Faststr:](https://docs.rs/faststr/latest/faststr/)** 避免了 `String` 拷贝。主要基于的假设是 `String` 在传递过程中不需要改变，那就可以做很多优化。`FastStr` 自身的 `Clone` 是零开销的。faststr 的易用性比较好，支持 `String`、`Arc<String>`、`Cow`、`&'static str` 这些都可以直接 into 到 `FastStr`；对 `Bytes`、`BytesMut`、`Vec<u8>` 也可以直接用 unsafe 方式转到 `FastStr`，而不需要clone。
-   **[LinkedBytes](https://crates.io/crates/linkedbytes)**: 是 `Bytes/BytesMut` 的 linklist 版本。非常适合在内存不连续，同时又想实现零拷贝的场景中使用。目前，Volo thrift 的 codec 部分就支持使用 `LinkBytes` 存储 thrift 编码，适合在大包场景下使用。

  

# 使用基于栈的变长数据结构

定长度的数据类型可以简单地存储在堆栈上，但对于动态大小的数据，并不能直接放在栈上。但是，我们可以使用 基于栈的变长数据结构，例如`smallvec`, `faststr`，允许在栈上存储少量元素，将多余的数据存放到堆上面，但是仍然保留动态长度。

例如，下面的 `smallvec` 将在栈上面存放 4 字节，多余数据会被放到堆上面：

```Rust
use smallvec::SmallVec;
let mut v = SmallVec::<[u8; 4]>::new(); // initialize an empty vector

// The vector can hold up to 4 items without spilling onto the heap.
v.extend(0..4);
assert_eq!(v.len(), 4);
assert!(!v.spilled());

// Pushing another element will force the buffer to spill:
v.push(4);
assert_eq!(v.len(), 5);
assert!(v.spilled());
```

  

# 使用合适的 json 库

Rust 的 `serde_json` 库虽然是事实上面的 json 标准，但是在特定场景下，`serde_json` 库性能仍然不够好，可以使用更加高效的 json 库。

如果是解析到结构体或者泛型解析，可以使用 Rust 版本的 `simdjson`库，性能比 `serde_json` 好 2~3 倍, benchmark 可以参考 https://github.com/serde-rs/json-benchmark。

在很多业务场景中，我们不需要解析全量的 json，而是按需解析。这时候推荐使用 Rust 版本的 `gjson` 库，支持按路径对json进行按需解析。

例如，通过 `gjson` 可以扫描得到整个 array 的特定目标字段：

```Rust
    const PROGRAMMERS: &str = r#"
    {
        "programmers": [
          {
            "firstName": "Janet", 
            "lastName": "McLaughlin", 
          }, {
            "firstName": "Elliotte", 
            "lastName": "Hunter", 
          }, {
            "firstName": "Jason", 
            "lastName": "Harold", 
          }
        ]
      }
    "#;
    let mut res = String::new();
    let value = get(PROGRAMMERS,  "programmers.#.lastName" ); // gjson 的 json path
    for name in value.array() {
        res.extend(format!("{}\n", name).chars());
    }
    assert_eq!(res, "McLaughlin\nHunter\nHarold\n");
```

  


# 使用 `Arena` 内存池

Rust 中 内存分配的开销比较大。如果在使用了非常多的小对象的场景，每次内存分配和释放的开销累加起来不可小觑。这时候我们可以通过 `Arena` 机制来减少内存分配、频繁drop 等开销。

  


`Arena` 本质上对一组生命周期相同的小对象进行预分配内存。比如说有时需要在某个生命周期内分配一堆对象，每次都调用系统内存分配器效率低下，更可取的是为这一组对象预分配一块连续的大内存。另外，为每个小对象分别调用 drop 也会有很大的性能损耗，使用 `Arena` 机制后，可以立即将所有对象一次性清理干净。

  


Rust 中现有的 `Arena` 轮子 有 `bumpalo` 和 `typed_arena`。

-   **Bumpalo:** 使用了 [Bump allocation](https://rust-hosted-langs.github.io/book/chapter-simple-bump.html) 模式，支持在一块 `arena` 上分配不同的 Rust 对象。

```Rust
use bumpalo::Bump;
use std::u64;struct Doggo {
    cuteness: u64,
    age: u8,
    scritches_required: bool,
}// Create a new arena to bump allocate into.
let bump = Bump::new();// Allocate values into the arena.
let scooter = bump.alloc(Doggo {
    cuteness: u64::max_value(),
    age: 8,
    scritches_required: true,
});// Exclusive, mutable references to the just-allocated value are returned.
assert!(scooter.scritches_required);
scooter.age += 1;
```

-   **typed_arena:** 只能支持单个 rust 类型的 `Arena`。

```Rust
use typed_arena::Arena;

struct Monster {
    level: u32,
}

let monsters = Arena::new();

let goku = monsters.alloc(Monster { level: 9001 });
assert!(goku.level > 9000);
```
