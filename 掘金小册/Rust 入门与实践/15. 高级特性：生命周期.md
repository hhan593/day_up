Rust 的生命周期特性是它与其他语言的最大的一个区别，那么想要写好一个 Rust 程序就要理解 Rust 的 borrow checker 是如何工作的。

  


## 一个🌰

```Rust
fn main() {
    let mut a = 123123;
    let b = &a;

    a += 1;
    *b == a;
}
```

让我们来编译这段代码，得到了一个编译器报错。

```Rust
  |     a += 1;
  |     ^^^^^^ `a` is assigned to here but it was already borrowed
```

因为这里变量 b 借用了变量 a，在变量 b 借用 a 的范围内，我们是无法对变量 a 进行修改操作的。

ok，那么问题来了编译器是如何计算这个借用范围呢？

  


## 编译器是怎么知道 lifetime 的

让我们把这段代码展开一下。

在这个表达式中`let b = &a`; 在这里 `b` 的类型是 `&i32`, 右边表达式 `&a` 的类型也是 `&i32`。

在我们的代码里面任何使用Ref的地方，都需要产生一个 Region。虽然我们这里没有显示的指明任何 Region。 但是编译器会产生一个 Infer Region 然后进行推导。

那么我们将这里的类型信息给补全。

表达式 `&a` 的 `&i32` 类型 进行展开之后变成了`&'0 i32`, 这里`'0` 就是一个Region的表示。

那么同理 表达式 b 的`&i32` 类型 进行展开之后变成了`&'1 i32`。

那么经过我们的展开之后，我们再来看一下这段代码吧

```Rust
fn main() {
    let mut a = 123123;
    let b: &'1 i32 = &'0 a;
    a += 1;
    *b == a;
}
```

既然`'0` 这个生命周期可以被赋值到`'1`，那么就需要`'0` outlives `'1`。

只要变量`b` 还存活，那么生命周期`'1` 就会存活，那么我们来看看变量b在哪儿被使用了。从line 3 ～ line 5 属于变量`b` 的存活行数。那么`'1`的范围就是{3, 5}。那么`'0`的范围需要 outlives {3, 5}。

但是在 line 5 之后 `&'0 a` 这个表达式是没有被其他地方使用的。所以可以直接认为`'0`的范围也是{3, 5}。

在{3, 5}这个范围内，a 已经被 `'0` 这个生命周期给借用了, 那么在{3, 5}内对a的修改是会被编译器报错的。所以在第4行对a进行的修改操作是会被报错的。

## 函数定义中的生命周期

在函数声明时，我们可以通过生命周期的标记来做一些约束。

```Rust
fn test<'a>(a: &'a str, b: &'a str) -> &'a str {
    if xxx {
        a 
    } else {
        b
    }
}
```

在上面这个例子中，我们通过 `'a` 这个生命周期的标志约束了参数`a`, `b` 以及返回值的生命周期一致。

我们可能经常会有这样的一种场景。

```Rust
fn test<'a>(a: &'a str) -> &'a str {
    if xxx {
        a // 返回的是参数a, 类型是&'a str
    } else {
        let b = format!("hello world {}", a);
        &b // 这里返回一个在当前函数内构造的 String 的 reference
    }
}
```

因为上面的这个例子里面的 `b` 是一个在当前函数内构造的 `String`。这个变量 b 的有效生命周期仅限于函数内部的一个block。我们用下面这种方式表示应该会更清晰

```Rust
fn test<'a>(a: &'a str) -> &'a str {
    if xxx {
        a // 返回的是参数a, 类型是&'a str
    } else {
        'b: {
            let b = format!("hello world {}", a);
            &b
            // 在这里 b 被 drop， 然而返回值需要的 lifetime 是 'a 
        }
    }
}
```

所以我们收到了一个编译报错。因为 `&b` 并不满足 `'a` 这个约束。

那么我们有什么办法可以做到在这个函数内不同的条件分支返回`&'a str`或者 `String` 呢

那么我们就需要使用 Enum 了

```Rust
enum CowStr<'a> {
    Borrowed(&'a str),
    Owned(String)
}

fn test<'a>(a: &'a str) -> CowStr<'a> {
    if xxx {
        Cow::Borrowed(a)
    } else {
        Cow::Owned(format!("hello world {}", a))
    }
}
```

在这里我们使用 `CowStr` 不同的 Variant 来接受不同的类型，通过这种方式可以让最终的返回类型都是 CowStr

  


## Coerce

当生命周期满足 `'b` outlives `'a` 时，且满足“型变（variance）”规则时。那么可以可以进行 coerce。

比如还是上面的例子

```Rust
fn test<'a, 'b: 'a>(a: &'a str, b: &'b str) -> &'a str {
    if xxx {
        a 
    } else {
        b
    }
}
```

这里通过`'b:'a` 约束 `'b` 需要 outlives `'a`。那么在这里`&'b str` 就是 `&'a str` 的子类型了。所以可以进行 coerce。

  


## 'static

`'static` 是一个特殊的生命周期，语义是整个程序的存活生命周期。

比如 `const a: &str = "hello world";` 这是一个常量字符串，可以在程序内的任意地方去使用这个常量字符串，都不会报错。

但是有时候又会比较特殊，

比如有这样的一个函数

```Rust
fn assert_static<T: 'static>(t: T) {
}

assert_static("hello world");

let s = String::from("hello world");

assert_static(s);
```

要求传入的类型T必须满足`'static` 这个约束。像第4行那样，我们传入一个`&'static str` 这是很好理解的。

但是在第8行，我们传入的参数的类型是`String`, 这就很奇怪了，为什么 String 这个类型是满足`'static`约束的呢？

其实 `'static` 在 Trait Bound 中的语义有些区别。只要满足下面的两个条件之一就符合`'static`的约束

1.  一个`'static` 引用
1.  拥有所有权，且不存在任何非`'static`引用。

  


第一个条件很好理解，`&'static str` 就符合条件1。但是第二个条件是指什么呢？

比如类型 `String` 这就是一个拥有所有权的类型，而且不存在引用。所以符合第二个条件。

但是类型 `Ref<'a, T>` 呢？这个类型存在了一个引用`'a`, 只有当`'a` == `'static` 的情况下，`Ref<'a, T>` 才满足 `'static` 约束

  


## 被省略的生命周期

比如下面的这个函数

```Rust
fn get_str(a: Option<&str>) -> &str {
    match a {
        Some(a) => a,
        None => "hello"
    }
}
```

在这个函数的类型签名中，我们并没有给`&str` 标注任何生命周期。

但是其实对编译器来说，这个签名是会被展开成 `fn get_str<'a>(a: Option<&'a str>) -> &'a str` 的形式。
