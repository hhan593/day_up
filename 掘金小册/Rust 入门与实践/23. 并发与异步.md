同时处理多个任务是一个常见的的需求，比如一个 web server 必须能够同时处理多个用户的请求、浏览器需要同时加载多个资源，甚至我们自己也会同时处理多件工作任务。在现代计算机系统中，多核 CPU 和分布式系统的出现，使得并发编程变得越来越重要。并发模型提供了一些基本概念和技术，可以帮助程序员更好地设计和实现并发程序。

  


**常见的并发模型包括多线程、事件驱动、消息传递、用户态协程等。**

1.  多线程是指在同一个进程中同时运行多个线程，每个线程独立执行任务，通过共享内存来进行通信和同步。
1.  事件驱动是一种基于回调函数的编程模型，程序响应事件的触发，从而实现并发执行。
1.  消息传递是一种基于消息队列的编程模型，通过发送和接收消息来实现任务之间的通信和协作。
1.  用户态协程也被称作绿色线程，例如 Golang 中的 goroutine，有自己的栈，可以当作线程来使用，但更轻量化。

  


## 多线程

1.  为每个任务创建新的线程

多线程既简单，又足够通用兼容。我们可以写一个例子：

```Rust
use std::{thread, time::{Duration, Instant}};

fn simulate_download(url: String) -> String {
    thread::sleep(Duration::from_secs(1));
    format!("demo result for {url}")
}

fn main() {
    let start = Instant::now();
    let t1 = thread::spawn(|| simulate_download("site1".to_string()));
    let t2 = thread::spawn(|| simulate_download("site2".to_string()));

    let (result1, result2) = (t1.join().unwrap(), t2.join().unwrap());
    println!("result1: {result1}; result2: {result2}");
    println!("cost {}ms", start.elapsed().as_millis())
}

// result1: demo result for site1; result2: demo result for site2
// cost 1000ms
```

在这个例子中，我们创建了两个线程分别运行模拟的下载请求（它暂停当前线程 1 秒），然后在主线程等待两个进程结束并接收返回结果。可以看出，虽然两个请求都要花费 1 秒，但我们总耗时仍约为 1 秒。

**多线程这么好用，而且不需要修改已有代码，那我们为什么要搞异步呢？核心原因是性能。**

创建线程的开销很大，同时也会增大 kernel 的调度开销。如果执行一个任务就去创建一次线程并销毁，这样是非常浪费资源的，同时也会因为跨核同步开销引入性能瓶颈（创建线程会分配新的线程栈空间，会涉及页表修改）。

  


**那么很容易想到一个优化：为什么不创建线程池呢？这样可以降低线程的频繁创建与销毁。**

  


2.  创建线程池

我们可以手写一个线程池。

1.  整体流程是创建 Worker 线程去接收任务并执行；
1.  在任务执行结束后将结果通过 channel 发送；
1.  主线程同样创建两个任务并利用线程池 spawn，之后等待接收执行结果；
1.  Worker 的销毁通过 Sender drop 来触发。

```Rust
use std::{
    sync::{mpsc, Arc, Mutex},
    thread::{self, JoinHandle},
    time::{Duration, Instant},
};

struct ThreadPool {
    worker: Vec<Worker>,
    sender: Option<mpsc::Sender<Job>>,
}

struct Worker {
    id: usize,
    handler: JoinHandle<()>,
}

type Job = Box<dyn FnOnce() + Send + 'static>;

impl ThreadPool {
    fn new(size: usize) -> Self {
        assert!(size > 0);
        let (sender, receiver) = mpsc::channel();
        let receiver = Arc::new(Mutex::new(receiver));
        let mut worker = Vec::with_capacity(size);
        for id in 0..size {
            worker.push(Worker::new(id, Arc::clone(&receiver)));
        }
        Self {
            worker,
            sender: Some(sender),
        }
    }

    fn execute<F, R>(&self, f: F) -> mpsc::Receiver<R>
    where
        F: FnOnce() -> R + Send + 'static,
        R: Send + 'static,
    {
        let (tx, rx) = mpsc::sync_channel(1);
        let job = Box::new(move || {
            let result = f();
            tx.send(result).unwrap();
        });
        self.sender.as_ref().unwrap().send(job).unwrap();
        rx
    }
}

impl Worker {
    fn new(id: usize, receiver: Arc<Mutex<mpsc::Receiver<Job>>>) -> Self {
        let handler = thread::spawn(move || {
            println!("[Worker] worker {id} is running");
            loop {
                let locked = receiver.lock().unwrap();
                if let Ok(job) = locked.recv() {
                    drop(locked);
                    job();
                } else {
                    break;
                }
            }
            println!("[Worker] worker {id} exit");
        });
        Self { id, handler }
    }
}

impl Drop for ThreadPool {
    fn drop(&mut self) {
        drop(self.sender.take().unwrap());
        while let Some(w) = self.worker.pop() {
            println!("[ThreadPool] waiting for worker {} exit", w.id);
            w.handler.join().unwrap();
        }
    }
}

fn simulate_download(url: String) -> String {
    thread::sleep(Duration::from_secs(1));
    format!("demo result for {url}")
}

fn main() {
    let tp = ThreadPool::new(2);

    let start = Instant::now();
    let r1 = tp.execute(|| simulate_download("site1".to_string()));
    let r2 = tp.execute(|| simulate_download("site2".to_string()));

    let (result1, result2) = (r1.recv().unwrap(), r2.recv().unwrap());
    println!("result1: {result1}; result2: {result2}");
    println!("cost {}ms", start.elapsed().as_millis())
}

// [Worker] worker 0 is running
// [Worker] worker 1 is running
// result1: demo result for site1; result2: demo result for site2
// cost 1000ms
// [ThreadPool] waiting for worker 1 exit
// [Worker] worker 1 exit
// [Worker] worker 0 exit
// [ThreadPool] waiting for worker 0 exit
```

  


但实际场景中，线程池大小怎么设定呢？在后端场景中，一个 Server 可以有几万个活跃连接，而创建几万个线程会大大增加内存占用和调度开销，不是一件可行的事。

  


## 异步

思考下前面的问题，我们真的需要这么多线程同时工作吗？在每个线程都充分利用 CPU 资源的情况下，要达到 CPU 性能上限，N core 只需要 N 个线程就够了，所以绝大多数时候这些线程仅仅是在等待 IO 而已。

那么能不能让线程不要等待，在某个任务不能执行时（比如等待网络 IO）去做别的任务呢？这样可以用少量的线程处理大量的任务，大大提升线程的利用率。

![](https://p3-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/f4bd4158531f4a78b13b7578433d5427~tplv-k3u1fbpfcp-jj-mark:0:0:0:0:q75.image#?w=587&h=335&e=png&b=fcf9f8)

（[Image Src](https://blog.51cto.com/u_15067223/2573472)）

  


异步就是这么一种编程模型，它可以使程序在执行耗时操作时，不会阻塞主线程或者其他操作的进行。在传统的同步编程模型中，程序会一直等待操作完成才会继续往下执行，而在异步编程模型中，程序可以继续执行其他操作，等待异步操作的结果返回后再去处理它。这种方式可以提高程序的并发性和响应速度，特别是在处理密集 IO 操作时更为明显。注意，并发并不等价于异步，异步只是实现并发的一种方式。

  


**不同的语言在处理异步的时候会有自己的倾向，也会带来不同的编程模式。**

-   在较旧版本的 C++ 中往往使用回调函数的形式实现，一般基于 libevent
-   C++ 20 提供 coroutine，支持 co_await、co_yield 和 co_return，允许外部实现 Runtime（无栈）
-   Golang 提供 goroutine，绑定官方实现 Runtime（有栈）
-   Rust 提供 Future、Waker 抽象与 async、await 语法糖，允许外部实现 Runtime（无栈）

  


### 回调模式

在基于回调的模式下，任务的状态是用户自行维护的（或者说没有明确的“任务”这个概念）。用户告诉框架：

1.  入口：执行 A 操作，当 A 好了去做 B；
1.  B：执行 B 操作，做好了去做 C；
1.  C：执行 C 操作，做好了去做 D；
1.  D：做 D 操作。

例如，A、B、C、D 可以对应为 HTTP Server 的读 Header、读 Body、写 Header、写 Body。最终用户用四段同步逻辑组合出了最终想表达的 A->B->C->D。

  


这种模式的问题显而易见：

1.  不直观，连续的逻辑被打断为多个同步段，分散在不同的地方，难以理解。
1.  生命周期不好管理，容易导致执行 callback 执行时相关依赖量已经被释放的问题。

在这套模式下，需要对整套逻辑依赖非常清楚才能写正确，并且很难编程。

而基于下面说的有栈协程或无栈协程，我们可以写出这种易于理解的代码：

```Rust
fn demo(){
    let a = A().await;
    let b = B(a).await;
    let c = C(b).await;
    let d = D(c).await;
}
```

  


### 有栈协程

如果说要提供一种类似线程一样，但由性能较高的模型，那显然就是有栈协程了。有栈协程的核心是为任务（coroutine）分配“栈”空间，在执行前替换当前栈指针，这样对于这个任务而言，就是在用当前线程执行它。任务的状态保存在“栈”上。

  


当遇到 IO 阻塞时，将任务挂至 poller 上（下次会等待相关任务的就绪），然后切换栈寄存器到其他任务并执行。另外，也可以由 monitor 监控任务执行耗时，耗时较久时也可以通过信号强制收回线程控制权并触发任务切换。

  


Golang 较新版本实现了这种抢占式调度，解决了用户执行同步 syscall 或死循环 loop 时无法切换任务的问题。对比线程的方案，如果不是为了充分利用多核性能，在存在异步调用时，有栈协程往往是一定优于启动多线程的。Golang 内嵌了 Runtime，但出于性能和实现方便考虑，并没有允许外部去实现。

### 无栈协程

无栈协程的状态存储在状态机内，每个任务都是一个状态机，它保存所有可能中断的点对应的状态（可理解为一个 enum 结构）。

状态机在执行时某些点可能遇到 IO 阻塞（比如 `.await` 时），此时它只需要返回并表示现在无法执行，依赖的条件不满足；并在依赖条件满足时告诉 Runtime 再来推动一下自己。

  


**在 Rust 中，这两个核心能力是这么暴露的：**

1.  推动任务执行：Future trait，Runtime 会 poll 任务，任务可以返回 Pending 或者 Ready
1.  唤醒任务：poll 任务时会传递 Waker，任务利用 Waker 可以将自己重新调度

  


### 模式对比

对比线程和有栈协程，无栈协程不需要分配和回收“栈”空间，性能较好；其占用的内存大小也是确定的，相比有栈协程分配固定的“栈”空间的内存利用率也较高（有栈协程的栈大小较难处理，分配大了浪费，分配小了要扩栈）。无栈协程的另一个优点是可组合，多个状态机可以很轻松地组合出新的状态机。

  


但无栈协程也有缺点：由于它只能保存固定的中断点时的状态，所以它不能在任意时刻被抢占，要拿到线程的执行权只能等待其自主退出，否则就会丢失任务状态。

  


这要求我们要明确控制在异步函数中的逻辑执行时间，不能包含长时间占用 CPU 的操作，不能发起阻塞式 syscall。比如 Rust 中一个常见的误区是在 async 函数中使用 `std::thread::sleep` 或 `block_on`。

  


站在更高的视角看，有栈协程类似线程，计算逻辑与状态管理分离，状态管理是 Runtime 的责任；无栈协程将状态管理与计算逻辑打包，Runtime 不感知状态，所以也就无法抢占。
