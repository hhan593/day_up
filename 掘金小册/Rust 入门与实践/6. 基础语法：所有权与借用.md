在上节中，我们学习了 Rust 的函数与控制流的知识。在本节中，我们会介绍 Rust 的一个独有的概念——所有权，还会介绍与之相关的概念 — 借用。在学习 Rust 的过程中，所有权系统应该是第一个难关。加油！

  


# 所有权

所有权是 Rust 用来管理内存的一套规则。在其他语言中，有些语言通过垃圾回收来管理内存，垃圾回收会在程序运行时找到所有不再使用的内存并回收；有些语言则需要显式的来申请和释放内存。Rust采用的是不同于这两种方式的第三种方式：通过所有权系统来管理内存。所有权系统的规则会在编译时就被检查，如果任意规则不满足，程序将无法编译通过。由于这些规则都是编译时的检查规则，从而不会拖慢程序运行时的性能。

  


在学习所有权系统之前，首先需要了解一些预备知识。

  


## 预备知识

### 堆和栈

堆和栈都可以在程序运行时给程序提供内存空间，但是他们是有区别的。

  


想象一下，栈就像一叠盘子。先来的盘子在底部，新盘子则放在顶部。取用时，总是先取最上面的盘子，就像在餐馆里洗完的盘子先用先拿。这就是先进后出的原则。但是，盘子的大小必须是确定的，否则无法存放在这个“栈”里。

  


而堆则像一个大杂货仓库。你可以随时放入各种大小和形状的物品。为了方便找到它们，你需要贴上标签，这些标签就像指针，告诉你物品的位置。虽然这样存放物品灵活多变，但要找到合适的空间并读取它们就不如“栈”中的盘子方便了。

  


在性能的赛跑中，栈就像速度迅猛的跑车，堆则更像一辆载重卡车。栈的读写速度快，因为一切都井井有条，就像把盘子放在最上面那样简单。而堆要先找到合适的存放空间，读取时还要通过标签（指针）找到物品，就像在杂货仓库里寻找一个小小的零件一样，可能需要多花点时间。

  


### String 类型

为了方便后续举例说明所有权系统，这里要先介绍一下 String 类型，后续的例子会用到这个类型。

  


在 Rust 编程的世界中，字符串有点像我们现实生活中的名片和便签。有两种类型让我们来认识它们：`String`和`&str`。

想象`&str`是一种名片，它被称为字符串字面值，就像是工厂里生产出来的名片，印好了，就再也不能改了。你可以这样给自己制造一张名片：

```Rust
let s = "hello";
```

  


这张名片是固定的，不可变的，因为它是硬编码到程序中的，就像是直接印在墙上的字。

  


但生活中不是所有的信息都是确定的，有时候我们需要一些便签。这就是`String`的角色了，比如你想要记录下用户的一些临时想法或命令。`String`的数据存储在堆上，就像前面提到的杂货仓库，可以随时变动。你还记得堆和栈的故事吗？

  


创建一个`String`就像是抓一张空白的便签纸，你可以从字符串字面值开始书写：

  


```Rust
let s = String::from("hello");
```

  


由于`String`存储在堆上，像便签纸一样，你可以随时添加更多的文字：

```Rust
let mut s = String::from("hello");

s.push_str(", world!"); // push_str() 在字符串后追加字面值

println!("{}", s); // 打印 `hello, world!`
```

  


## 所有权规则

到此，我们就可以正式开始讲解所有权系统了。首先来看下所有权系统的三条规则：

  


-   Rust 中每个值都有一个所有者
-   一个值同时只能有一个所有者
-   当所有者离开作用域范围，这个值将被丢弃

  


我们通过一个例子来简单说明这三条规则：

```Rust
fn main() {
    // 规则 1：Rust中每个值都有一个所有者
    let s1 = String::from("hello"); // s1 是值 "hello" 的所有者
    
    {
        // 规则 2：一个值同时只能有一个所有者
        let s2 = s1; // 所有权从 s1 转移到 s2（s1 不再有效）
        // println!("{}", s1); // 这会导致编译时错误，因为 s1 不再有效
        println!("{}", s2); // 这是允许的，因为 s2 现在拥有该值
    } // s2在此处超出范围

    // 规则 3：当所有者离开作用域范围，这个值将被丢弃
    // 由于 s2 超出范围，因此为值 "hello" 分配的内存在此处自动释放。
}
```

  


在此示例中，演示了 Rust 所有权系统的三个规则：

1.  Rust 中每个值都有一个所有者：`String`值 "hello" 由变量 `s1` 拥有。
1.  一个值同时只能有一个所有者：所有权从 `s1` 转移到 `s2`，使 `s1` 不再有效。在此之后尝试使用 `s1` 将导致编译时错误。
1.  当所有者离开作用域范围，这个值将被丢弃：当 `s2` 在内部块的末尾超出范围时，为值 "hello" 分配的内存将自动释放。

  


对于上例中的第二点，所有权发生了从`s1`到`s2`的转移。下面我们会更详细的介绍一下所有权的转移。

  


## 所有权转移

  


先来看一个简单的例子：

```Rust
let x = 5;
let y = x;
```

  


这段代码中，首先将 5 绑定到 x，接着再将 x 的值拷贝给 y。这两行执行完， x 和 y 都是 5，且都可以正常使用。稍稍改变一下这个例子。

```Rust
let s1 = String::from("hello");
let s2 = s1;
```

  


我们把数据类型从`i32`变成了`String`，看起来跟刚才的例子没有太大对吧？但是实际上，背后发生的事情千差万别。

  


对于类似`i32`这样的简单类型，赋值的时候 Rust 会自动进行拷贝（Copy）。而对于 String 这样的分配到堆上的复杂类型，发生的却是所有权的转移，而不是拷贝。为了理解这一点，我们需要看一下 String 在内存中是什么样的。

![](https://p3-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/a5b794d1fe86469d9c09af1de94336d4~tplv-k3u1fbpfcp-jj-mark:0:0:0:0:q75.image#?w=3070&h=2442&e=png&a=1&b=000000)

  


如上图左半部分，一个 String 类型实际上由三部分组成，一个指针指向实际存储 String 内容的位置，一个 len 表示长度，一个 capacity 表示容量。左半部分是存储在栈上的。而右半部分表示的是 String 的实际内容，存储在堆上。

  


当把`s1`的值赋给`s2`时，左半部分的 String 结构会被拷贝。这时候内存中`s1`和`s2`是这样的：

![](https://p3-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/1aa0551ae677444d884229a9eae9b9b2~tplv-k3u1fbpfcp-jj-mark:0:0:0:0:q75.image#?w=3070&h=3400&e=png&a=1&b=000000)

  


  


还记得我们前面提到的所有权的三条规则吗？第三条规则是当所有者离开作用域范围，值将被丢弃（drop）。上图中`s1`和`s2`都指向同一个值，当他们离开作用域时，都会去执行 drop 的动作。这就产生了**双重释放**（double free）这一内存安全问题。

  


因此 Rust 会进行所有权的转移，在`let s2 = s1;`执行完之后，所有权会从`s1`转移给`s2`，从而`s1`不再有效。这样一来，上述 String 值只会在`s2`离开作用域的时候被释放。所有权转移之后内存的示意图如下：

  


![](https://p3-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/7b70c2f9b9d84a83a69148d991be3f1e~tplv-k3u1fbpfcp-jj-mark:0:0:0:0:q75.image#?w=3075&h=3400&e=png&a=1&b=d0cece)

  


  


`s1`被标记成灰色，表示不再有效。

  


有同学可能会问，如果我把 String 实际的内容也拷贝一份，那是不是也能满足 Rust 对所有权的要求？如果这样做的话，内存会变成这样：

![](https://p3-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/e967d813230f4ebdb5732dd3091ea9bb~tplv-k3u1fbpfcp-jj-mark:0:0:0:0:q75.image#?w=3125&h=4963&e=png&a=1&b=000000)

  


  


这个行为在 Rust 当中叫做克隆（Clone），有点类似其他语言的深拷贝。这样做确实可以满足 Rust 对于所有权的要求。但是如果这样做，`s2 = s1;`这样的行为开销会变得很大。因此 Rust 所有的自动的复制都不是深拷贝。

  


那如果我就想创建深拷贝要怎么办呢？通过`clone`方法可以实现：

```Rust
let s1 = String::from("hello");
let s2 = s1.clone();

println!("s1 = {}, s2 = {}", s1, s2);
```

可以看到不同于前面所有权转移的例子，在clone之后，s1和s2都是可以正常使用的。

  


还记得最早举的这个例子吗？

```Rust
let x = 5;
let y = x;

println!("x = {}, y = {}", x, y);
```

  


我们前面说过，对于`i32`这样的简单类型，Rust 会在赋值的时候自动拷贝。这是因为像这类简单类型都是存储在栈上的，对他们执行拷贝动作开销很低。因此不同于 String 的所有权转移，在赋值完成之后，x 和 y 都可以继续正常使用。

  


在后续的章节中，会讲到 Rust 的`Copy`特征。如果一个类型拥有`Copy`特征，那么旧的值在赋值给其他变量之后，依然可以使用。

  


## 函数传值和返回

将值传给函数跟上面讲的赋值类似，都会进行转移或者拷贝的过程。函数返回一个值的时候，也会经历所有权转移的过程。我们用下面的例子来说明：

```Rust
fn takes_ownership(s: String) {
    println!("Received string: {}", s);
} // s 离开作用域，被丢弃

fn gives_ownership() -> String {
    String::from("hello")
} // 返回了String的所有权

fn main() {
    let s = String::from("hello");
    takes_ownership(s); // s转移到了函数内，不再可用

    // s 不再可用

    let s = gives_ownership(); // s 获得了返回值的所有权
}
```

  


当把 String 类型的`s`传给`takes_ownership`时，`s`的所有权也跟着转移了，因此在外部不再可用。随着`takes_ownership`函数执行完毕，`s`离开了作用域，被丢弃。另一个函数`gives_ownership`则展示了函数返回 String 的时候，所有权会跟着一起被返回，被外部的`s`获取。

  


有同学要问了，如果经历了函数传参，所有权就丢了，那代码不是很难写？我就是要在函数调用之后再用这个变量怎么办？这就要讲到本节课程的第二部分——借用。

  


# 引用与借用

刚才说到，按照前文`takes_ownership`函数的方式传参之后，所有权也会跟着转移。这样的话原来的变量在函数调用之后就不再能使用了。那么有没有办法只使用变量，但是不拿走所有权呢？这就有了这部分要介绍的“借用”的概念。

  


## 不可变引用

想象你有一辆自行车名为 "hello"（s1），你可以将它借给你的朋友，但最终它仍然是你的自行车。你的朋友不允许把它带走，但完全可以测量看看它有多长。让我们看看下面的例子：

```Rust
fn main() {
    let s1 = String::from("hello");
    let len = calculate_length(&s1);
    println!("The length of '{}' is {}.", s1, len);
}

fn calculate_length(s: &String) -> usize {
    s.len()
}
```

  


这个例子中，`calculate_length` 函数接受的参数类型是 `&String`，这是 `String` 类型的引用，或者说你的朋友借用你的自行车。实际传入的参数是 `&s1`，即你借出的自行车。通过这样的方式，`calculate_length` 可以使用 `s1` 进行测量，但是并没有拿走它的所有权。就像你的朋友能够借用并了解自行车的长度，但最终还是归还给你，`s1` 在函数执行完成之后，依然可以正常使用。

  


通过下面这张图，我们可以看一下在传参过程中发生了什么。

![](https://p3-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/f99473abe6ff454c8f9df22ba4633793~tplv-k3u1fbpfcp-jj-mark:0:0:0:0:q75.image#?w=4867&h=2442&e=png&a=1&b=000000)

  


在给`s`传入`&s1`的时候，实际是创建了一个指向`s1`的引用，这个引用没有值的所有权。因此当函数体内的`s`超出作用域范围被丢弃时，不会导致`String`被丢弃。

  


创建引用的这个动作就被称为借用。

  


## 可变引用

  


如果我们尝试去修改引用的值（朋友尝试改装你的自行车），那会发生什么？看下面的例子：

```Rust
fn main() {
    let s = String::from("hello");
    change(&s);
    println!("The updated string is: {}", s);
}

fn change(s: &String) {
    s.push_str(", world!");
}
```

  


尝试编译，得到下面的报错：

```Bash
error[E0596]: cannot borrow `*s` as mutable, as it is behind a `&` reference
 --> src/main.rs:8:5
  |
8 |     s.push_str(", world!");
  |     ^^^^^^^^^^^^^^^^^^^^^^ `s` is a `&` reference, so the data it refers to cannot be borrowed as mutable
  |
help: consider changing this to be a mutable reference
  |
7 | fn change(s: &mut String) {
  |              ~~~~~~~~~~~

For more information about this error, try `rustc --explain E0596`.
error: could not compile `playground` due to previous error
```

  


还记得我们在变量声明那一章讲过，Rust 中声明的变量默认是不可变变量吗？同样的，默认的引用也是不可变的引用。只有加了`mut`关键字，才能变成可变引用。你必须告诉你的朋友他们可以改装自行车，他们才可以进行操作。

  


修改上述报错的代码：

```Rust
fn main() {
    let mut s = String::from("hello");
    change(&mut s);
    println!("The updated string is: {}", s);
}

fn change(s: &mut String) {
    s.push_str(", world!");
}
```

  


首先，你需要声明你的自行车（`s`）是可修改的，所以使用了 `mut` 关键字。然后通过 `&mut s` 创建了一个可变引用，同时 `change` 函数接受的参数类型也变成了 `&mut String`。这样一来，`change` 函数就可以利用这个可变引用来装饰你的自行车。

  


关于可变引用和不可变引用有一个很重要的规则：

-   对于一个变量，同时只能存在一个可变引用或者多个不可变引用

  


我们举个例子来说明：

```Rust
fn main() {
    let mut s = String::from("hello");

    // 多个不可变引用是允许的
    let r1 = &s;
    let r2 = &s;
    println!("r1: {}, r2: {}", r1, r2);
    // 在这里，多个不可变引用是允许的，因此打印不会引发错误。

    // 一个可变引用
    let r3 = &mut s;
    println!("r3: {}", r3);
    // 在这里，只有一个可变引用，因此打印不会引发错误。

    // 不允许同时存在可变引用和不可变引用
    // let r4 = &s; // 这会导致编译时错误
    // println!("r3: {}, r4: {}", r3, r4);
    // 如果取消注释上面两行，同时存在可变引用和不可变引用将导致编译时错误。
}
```

  


在上面的例子中，展示了这样几种情况：

1.  多个不可变引用是允许的：我们可以同时创建`r1`和`r2`，它们都是`s`的不可变引用。
1.  同时只能有一个可变引用：我们可以创建一个名为 `r3` 的可变引用，但在此期间不能再创建其他引用（无论是可变还是不可变）。
1.  同时存在可变引用和不可变引用是不允许的：如果我们尝试创建一个名为 `r4` 的不可变引用，当 `r3` 还处于活动状态时，这将导致编译时错误。

  


有细心的同学可能会问，不对啊，`r1`和`r2`要到`main`函数结束才离开作用域，那创建`r3`的时候不就同时有可变引用和不可变引用吗？这就要讲到 Rust 编译器的一个优化行为——Non-Lexical Lifetimes(NLL)

  


## NLL

  


在老版本的 Rust 编译器中（1.31之前），确实上述的`r1`,`r2`和`r3`是会报错的。但是这样其实会带来很多麻烦，导致代码很难写。于是 Rust 编译器做了一项优化：**引用的作用域结束的位置不再是花括号的位置，而是最后一次使用的位置**。因此，在现在 Rust 的版本中，上面的例子并不会报错。

  


## 悬垂引用

  


悬垂引用指的是指针指向的是内存中一个已经被释放的地址，这在其他的一些有指针的语言中是很常见的错误。而 Rust 则可以在编译阶段就保证不会产生悬垂引用。也就是说，如果有一个引用指向某个数据，编译器能保证在引用离开作用域之前，被指向的数据不会被释放。

  


来看下面这段代码：

```Rust
fn main() {
    let reference_to_nothing = dangle();
}
fn dangle() -> &String {
    let s = String::from("hello");

    &s
}
```

  


尝试编译，会报下面的错：

```Plain Text
error[E0106]: missing lifetime specifier
 --> src/main.rs:5:16
  |
5 | fn dangle() -> &String {
  |                ^ expected named lifetime parameter
  |
  = help: this function's return type contains a borrowed value, but there is no value for it to be borrowed from
help: consider using the `'static` lifetime
  |
5 | fn dangle() -> &'static String {
  |                 +++++++

For more information about this error, try `rustc --explain E0106`.
error: could not compile `playground` due to previous error
```

  


报错信息中提到了生命周期（lifetime）这个概念我们后面的章节会介绍。重点看下面这句话：

```Plain Text
this function's return type contains a borrowed value, but there is no value for it to be borrowed from
该函数返回了一个借用的值，但是没有可以借用的来源
```

  


上述代码的`dangle`函数中，首先创建了一个`String`类型的`s`，之后又返回了`s`的引用。但是`s`在离开函数`dangle`之后就离开作用域了，会被丢弃。此时返回的`s`的引用已经指向的是不合法的位置了。Rust编译器检查到了这个问题，从而报错。

  


## 引用规则总结

  


关于引用，本文介绍的规则总结来说就是这两条：

-   对于一个变量，同时只能存在一个可变引用或者多个不可变引用
-   引用必须是有效的

  


介绍完引用与借用，最后我们再来介绍一种特殊的引用：切片（slice）。

  


# 切片

切片可以让我们引用集合中的一段连续空间。切片也是一种引用，因此没有所有权。

  


## 字符串切片

我们来看下面这段代码：

```Rust
    let s = String::from("hello world");

    let hello = &s[0..5];
    let world = &s[6..11];
```

  


上面例子中的`hello`和`world`就是两个`s`的切片。其中`[0..5]`这种写法，表示`[0,5)`这个范围都在切片内。注意这里是左闭右开，也就是说下标0是切片的起始位置，下标4是切片的结束位置。类似的`world`这个切片就是从下表6开始，到下标10结束，长度为5。见下图：

关于`..`这个语法，Rust 还提供了一些简化的写法，例如：

```Rust
let s = String::from("hello");
let len = s.len();

//以0开始时，0可以省略
let slice = &s[0..2];
let slice = &s[..2];

//以最后一位结束时，len可以省略
let slice = &s[3..len];
let slice = &s[3..];

//同时满足上述两条，那么两头都可以省略
let slice = &s[0..len];
let slice = &s[..];
```

  


## 其他切片

  


除了 String，数组类型也有切片。例如：

```Rust
let a = [1, 2, 3, 4, 5];
let slice = &a[1..3];
assert_eq!(slice, &[2, 3]);
```

  


`slice`这边切片就包含了 2 和 3 这两个数字。

  


# 总结

本节内容较多，主要包含了三部分的知识：所有权，借用和切片。所有权这套系统是 Rust 内存安全的重要保障。有了这套系统，我们既可以享受不需要手动释放内存的便利，又可以对内存使用有足够的控制，保证内存安全。
