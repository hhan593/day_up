# Waker

在 Rust 的异步编程中，任务通常被封装在 `Future` 类型中。当任务处于等待状态时，它们将被挂起，直到某个条件得到满足。这个条件可能是一个计时器到期、一个 I/O 操作完成或者其他任何异步事件。

在这种情况下，`Waker` 被用来通知任务它可以恢复执行了。当某些条件得到满足时，`Waker` 将被调用，唤醒任务并使其继续执行。这个过程通常由异步运行时系统自动处理，但你也可以手动创建和管理 `Waker` 实例。

通常情况下，只有当组件作为事件源的时候才需要感知 Waker 的存在——例如写一个异步 channel，或者写一个异步网络组件：异步 channel 需要在有空余空间时叫醒等待中的 send 任务，并在有数据时叫醒等待中的 recv 任务；网络组件需要感知 fd 就绪状态然后叫醒对应的 read 或 write 任务。

  


Waker 通常由 Runtime 生成，一般会与 task 一一绑定（async 逻辑最终生成为单个状态机，spawn 出去的部分不包含在内）。

  


Runtime 通过 vtable 的形式在固定类型的情况下为 Waker 生成自定义行为。

```Rust
pub struct Waker {
    waker: RawWaker,
}

impl Waker {
    pub fn wake(self) {
        ...
    }

    pub fn wake_by_ref(&self) {
        ...
    }

    pub fn will_wake(&self, other: &Waker) -> bool {
        ...
    }
    
    pub const unsafe fn from_raw(waker: RawWaker) -> Waker {
        ...
    }
    
    pub fn as_raw(&self) -> &RawWaker {
        ...
    }
}

pub struct RawWaker {
    data: *const (),
    vtable: &'static RawWakerVTable,
}
pub struct RawWakerVTable {
    clone: unsafe fn(*const ()) -> RawWaker,
    wake: unsafe fn(*const ()),
    wake_by_ref: unsafe fn(*const ()),
    drop: unsafe fn(*const ()),
}
```

通常来讲，`wake_by_*` 会被实现为将 Task 加入可调度队列，稍后该任务即可被 Runtime 执行。当不再需要这个 waker 时，我们调用 `wake` 方法可以节省潜在的 clone 开销；当需要存储 waker 时，我们先通过 `will_wake` 判断当前存储的 waker 是否与即将覆盖的 waker 一致，也能避免潜在的 clone 开销。

  


当然，Waker 的设计也存在一些问题。当前对 Waker 实现的约束是，它必须能够安全地 Send/Sync —— 但这其实限制了用户对线程模式的选型。关于该讨论的参考链接：https://without.boats/blog/wakers-ii/

  


## 实现一个 yield_now

同步模型下，当我们需要短暂等待时，我们会 spin，这在实现锁的时候很常见，因为对于竞争较弱的场景，大概率通过简单 spin 即可获得锁，这比通过 futex 等跨线程通信手段要廉价的多。

`yield_now` 可以让我们在异步生态中低成本地 spin。

  


它的实现原理是：当判定需要 spin 时，直接原地 wake Waker，然后返回 Pending。稍后该任务还将被再次执行，再次执行时，它返回 Ready。

```Rust
pub async fn yield_now() {
    struct YieldNow {
        yielded: bool,
    }

    impl Future for YieldNow {
        type Output = ();

        fn poll(mut self: Pin<&mut Self>, cx: &mut Context<'_>) -> Poll<()> {
            if self.yielded {
                return Poll::Ready(());
            }

            self.yielded = true;

            cx.waker().wake_by_ref();

            Poll::Pending
        }
    }

    YieldNow { yielded: false }.await
}
```

  


## 实现一个 Oneshot Channel

这里我们实现一个简单的 oneshot channel，它具备以下功能：

1.  一次性的 value 发送和接收
1.  当发送端 close/drop 时，接收端能够退出

  


1.  首先我们实现一个共享的数据结构 Shared，它存储 Waker、T 和是否关闭：

```Rust
struct Shared<T> {
    waker: Option<Waker>,
    value: Option<T>,
    closed: bool,
}
```

2.  之后我们基于 Shared 实现 Sender 和 Receiver：

```Rust
struct Sender<T> {
    shared: Arc<Mutex<Shared<T>>>,
}

struct Receiver<T> {
    shared: Arc<Mutex<Shared<T>>>,
}
```

-   对于 Sender，我们要提供 send 方法。由于是 oneshot channel，所以可以确保有空间存储值，send 方法可以被定义为同步方法：

```Rust
impl<T> Sender<T> {
    fn send(&self, data: T) {
        let maybe_waker = {
            let mut inner = self.shared.lock().unwrap();
            inner.value = Some(data);
            inner.closed = true;
            inner.waker.take()
        };
        if let Some(waker) = maybe_waker {
            waker.wake();
        }
    }
}
```

-   当 Sender drop 时，我们也需要及时唤醒可能的接收方：

```Rust
impl<T> Drop for Sender<T> {
    fn drop(&mut self) {
        let mut inner = self.shared.lock().unwrap();
        inner.closed = true;
        if let Some(waker) = inner.waker.take() {
            waker.wake();
        }
    }
}
```

-   对于 Receiver，我们要提供 recv 方法 —— 这个是异步的。出于通用性考虑，我们往往会提供一个 poll 风格的方法，并基于这个实现 async 版本。

```Rust
impl<T> Receiver<T> {
    fn poll_recv(self: Pin<&Self>, cx: &mut Context<'_>) -> Poll<Option<T>> {
        let mut inner = self.shared.lock().unwrap();
        if !inner.closed {
            if let Some(w) = inner.waker.as_ref() {
                if w.will_wake(cx.waker()) {
                    return Poll::Pending;
                }
            }
            inner.waker = Some(cx.waker().clone());
            return Poll::Pending;
        }
        Poll::Ready(inner.value.take())
    }

    async fn recv(&self) -> Option<T> {
        poll_fn(|cx| unsafe { Pin::new_unchecked(self) }.poll_recv(cx)).await
    }
}
```

3.  最后我们提供以下 channel 函数来创建这对 tx、rx 即可，完整代码展示如下：

```Rust
use std::{
    pin::Pin,
    sync::{Arc, Mutex},
    task::{Context, Poll, Waker},
};

use tokio::macros::support::poll_fn;

#[tokio::main]
async fn main() {
    // send and recv
    let (tx, rx) = channel::<u8>();
    tx.send(1);
    println!("recv: {:?}", rx.recv().await);

    // recv and send
    let (tx, rx) = channel::<u8>();
    let t = tokio::spawn(async move { println!("recv: {:?}", rx.recv().await) });
    tx.send(2);
    let _ = t.await;

    // recv and close
    let (tx, rx) = channel::<u8>();
    let t = tokio::spawn(async move { println!("recv: {:?}", rx.recv().await) });
    drop(tx);
    let _ = t.await;
}

fn channel<T>() -> (Sender<T>, Receiver<T>) {
    let shared = Arc::new(Mutex::new(Shared {
        waker: None,
        value: None,
        closed: false,
    }));
    let sender = Sender {
        shared: shared.clone(),
    };
    let receiver = Receiver { shared };
    (sender, receiver)
}

struct Shared<T> {
    waker: Option<Waker>,
    value: Option<T>,
    closed: bool,
}

struct Sender<T> {
    shared: Arc<Mutex<Shared<T>>>,
}

struct Receiver<T> {
    shared: Arc<Mutex<Shared<T>>>,
}

impl<T> Sender<T> {
    fn send(&self, data: T) {
        let maybe_waker = {
            let mut inner = self.shared.lock().unwrap();
            inner.value = Some(data);
            inner.closed = true;
            inner.waker.take()
        };
        if let Some(waker) = maybe_waker {
            waker.wake();
        }
    }
}

impl<T> Drop for Sender<T> {
    fn drop(&mut self) {
        let mut inner = self.shared.lock().unwrap();
        inner.closed = true;
        if let Some(waker) = inner.waker.take() {
            waker.wake();
        }
    }
}

impl<T> Receiver<T> {
    fn poll_recv(self: Pin<&Self>, cx: &mut Context<'_>) -> Poll<Option<T>> {
        let mut inner = self.shared.lock().unwrap();
        if !inner.closed {
            if let Some(w) = inner.waker.as_ref() {
                if w.will_wake(cx.waker()) {
                    return Poll::Pending;
                }
            }
            inner.waker = Some(cx.waker().clone());
            return Poll::Pending;
        }
        Poll::Ready(inner.value.take())
    }

    async fn recv(&self) -> Option<T> {
        poll_fn(|cx| unsafe { Pin::new_unchecked(self) }.poll_recv(cx)).await
    }
}
```

  


## Wake 分发

通常，异步事件的触发是一对一的，例如上面实现的 channel：当发送端有数据输入时，接收端会被唤醒，实际唤醒的任务是 poll_recv 是带的 cx 内的 waker 所绑定的任务。

  


但是，借助 waker 我们能做到的远不止这些。例如，我们想要实现一个多路复用机制，基于一条连接暴露多条异步连接 —— 每条连接都可以独立异步 read/write，那么我们就需要分别存储每条连接上的读、写对应的 Waker，并在处理数据包后修改状态、分发数据并唤醒任务，例如 [tokio-tungstenite](https://github.com/snapview/tokio-tungstenite/tree/master) 中利用 WakerProxy 分发读、写唤醒： https://github.com/snapview/tokio-tungstenite/blob/master/src/compat.rs#L106 。
