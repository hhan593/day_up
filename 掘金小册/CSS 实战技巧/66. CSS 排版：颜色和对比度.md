![](https://p3-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/2bc01caf049348ce8fc1d6d0e8b637ea~tplv-k3u1fbpfcp-jj-mark:0:0:0:0:q75.image#?w=1920&h=1080&s=80506&e=png&b=e5efff)

  


颜色对比度指的是文本与背景之间的明暗差异。良好的对比度有助于提高文本的可读性，特别是对于视力有障碍的用户来说。WCAG 提出了对比度要求，帮助确保网站对不同能力的用户都能访问：

  


![](https://p3-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/2398136a70294eadb3ab6fd8faea81ba~tplv-k3u1fbpfcp-jj-mark:0:0:0:0:q75.image#?w=3200&h=1800&s=100429&e=png&b=f6f3f1)

  


在此之前，CSS 无法自动指定在任意背景上都能保证可读性的文本颜色，例如在较暗的背景上使用白色文本，而在较亮的背景上使用黑色文本。这也算是 CSS 长期以来的一个主要痛点吧。庆幸的是，[CSS 新增了 `contrast-color()` 函数](https://juejin.cn/book/7223230325122400288/section/7239644346935410720)，允许我们可以更轻松地在动态背景上提供可读性更强的文本颜色，从而大幅提升样式的可用性和灵活性。

  


除此之外，CSS 还新增了“**[相对颜色](https://juejin.cn/book/7223230325122400288/section/7231515598088306720#heading-14)**”（Relative Color Syntax ，简称 RCS）特性，它允许我们通过在任意支持的颜色空间中对颜色进行数学运算，从现有的颜色值中派生出新的颜色。甚至可以用来模拟出 `contrast-color()` 函数的功能（或至少很接近），可以用来作为 `contrast-color()` 函数的备用方案（优雅降级）。因为，到目前为止，支持相对颜色的浏览器要比支持 `contrast-color()` 函数的多。

  


接下来，我们来看看这两种方式是如何帮助我们创建颜色对比度，以及相对颜色又是如何模拟 `contrast-color()` 函数？在模拟 `contrast-color()` 函数的过程中需要考虑哪些东西，又如何在这些东西中找到最佳平衡点？

  


为了更好的理解接下来的内容，你需要对以下基础知识有所了解：

  


-   [现代 CSS 中的颜色格式： RGB，HSL，HWB，LAB 和 LCH](https://juejin.cn/book/7223230325122400288/section/7231515598088306720)
-   [新的 CSS 颜色空间：为 Web 设置高清颜色](https://juejin.cn/book/7223230325122400288/section/7233227753909125178)
-   [CSS 中的 OKLCH 和 OKLAB](https://juejin.cn/book/7223230325122400288/section/7235205520154427429)
-   [颜色对比度：`color-contrast()`](https://juejin.cn/book/7223230325122400288/section/7239644346935410720)
-   [CSS 比较函数：`clamp()`](https://juejin.cn/book/7223230325122400288/section/7241401565653762108)
-   [CSS 自定义属性](https://juejin.cn/book/7223230325122400288/section/7249357815410589733)

## CSS 的 `contrast-color()` 函数

  


![](https://p3-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/c099403f3a644243b56a547933af6e81~tplv-k3u1fbpfcp-jj-mark:0:0:0:0:q75.image#?w=3312&h=1708&s=486290&e=jpg&b=ffffff)

  


我想大家对于上图应该不会感到陌生吧。我们可以在 Github 上创建标签，在创建标签的时候可以选择任意颜色，而 Github 会自动为你选择文本颜色，但这种选择往往效果不佳——**标签的文本颜色和背景颜色的对比度不够**！

  


在 Github 上创建标签时，你只能决定标签的背景颜色，文本颜色是由平台帮你决定的。这意味着你无法控制文本的颜色。然而，在平时开发过程中，你可能会选择为文本和背景分别定义单独的 CSS 变量，但这样会降低 Web 组件的可用性，让别的开发者在使用时更加麻烦。如果你开发的 Web 组件能够直接使用合理的默认值，并且只需在极少数情况下覆盖，那不是更好吗？

  


![](https://p3-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/53b19d7016084b65bf3e5c95a3a3fd37~tplv-k3u1fbpfcp-jj-mark:0:0:0:0:q75.image#?w=1720&h=960&s=78378&e=png&b=f0f4fa)

  


是的，现在我们可以通过 CSS 的 `contrast-color()` 函数来实现—— Web 开发者可以轻松地在动态背景上提供可读性更强的文本颜色，例如在较暗的背景上使用白色文本，而在较亮的背景上使用黑色文本。

  


`contrast-color()` 函数的使用非常简单：

  


![](https://p3-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/bd5d5e7efa244748b4fcfe1e185dc7b6~tplv-k3u1fbpfcp-jj-mark:0:0:0:0:q75.image#?w=2000&h=2407&s=551854&e=jpg&b=9cceb5)

  


我们先来看不带“目标对比度”的示例：

  


```CSS
article {
    --article-bg: #0a9396;
    --article-color: color-contrast(var(--article-bg) vs #FFF, #000);

    background: var(--article-bg);
    color: var(--article-color);

    ::selection {
        background: color-contrast(var(--article-color) vs #FFF, #000);
    }
}
```

  


![](https://p3-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/bb8aa79acc1d4e6487d95abd9ec0b2e2~tplv-k3u1fbpfcp-jj-mark:0:0:0:0:q75.image#?w=1188&h=650&s=1559704&e=gif&f=136&b=000f16)

  


> Demo 地址：https://codepen.io/airen/full/LEPmzBg

  


不难发现，当你调整 `--article-bg` 自定义属性值时，`--article-color` 的值会是 `#fff` 和 `#000` 两者中的一个，因为 `contrast-color()` 会将 `#fff,#000` （颜色列表）与基本色（`--article-color`）进行对比，它会选择对比度更高的值。

  


另外一种更高级的用法是，给 `contrast-color()` 指定颜色的目标对比度：

  


```CSS
:root {
    --body-base: #001219;
}

article {
    --contrast-target: AA;
    --color-set: #001219, #005f73, #0a9396, #94d2bd, #e9d8a6, #ee9b00, #ca6702,#bb3e03, #ae2012, #9b2226;
    --base: color-contrast(var(--body-base) vs var(--color-set) to var(--contrast-target));

    background: var(--base);
    
    > * {
        color: color-contrast(var(--base) vs var(--color-set), #fff to AA-Large);
    }
}
```

  


![](https://p3-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/3c628cbad63248d49de0357b019ca6a2~tplv-k3u1fbpfcp-jj-mark:0:0:0:0:q75.image#?w=1190&h=614&s=2446556&e=gif&f=366&b=000f16)

  


> Demo 地址：https://codepen.io/airen/full/KwPRyKq

  


在这个示例中，`color-contrast()` 函数引用了第三个参数——**目标对比度**。该参数可以是关键字（如 `AA`、`AA-large`、`AAA` 和 `AAA-large`）或一个具体的数字。当定义了目标对比度时，将从颜色列表中选择第一个满足或超过该目标的颜色。

  


有一点需要知道的是，当函数将基础颜色与颜色列表进行比较且未指定目标对比度时，它会选择对比度最高的值。然而，最高的对比度比值并不意味着它是可访问的。

  


```CSS
h1 {
  background: #000;
  color: color-contrast(#000 vs #111, #222);
}
```

  


在这个示例中，黑色背景（`#000`）与两个深灰色进行比较。虽然 `#222` 具有“最高”的对比度比值，但将其与黑色搭配却并不理想。

  


在不久的将来，我们还可以直接像下面这样使用 `contrast-color()` ：

  


```CSS
.ele {
    background: var(--color);
    color: contrast-color(var(--color));
}
```

  


是不是很棒？当然，在规范的时间尺度里，“不久”通常意味着几年。

  


虽然 `color-contrast()` 函数在处理颜色对比度方面非常优秀，但它也有一些缺陷存在。比如，使用 `color-contrast()` 函数根据背景颜色确定文本颜色时，它只比较颜色本身，并不会考虑可能影响视觉对比的其他样式，例如字体大小、字体粗细和透明度等。

  


这意味着，尽管颜色组合在技术上满足了特定的对比度阈值，但如果字体过小、太细或透明度过高，最终仍可能导致文本难以阅读。

  


正如你所看到的，`contrast-color()` 函数正是为了解决颜色对比度这个问题而设计的，然而，到目前为止还没有得到更多主流浏览器的支持。这意味着，你暂时还不能用于实际的生产环境中。庆幸的是，CSS 是具有魔性的，虽然不能直接使用 `contrast-color()` 函数来设置处理颜色的对比度，但 CSS 提供了其他的方案可以实现类似的效果。比如 CSS 的相对颜色（RCS）。

  


> 温馨提示，如果你想更深入的了解 `contrast-color()` 函数，请移步阅读《[颜色对比度：`contrast-color()` ](https://juejin.cn/book/7223230325122400288/section/7239644346935410720)》！

  


## CSS 相对颜色（RCS）

  


![](https://p3-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/7cbf6d47d7914eab8cb73cbdfbd491e4~tplv-k3u1fbpfcp-jj-mark:0:0:0:0:q75.image#?w=1600&h=965&s=192804&e=png&b=f7fbfe)

  


“**[相对颜色](https://juejin.cn/book/7223230325122400288/section/7231515598088306720#heading-14)**”（RCS）无疑是 **[现代 CSS](https://s.juejin.cn/ds/iywFJSMA/)** 中另一个强大的功能。简单地说，它允许 Web 开发者通过在任意支持的[颜色空间](https://juejin.cn/book/7223230325122400288/section/7231515598088306720)中对颜色进行数学运算，从现有的颜色值中派生出新的颜色：

  


![](https://p3-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/95d408d59a77462ea74d97bd69c3594b~tplv-k3u1fbpfcp-jj-mark:0:0:0:0:q75.image#?w=1660&h=1340&s=369089&e=jpg&b=ffffff)

  


上图所展示的仅仅是使用相对颜色来生成色调的用例，但这只是相对颜色所能实现功能的冰山一角。接下来，我们来一起探索相对颜色的另一个更高比的功能——**使用相对颜色来模拟** **`contrast-color()`** **函数的功能**！

  


## 使用 RCS 自动计算对比度适合的文本颜色

  


接下来的，我们将使用 **[OKLCH 颜色空间](https://juejin.cn/book/7223230325122400288/section/7235205520154427429)**，这是 CSS 支持的最符合感知均匀性的极坐标颜色空间：

  


![](https://p3-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/aff8a83e4a7344f6a3a005a8df488cfd~tplv-k3u1fbpfcp-jj-mark:0:0:0:0:q75.image#?w=2742&h=1780&s=372652&e=jpg&b=f2f1f6)

  


> URL: https://oklch.com

  


在感知均匀的颜色空间中，两个颜色之间的[欧几里得距离](https://en.wikipedia.org/wiki/Euclidean_distance)与它们的感知差异成正比。RGB 颜色空间（及其极坐标形式，如 HSL、HSV、HSB 和 HWB 等）通常不是感知均匀的。具有感知均匀的颜色空间包括 Lab、LCH、OKLAB 和 OKLCH。

  


极坐标颜色空间是使用角度（即色相 Hue）表示颜色的核心，并通过颜色的饱和度和亮度来控制该颜色的具体色调：

  


![](https://p3-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/a67828fe8ba449e7a2e3e247143ff700~tplv-k3u1fbpfcp-jj-mark:0:0:0:0:q75.image#?w=2400&h=1800&s=951190&e=png&a=1&b=e5ec16)

  


假设我们有一个亮度阈值，当亮度高于该值时，无论饱和度和色调如何，黑色文本都能被清晰读取；反之，当亮度低于该值时，白色文本能够被清晰读取。

  


接下来，我们来证明这个亮度阈值的确存在，但现在我们先用 `0.7` （即 `70%`）作为默认值。我们可以使用 [CSS 自定义属性](https://juejin.cn/book/7223230325122400288/section/7249357815410589733) `--l-threshold` 来存储它，以便后续调整：

  


```CSS
:root {
    --l-threshold: 0.7; /* 亮度的阈值 */
}
```

  


刚才提到过，当亮度小于“假设的阈值”时，黑色文本都能被清晰读取；反之则白色文本能被清晰读取。为了便于描述，同样用 CSS 自定义属性 `--l` 来描述颜色的亮度。这样一来，我们可以通过下面的方式来描述 `--l` 和 `--l-threshold` 两者的关系：

  


-   当 `--l` 大于 `--l-threshold` 时返回 `0` （黑色文本可被清晰读取）
-   当 `--l` 小于或等于 `--l-threshold` 时返回 `1` （白色文本可被清晰读取）

  


现在，我们需要一个表达式——用作颜色的亮度值：

  


```CSS
:root {
    --l: ❓;
    --l-threshold: 0.7; /* 亮度的阈值 */
    --c: oklch(var(--l) 0 0); 
}
```

  


关键是 “❓ ”这个表达式如何获取？一种方法是放宽对表达式返回值的要求。实际上，我们并不需要精确的 `0` 或 `1` 。如果可以找到一个表达式：

  


-   当 `--l` 大于 `--l-threshold` 时返回 `0`
-   当 `--l` 小于或等于 `--l-threshold` 时返回大于 `1` 的值

  


我们就可以通过 [CSS 的 `clamp()` 函数](https://juejin.cn/book/7223230325122400288/section/7241401565653762108)来获得期望的结果：

  


```CSS
:root {
    --l: clamp(0, ❓ , 1);
    --l-threshold: 0.7; /* 亮度的阈值 */
    --c: oklch(var(--l) 0 0); 
}
```

  


我们可以使用 `--l` 和 `--l-threshold` 比率（即 `l ÷ (--l-threshold)`）来获得这个表达式。因为比率具有一个特性：当分子大于分母时，结果大于 `1` ；否则，结果小于等于 `1` 。

  


-   当 `--l` 小于等于 `--l-threshold` 时，`l ÷ (--l-threshold)` 的值小于 `1`
-   当 `--l` 大于 `--l-threshold` 时，`l ÷ (--l-threshold)` 的值大于 `1`

  


这意味着 `l ÷ (--l-threshold) - 1` 表达式在 `--l` 小于 `--l-threshold` 时为负值，在 `--l` 大于 `--l-threshold` 时为正值。我们只需要将这个表达式乘以一个非常大的负数（数值的绝对值很大），这样当表达式为负数时，结果就能保证大于 `1` 。用 CSS 来描述的话，大致如下：

  


```CSS
:root {
    --l-threshold: 0.7;
    --l: clamp(0, (l / var(--l-threshold) - 1) * -infinity, 1);
    --c: oklch(from var(--color) var(--l) 0 h);
}
```

  


我们的 CSS 还可以像下面这样编写：

  


```CSS
:root {
    --l-threshold: 0.7;
    --l: clamp(0, (var(--l-threshold) / l - 1) * infinity, 1);
    --c: oklch(form var(--color) var(--l) 0 h);
}
```

  


我们通过一个简单的用例来验证它：

  


```CSS
article {
    --tint: #0a9396;
    --highlight: hsl(from var(--tint) h s calc(l + 10));
    --l-threshold: 0.7;
    --l: clamp(0, (l / var(--l-threshold) - 1) * -infinity, 1);
    --c: oklch(from var(--highlight) var(--l) 0 h);
    background-color: var(--highlight);
    color: var(--c);
}
```

  


![](https://p3-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/53cc8cc512a84c059d14937db6ec4d94~tplv-k3u1fbpfcp-jj-mark:0:0:0:0:q75.image#?w=1224&h=606&s=4416705&e=gif&f=204&b=000f16)

  


> Demo 地址：https://codepen.io/airen/full/QwLrPaz

  


整个效果与之前使用 `contrast-color()` 函数是相似的：

  


```CSS
article {
    --tint: #0a9396;
    --highlight: hsl(from var(--tint) h s calc(l + 10));
    --c: color-contrast(var(--highlight) vs #FFF, #000);
    background-color: var(--highlight);
    color: var(--c);
}
```

  


示例的结果告诉我们，使用 RCS 是可以模拟出 `contrast-color()` 函数的效果。

  


接下来，我们来验证 `--l-threshold` 阈值（亮度阈值）是否真的存在？在此之前，我们做了一个大胆的假设：

  


> **存在一个亮度阈值，高于该值时，黑色文本能被清晰读取；低于该值，白色文本能被清晰读取，而且不需要考虑颜色的饱和度和色调！**

  


存疑的是，这个亮度阈值真的存在吗？

  


当人们第一次听说感知均匀的颜色空间时，通常会认为可以通过简单地比较颜色亮度值来推断两个颜色之间的对比度。不幸的是，这并不完全正确，因为对比度不仅仅取决于感知亮度。然而，亮度值与对比度之间确实存在显著的关联性。

  


大多数的时候，我们采用的是 [WCAG 2.1 对比度算法](https://www.w3.org/TR/WCAG21/#contrast-minimum)。然而，多年来大家发现，该算法的结果非常糟糕。通过测试你会发现，对于任何不是非常浅或非常深的颜色，它的表现几乎与随机选择一样糟糕。目前有一个更新的对比度算法，称为 [APCA](https://apcacontrast.com/)（高级感知对比度算法），其效果要好得多。

  


![](https://p3-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/3f3fa936951843a3b82afc829344b711~tplv-k3u1fbpfcp-jj-mark:0:0:0:0:q75.image#?w=2024&h=1012&s=40008&e=png&b=e5e5e5)

  


> 有关于这方面更详细的讨论，[可以点击这里获取](https://github.com/Myndex/SAPC-APCA/discussions/30)！

  


这意味着，Web 开发者陷入了两难境地。当前创建无障碍颜色搭配的最佳是使用 APCA 确保实际可读性，使用 WCAG 2.1 标准遵守合规性底线。

  


我们可以使用 [Color.js](https://colorjs.io/) 来做一个实验，遍历 OKLCH 的参考范围（大致基于 P3 色域），以递增的粒度计算亮度范围。在这些范围内，白色是“最佳”文本颜色（即与黑色相比产生更高对比度）的区域，以及反之的区域：

  


![](https://p3-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/bb5e31eecadf4bd7860e0c8090ba9da2~tplv-k3u1fbpfcp-jj-mark:0:0:0:0:q75.image#?w=1138&h=818&s=3129573&e=gif&f=309&b=fefefe)

  


> URL: https://lea.verou.me/blog/2024/contrast-color/research

  


上图中的表格是使用 `C ∈ [0, 0.4] (step = 0.025)` 和 `H ∈ [0, 360) (step = 1)` 生成的。需要注意的是，这些是每个级别的最小和最大 `L` 值。例如，白色文本在 `L ∈ [62.4%, 100%]` 时可能会违反 WCAG，这并不意味着所有 `L > 62.4%` 的颜色都会违反 WCAG，而只是说有些颜色会。因此，我们只能通过反向逻辑得出有意义的结论：由于所有白色文本失败的情况都出现在 `L ∈ [62.4%, 100%]`，所以可以逻辑推断出，如果 `L < 62.4%`，白色文本将通过 WCAG，无论颜色是什么。

  


通过将这一逻辑应用于所有范围，我们可以为这些区间中的许多情况得出类似的保证：

  


![](https://p3-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/0f490a75fb17412c9e93147dcb6c6e35~tplv-k3u1fbpfcp-jj-mark:0:0:0:0:q75.image#?w=3308&h=640&s=239564&e=jpg&b=ffffff)

  


你可能已经注意到，WCAG 对白色文本通常存在大量误判（假阴性），并且倾向于将亮度阈值设定得比 APCA 低得多。这是 WCAG 算法的已知问题。

  


因此，为了在可读性和合规性之间取得最佳平衡，我们应该使用能接受的最高阈值。这意味着：

  


-   如果通过 WCAG 是必要条件，最高可以使用的阈值是 **`62.3%`**
-   如果实际的可读性是我们唯一关心的问题，可以忽略 WCAG，并选择一个介于 **`68.7%`** **和** **`71.6%`** 之间的阈值，例如 **`70%`**

  


需要注意的是，如果你的实际颜色范围受到更多限制（例如某些特定的色相或色度范围，或特定的色域），你可以通过使用不同的阈值更好地平衡这些权衡。你可以使用实际的颜色范围自己运行实验并找出适合的阈值！以下是我一些较窄范围的示例，以及仍能通过 WCAG 2.1 的最高阈值：

  


| **描述**    | **颜色范围**           | **亮度阈值** |
| --------- | ------------------ | -------- |
| 现代低端屏幕    | 位于 sRGB 色域内的颜色     | 65%      |
| 现代高端屏幕    | 位于 P3 色域内的颜色       | 64.5%    |
| 未来高端屏幕    | 位于 Rec.2020 色域内的颜色 | 63.4%    |
| 中性色       | C ∈ [0, 0.03]     | 67%      |
| 柔和色       | C ∈ [0, 0.1]      | 65.6%    |
| 暖色（红/橙/黄） | H ∈ [0, 100]      | 66.8%    |
| 粉色/紫色     | H ∈ [300, 370]    | 67%      |

  


特别有趣的是，通过忽略现代屏幕上实际上无法显示的颜色，亮度阈值被提高到了 `64.5%` 。因此，假设浏览器在进行色域映射时优先保留亮度，我们可以使用 `64.5%` 的阈值，并且仍然可以保证符合 WCAG 要求。

  


简单地说，通过上面的验证，颜色亮度的阈值的确存在，而且只要达到 `62.3%` 就能满足 WCAG 要求。因此，我们可以把 `--l-threshold` 阈值设置为 `0.623` ：

  


```CSS
:root {
    --l-threshold: 0.623;
    --l: clamp(0, (l / var(--l-threshold) - 1) * -infinity, 1);
    --c: oklch(from var(--color) var(--l) 0 h);
}
```

  


## 小结

  


在不久的将来，我们可以直接使用 CSS 的 `contrast-color()` 根据背景颜色动态设置文本颜色，使其达到WCAG 颜色对比度的要求。在 `contrast-color()` 还未得到更多主流浏览器支持之前，我们也可以考虑使用 CSS 的相对颜色（RCS）来模拟 `contrast-color()` 函数的功能，毕竟 RCS 得到了更多的主流浏览器支持。
