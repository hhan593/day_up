Node.js 的 `http` 模块除了可以写 HTTP 服务端之外，也能作为 HTTP 客户端对外进行请求。一个最简单的例子便是：

```js
const http = require('http');

http.get('http://httpbin.org/get', (res) => {
  console.log(`状态码: ${res.statusCode}`);
  res.on('data', (d) => {
    process.stdout.write(d);
  });
}).on('error', (e) => {
  console.error(e);
});
```

或者，我们也可以用一些更方便的类库，例如 [spidex](https://www.npmjs.com/package/spidex)。用它来实现上面的效果，代码如下：

```js
const spidex = require("spidex");

spidex.get("http://httpbin.org/get", function(content, statusCode, responseHeaders) {
    console.log(`状态码: ${statusCode}`);
    console.log("响应体: ", content.toString());
}, "utf8").on("error", function(err) {
    console.error("请求失败: ", err);
});
```

但这都不是我们本章主要要讲的，本章的主要内容还是讲讲 WinterCG 里面的 `fetch`，以及 Node.js 里面也有 `fetch`。



## 什么是 `fetch`

如果你是个老生代的前端，应该对 `XMLHttpRequest`（XHR）不会感到陌生。

`XMLHttpRequest`（XHR）是一种浏览器内置的对象，允许进行异步 HTTP 请求。它在 AJAX（Asynchronous JavaScript and XML）编程中被广泛使用，尽管名称中包含“XML”，但它可以用来处理任何类型的数据，包括 JSON、XML、HTML 以及纯文本。

> `XMLHttpRequest` 的名字来源于它最初被设计出来的主要用途——在客户端（如浏览器）和服务器之间异步交换 XML 数据。在 AJAX（Asynchronous JavaScript and XML）技术开始流行的早期，XML 是主要的数据交换格式，因此这个 API 被命名为 `XMLHttpRequest`。**AJAX 同理。**
>
> 这就跟 `JSON` 一样，道格拉斯对自己命名的 `JSON` 这个名字牢骚满满，命名现在已经是个通用的数据交换格式，还背着个 JavaScript Object Notation 的全称。
>
> 类似的现象我们在[第七章](https://juejin.cn/book/7196627546253819916/section/7197301670261293095)也提到过，npm 最开始是 Node.js Package Manager，但现在已经不再提这个全称了。算是一种时代的局限性吧。


`XMLHttpRequest` 的特点有：

-   支持**同步和异步**通信；
-   可以处理各种数据类型；
-   在使用时需要创建 `XMLHttpRequest` 对象的实例，然后通过调用其方法和设置回调函数来发送请求和处理响应；
-   比较繁琐，需要手动处理整个请求的生命周期（如打开连接、设置请求头、发送请求及处理响应）。 


示例代码：

```js
var xhr = new XMLHttpRequest();
xhr.open("GET", "http://example.com/api/data", true);
xhr.onreadystatechange = function() {
  if (xhr.readyState === 4 && xhr.status === 200) {
    console.log(xhr.responseText);
  }
};
xhr.send();
```

其实 Node.js 的 `http` 模块设计也一定程度上受了它的影响。它提供了一个低级的 API 来处理 HTTP 请求和响应，也需要一定程度上处理各种生命周期，监听事件等等。

`fetch` 就不同了。Fetch API 提供了一种更现代、更强大且更灵活的方式来发起 HTTP 请求。它基于 Promises 设计，使得处理异步操作更加简洁。特点有：

-   基于 Promise，使得异步代码更加简洁和易于维护；
-   默认情况下不发送或接收 cookies，如果站点依赖于用户会话，则需要配置 credentials 选项；
-   不支持取消正在进行的请求（不过现在有 `AbortController` 了）。
-   更简洁的 API，易于使用和学习。


示例代码：

```js
const response = await fetch("http://example.com/api/data");
if (response.ok) { // 状态码在 200-299 之间
  const data = await response.text(); // 假设响应是纯文本
  console.log(data);
}
```

我个人觉得，如果是个大魔导师，`XMLHttpRequest` 是需要念冗长咒语的魔法，`fetch` 则是瞬发。

## WinterCG 里的 Fetch API

WinterCG 里面的 Fetch API 是 WHATWG 里面 Fetch API 的一个分支版本，基本上大体一致，但也有部分的不同。截止发文当天，这个分支主要：


> This specification is a fork of the WHATWG Fetch spec [[FETCH]](https://fetch.spec.wintercg.org/#biblio-fetch), made as part of the [WinterCG](https://wintercg.org/) effort, to make it more suitable for server-side JavaScript runtimes to implement.
>
> The goal of this fork on a short to medium term is to agree on the behavior of fetching in server-side runtimes, and on the long term to upstream those changes into the WHATWG specification.
>
> The changes from the WHATWG spec so far are:
>
> -   Added [§ 2 Conformance classes](https://fetch.spec.wintercg.org/#conformance-classes) to describe the various types of runtimes implementing this fork.
> -   Made the [forbidden request-header](https://fetch.spec.wintercg.org/#forbidden-request-header) and [forbidden response-header name](https://fetch.spec.wintercg.org/#forbidden-response-header-name) definitions dependent on whether the user agent [supports CORS](https://fetch.spec.wintercg.org/#supports-cors) and [cookies](https://fetch.spec.wintercg.org/#supports-cookies). This affects the headers of [basic filtered responses](https://fetch.spec.wintercg.org/#concept-filtered-response-basic), as well as [validation](https://fetch.spec.wintercg.org/#headers-validate) in [Headers](https://fetch.spec.wintercg.org/#headers) objects. The behavior for web browsers doesn’t change.
>
>   
>
>
> This spec is still a work in progress, and some things we hope to specify in the future are:
>
> -   Remove CORS restrictions for runtimes with no concept of origins.
> -   Specify how relative URLs resolve in [fetch](https://w3c.github.io/ServiceWorker/#service-worker-global-scope-fetch-event) and the [Request](https://fetch.spec.wintercg.org/#request) constructor for runtimes without a concept of [API base URL](https://html.spec.whatwg.org/multipage/webappapis.html#api-base-url).
> -   Full-duplex request streaming (see [WHATWG issue #1254](https://github.com/whatwg/fetch/issues/1254)). Although this is not specific to server-side runtimes, it’s unlikely for browsers to implement it anytime soon.
> -   Specify how [Request](https://fetch.spec.wintercg.org/#request) and [Response](https://fetch.spec.wintercg.org/#response) behave in (runtime-specific) HTTP server APIs. Do they behave as they do in [service workers](https://w3c.github.io/ServiceWorker/#dfn-service-worker)?
> -   Etc.

  


总结下来几点：

1.  **适配服务器端环境：** 专门为服务器端 JavaScript 运行时设计，考虑到服务器环境的特殊需求和上下文；
1.  **头部名称定义的调整：** 基于运行时是否支持 CORS 和 cookies，调整了禁用的请求头和响应头名称的定义，这是为了更好地适应服务器端应用的安全模型和需求；
1.  **未来规划：** 包括去除对没有源概念的运行时的 CORS 限制、指定无 API 基础 URL 概念的运行时中相对 URL 的解析方式、支持全双工请求流，以及明确 HTTP 服务器 API 中 Request 和 Response 的行为等。


其实还有一个更简单的大白话就是：突破了浏览器 CORS 和 cookies 的限制，可访问任意网站了，可以自己传 cookies 了。

当然，还有更多的规范内容还在制定过程中，这个规范也在持续迭代的。



## Node.js 里的 Fetch API

Node.js 在 [v16.15.0 版本](https://nodejs.org/en/blog/release/v16.15.0#add-fetch-api)中首次引入 Fetch API。那个时候还需要使用 `--experimental-fetch` 命令行参数来开启。到 Node.js [v18](https://nodejs.org/en/blog/announcements/v18-release-announce) 后，Fetch API 就被默认开启了。


### Undici

Fetch API 是基于 Undici 实现的。Undici 是一个专为 Node.js 环境设计的高性能 HTTP/1.1 客户端。它旨在提供比 Node.js 标准 `http` 模块更高效的 HTTP 请求处理能力。Undici 的开发主要考虑到了性能和最佳实践，尝试解决 Node.js 核心 `http` 模块在某些情况下可能遇到的性能瓶颈和限制。

> 关于 Undici 的名字，源于 HTTP/1.1。1.1 即 11，而 Undici 是意大利语里的 11。

Undici 有几大特点：

1.  **高性能：** Undici 专注于提高 HTTP 请求和响应的处理速度，实现了更快的数据解析和传输；
1.  **连接复用：** 通过支持连接池和 HTTP Keep-Alive，可以复用 TCP 连接，减少连接的建立和关闭开销，进一步提升性能；
1.  **更现代的 API：** 提供了符合现代 JavaScript 异步编程实践的 API，包括支持 Promise、async/await 以及 Stream 处理；
1.  **HTTP/1.1 客户端：** 虽然 Node.js 已经支持 HTTP/2，但 Undici 依然专注于提供一个高效的 HTTP/1.1 客户端实现。

总之，Undici 就是快，比 Node.js 的 `http` 模块还快。不过，它并没有在 Node.js 的内置模块中导出给用户用，而是被 `fetch` 封装起来了。若你想直接使用 Undici，直接装它的 [npm 包](https://www.npmjs.com/package/undici)就好了。


### `fetch`

为了可以让 `fetch` 的迭代小步快跑，`fetch` 这个 API 的实际实现也在 Undici 这个包中。它的 API 是：

```js
import { fetch } from 'undici';

const res = await fetch('https://example.com');
const json = await res.json();
console.log(json);
```

> #### **`undici.fetch(input[, init]): Promise`**
>
> Implements [fetch](https://fetch.spec.whatwg.org/#fetch-method).
>
> -   <https://developer.mozilla.org/en-US/docs/Web/API/WindowOrWorkerGlobalScope/fetch>
> -   <https://fetch.spec.whatwg.org/#fetch-method>



如此一来，在 Node.js 中的 `fetch` 实际上就是[这样封装的](https://github.com/nodejs/node/blob/v18.19.1/lib/internal/process/pre_execution.js#L263-L310)：

```js
// https://fetch.spec.whatwg.org/
function setupFetch() {
  if (process.config.variables.node_no_browser_globals ||
      getOptionValue('--no-experimental-fetch')) {
    return;
  }
```

如果命令行参数里面有 `--no-experimental-fetch`，则跳过不赋值。

```js
  let undici;
  function lazyUndici() {
    if (undici) {
      return undici;
    }

    undici = require('internal/deps/undici/undici');
    return undici;
  }
```

由于 Undici 其实也是有不少代码的，能不加载就不加载。所以这里懒加载，当用户真的用到 `fetch` 函数时，才去加载 Undici。

```js
  async function fetch(input, init = undefined) {
    return lazyUndici().fetch(input, init);
  }

  defineOperation(globalThis, 'fetch', fetch);
```

这里就看出来了，实际上 Node.js 的 `fetch` 函数里面就是：

1.  懒加载 Undici；
1.  直勾勾调用 Undici 里的 `fetch` 函数。


这就像是我之前在知乎的一个抖机灵回答：[原生 Promise 和手写 Promise 的区别是什么？](https://www.zhihu.com/question/456775221/answer/1880654950)

> 原生 `Promise` 和手写 `Promise` 的区别是原型链长度加一。
>
> ```js
> class MyPromise extends Promise {}
> ```


然后是一些不重要的代码，把 `FormData`、`Headers` 这些 WHATWG 的类也给挂到 `globalThis` 上。

```js
   // 不重要代码
   ...
 }
```

最后，在适当位置调用 `setupFetch()`，就可以把 Fetch API 和它的周边东西给挂上去了。心细一点可能会发现，上面命名是 `defineOperation`，不是 `defineProperty` 这种。它在 Node.js 中的实现如下：

```js
// https://heycam.github.io/webidl/#define-the-operations
function defineOperation(target, name, method) {
  Object.defineProperty(target, name, {
    __proto__: null,
    writable: true,
    enumerable: true,
    configurable: true,
    value: method,
  });
}
```

实际上就是以 `writable`、`enumerable`、`configurable` 的方式去往目标上挂载某个事物。


##### 小插曲：为什么不直接使用 `Object.defineProperty()` 呢？

这又关系到标准了。WHATWG 里面有一个概念叫 WebIDL。WebIDL 是一种用于描述浏览器接口的规范语言，它为开发者提供了一种清晰的方式来定义浏览器提供的各种功能和接口。

大部分 WHATWG 规范中定义的函数都会有相应的 WebIDL 声明。Fetch API 自然也有，如下：

```js
partial interface mixin WindowOrWorkerGlobalScope {
  [NewObject] Promise<Response> fetch(RequestInfo input, optional RequestInit init = {});
};
```

> 大家如果对 WebIDL 的定义和声明感兴趣，可自行上网搜索和学习。


在 WebIDL 中，有个概念叫 Operation，它用于描述接口中方法（或函数）。一个 Operation 定义了接口中的操作的名称、参数和返回类型。

有了 Operation 这个概念，自然就有定义一个 Operation 这种做法。上面的代码块的含义就是在 `globalThis` 这个 Scope 中定义了 `fetch` 这个 Operation，里面表明了它的出入参和类型。

那么，“定义一个 Operation”，在代码实现里面的具体含义是什么呢？WebIDL 规范里面也[给了说明](https://webidl.spec.whatwg.org/#define-the-operations)：


创建一个属性描述符，使其 `writable`、`enumerable`、`configurable` 都为 `true`，然后将其以该描述符的形式定义到目标对象去。

这里的目标对象就是指 `globalThis` 了。所以在 Node.js 代码中，使用 `defineOperation` 这个函数语义去挂载 `fetch`，是为了符合 WebIDL。但它的本质其实就是以 `writable`、`enumerable`、`configurable` 的方式去挂载。



## 本章小结

本文介绍了在 Node.js 中进行 HTTP 请求的现代方式——Fetch API。Fetch API 是一种现代、灵活的 HTTP 请求方式，与传统浏览器端的 XMLHttpRequest 相比，它提供了更简洁、更易用的 API，并且基于 Promise，使得异步操作更加方便。在 Node.js 中，与 `http` 模块相比，同样更简洁、更易用、更标准。

在 Node.js 中，Fetch API 的实现基于 Undici，这是一个专为 Node.js 设计的高性能 HTTP/1.1 客户端。Undici 旨在提高 HTTP 请求处理的效率，与 Node.js 标准的 http 模块相比，具有更高的性能和更现代化的 API。

所以 Node.js 里面的 Fetch API 又快又狠又准！
