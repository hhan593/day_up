### Nginx

> 实际上 Master - Servant 这个词是我编的，大家都是叫 Master - Worker、Cluster - Worker 的。

Nginx 是 Master - Worker 机制的非常有名的实践项目。Nginx 启动时，首先会有一个主进程（Master）被创建。这个 Master 进程主要负责管理工作，包括读取和验证配置文件、管理工作进程（Worker）、处理信号等。Master 进程不处理实际的客户端请求，其主要职责是监督和控制。

Master 进程会根据配置生成一定数量的工作进程（Workers）。这些工作进程是真正处理客户端请求的进程。每个工作进程都能独立处理进入的客户端连接和请求，且相**互之间是独立的，这意味着它们不需要共享资源（例如内存），从而减少了锁的竞争，提高了效率**。

不过 Nginx 是主进程首先通过 `socket()` 来创建一个文件描述符用来监听，然后 `fork` 生成工作进程，工作进程将继承父进程的 socket 文件描述符，之后子进程 `accept()` 后将创建已连接描述符，然后通过已连接描述符来与客户端通信。

### Egg.js

说到 Node.js 的多核，就不得不提一下 Egg.js 了。Egg.js 用的就是 [Cluster - Worker 机制](https://www.eggjs.org/zh-CN/core/cluster-and-ipc)。它在自己的文档中这么写：

> 简单地说，Cluster 是：
> - 在服务器上同时启动多个进程；
> - 每个进程里都运行着同一份源代码（好比把以前一个进程的工作分给多个进程去做）；
> - 更神奇的是，这些进程可以同时监听一个端口。
> 
> 其中：
> - 负责启动其他进程的叫做 Master 进程，好比是一个“包工头”，不做具体的工作，只负责启动其他进程；
> - 其他被启动的进程叫 Worker 进程，顾名思义就是干活的“工人”。它们接收请求，对外提供服务；
> - Worker 进程的数量一般根据服务器的 CPU 核数来定，这样可以完美利用多核资源。

这个方法底下[调用的是 libuv 中的 ](https://github.com/nodejs/node/blob/v20.11.1/src/process_wrap.cc#L268)`uv_spawn()` 函数。

#### Spawn 不离其宗

根据上面的图，我们能知道，`child_process` 中的几个函数最底下都是 `uv_spawn()`。那么 `uv_spawn()` 到底做了什么呢？



> 此处只讲 Linux 下的实现。

`uv_spawn()` 里面做的两个核心逻辑是：`fork()` + `execvp()`。`spawn` 从语义上来说，就是从父进程中 `fork()` 出一个子进程，然后马上在子进程里面执行需要执行的命令，去启动一个新的程序。



##### `fork()`
`fork()`函数是 UNIX 和类 UNIX 操作系统（比如 Linux）中的一个系统调用，用于创建一个新的进程，这个新进程被称为子进程。它是从调用它的进程（父进程）克隆出来的，子进程获得父进程数据段、堆和栈的副本。这意味着子进程拥有与父进程相同的内存图像，但是这些内存页在写入时通常会被复制（写时复制，CoW，copy-on-write），因此父子进程的内存空间实际上是隔离的。



父子进程内存写时复制，是一种计算机内存管理技术。假设卡卡西和鸣人在读同一本书，因为内容完全一样，卡卡西没必要买两本。所以父子进程 fork 出来的时候，内存并不需要一模一样复制一份。

反正是复印件，爱咋改咋改。子进程的写时复制也一样，本来是一份内存。子进程或者父进程要改其中一块内存的话，就在当时再把内存页复制出来，进行修改。



因为最开始的父进程是一条线，调用 `fork()` 后，就出来两条线了。两条线都有自己的想法，肯定会产生内存页不同的修改，当然不能一起修改原件了。



父子进程是两条线，所以在 `fork()` 之后，你得知道自己在哪条线，才有可能做不一样的事。比如父进程要输出“我是恁爹”，子进程说“我是崽子”。代码就可能是这样的：

```cpp
#include <iostream>
#include <unistd.h>  // 包含 fork() 函数
#include <sys/wait.h> // 包含 wait() 函数

int main() {
    int a = 10;
    // 调用 fork 创建子进程
    pid_t pid = fork();

    if (pid == -1) {
        // fork 失败的情况
        std::cerr << "创建子进程失败！" << std::endl;
        return 1;
    } else if (pid == 0) {
        // 这里是子进程执行的代码
        std::cout << "我是崽子" << a << std::endl;
    } else {
        // 父进程会得到子进程的 PID
        std::cout << "我是恁爹" << a << std::endl;
        wait(NULL);  // 父进程等待子进程结束
    }

    return 0;
}
```



上面就是最简单的一个 `fork` 代码了。代码里面调用 `fork()` 函数，实际上就会出来两条线。如果 `pid` 是 `-1`，则表示当前是父进程，且 `fork()` 失败了，我们输出失败逻辑；如果 `pid` 是 `0`，则代表当前是子进程，那么输出 `我是崽子`；如果 `pid` 是别的数，则说明当前是父进程，且 `pid` 是 `fork()` 出来的子进程的进程号，那么在父进程里面输出 `我是恁爹` 就好了。最后在父进程的逻辑中调用 `wait(NULL)` 来等待所有子进程退出。

##### `execvp()`
`execvp()` 函数是 POSIX (Portable Operating System Interface) 标准的一部分。POSIX 是一系列 IEEE 标准，旨在保持 UNIX 类操作系统之间的兼容性。这包括了对 API、命令行命令等的标准化。



作为 POSIX 标准的一部分，`execvp()` 以及其他 `exec` 类函数，如 `execl()`、`execle()`、`execlp()`、`execv()`、`execve()` 和 `execvp()`，都定义了允许兼容 POSIX 的操作系统（如 Linux、macOS、BSD 等）中的进程执行其他程序的方法。这些函数通过替换当前进程的地址空间，包括代码和数据，来执行一个新的程序，而不是创建新的进程。



`execvp()` 会加载并运行一个新程序，但是这个新程序会使用调用它的进程的同一个进程编号。这意味着原进程停止执行，而被一个全新的程序替代。



```c
int execvp(const char *file, char *const argv[]);
```

- **file**：指定要执行的程序。如果该名称不包含斜线（即没有给出路径），`execvp()` 会在环境变量 `PATH` 中指定的目录搜索这个程序；

- **argv**：是一个字符串数组，用于提供给新程序的参数列表。数组的第一个元素应为程序的名称，数组最后一个元素必须是 `NULL`，标识数组的结束。



这个函数不返回，因为一旦新程序被加载并开始运行，原进程的映像就已经被替代。除非是执行失败，那么它会返回 `-1` 并设置 `errno` 以指示错误原因。比如下面这段代码：

```cpp
#include <unistd.h>
#include <iostream>

int main() {
    char *args[] = {"ls", "-l", NULL}; // 要执行的命令及参数

    // 执行 ls 命令
    if (execvp("ls", args) == -1) {
        std::cerr << "执行失败" << std::endl;
        return 1;
    }

    return 0; // 这行代码不会被执行，除非 execvp 失败
}
```



当我们执行了 `execvp()` 后，首先判断它返回值是不是 `-1`。如果是 `-1` 则说明执行失败。否则，后面的代码永远不会被执行，比如最后那段 `return 0` 就永远执行不到。因为一旦 `execvp()` 执行成功，这个进程就已经被替代了。这就像是夺舍，`-1` 就是夺舍失败。如果夺舍成功，它就不再是它。就跟诡秘之主里的克莱恩一样。

> 克莱恩·莫雷蒂是鲁恩王国阿霍瓦郡廷根市人，毕业于霍伊大学历史系，由于安提哥努斯家族的笔记身亡，之后成为周明瑞的穿越对象。



换个简单易懂的代码：

```cpp
#include <unistd.h>
#include <iostream>

int 克莱恩() {
    char *args[] = {"穿越", "周明瑞", NULL}; // 要执行的命令及参数

    if (被夺舍("穿越", "周明瑞", args) == -1) {
        std::cerr << "执行失败" << std::endl;
        return 1;
    }

    return 0; // 这行代码不会被执行，除非 execvp 失败
}
```

在上面执行完之后，虽然程序还顶着克莱恩的程序编号，但是内在已经变成周明瑞了。

**原因很简单，宿主会被替换掉。**

#### 同步 Spawn
在 Node.js 早期的版本中，所有的 `spawn()` 类函数都是异步的。比如我想在 Node.js 里面想执行一个 `ls` 直接列出当前文件等操作，都得 `exec()` 后，在 `callback()` 函数中拿到结果并输出，而不能像这样：

```javascript
const output = exec('ls');
```

当年我为了解决这个问题，自己写了个 `sync-runner` 的 [npm 包](https://github.com/XadillaX/syncRunner)。让我们可以类似这样执行：

```javascript
var run = require("sync-runner");
var result = run("command", "current_work_directory", max_million_seconds);
console.log(result);
```



原理是，我不用 libuv 的 `uv_spawn()`。而是我自己去做 `fork()` 以及类似 `execvp()` 的操作（用了一个叫 [popen_plus](https://github.com/Codingrecipes/Popen-) 的库），然后在 `fork()` 后的主进程中通过 `waitpid()` 来等待子进程执行完毕或者执行超时，然后主进程再开始执行下一步操作。



后来，Node.js 自身也出了 `execSync()` 这些函数了。我们之前也介绍了，`execSync()` 这些同步函数的一脉相承如下：


![image.png](https://p6-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/81ebb93896494591a1ebbca58a39ccb5~tplv-k3u1fbpfcp-jj-mark:0:0:0:0:q75.image#?w=1242&h=1140&s=92533&e=png&b=fffefd)

`SpawnSync` 就是黑魔法的所在了。它的作用跟我之前写的 `sync-runner` 异曲同工。只不过我是通过我自己一个 `while()` 死循环加一个 `waitpid()` 来做的；而 `SpawnSync` 则是通过套了个 libuv 的 loop 来解决。



我们还记得，libuv 的事件循环是大乘套小乘，其实也是一个 `while()` 死循环。前文中我们稍微看了一下它的实现，这里我们把相关逻辑细化讲一下：

```cpp
// 当前是主事件循环里其中一个 Tick 中的逻辑
Maybe<bool> SyncProcessRunner::TryInitializeAndRunLoop(Local<Value> options) {
  int r;

  // 初始化一个新的用于跑子进程的事件循环
  uv_loop_ = new uv_loop_t;
  r = uv_loop_init(uv_loop_);

  // 如果 options 中设置了超时时间
  if (timeout_ > 0) {
    // 为跑子进程的事件循环初始化一个超时时间的定时器
    r = uv_timer_init(uv_loop_, &uv_timer_);

    uv_timer_.data = this;
    kill_timer_initialized_ = true;

    // 开始定时器,定时器到期后会调用 KillTimerCallback 去杀死子进程(如果到时候还存在的话)
    r = uv_timer_start(&uv_timer_, KillTimerCallback, timeout_, 0);
  }

  // 为子进程退出设置一一个退出回调
  uv_process_options_.exit_cb = ExitCallback;

  // 通过 uv_spawn 在子进程中执行对应的命令
  r = uv_spawn(uv_loop_, &uv_process_, &uv_process_options_);
  uv_process_.data = this;

  // 开始用于跑子进程的事件循环,相当于在这个 Tick 中应该新的死循环,直到子进程退出,或者超时
  r = uv_run(uv_loop_, UV_RUN_DEFAULT);

  ...
}
```

> 为了讲解逻辑简单，把很多分支逻辑、失败逻辑都摘掉了。



我们能看到，首先就是初始化一个新的时间循环，然后为新事件循环设置好定时器，然后通过 `uv_spawn()` 在新的事件循环中执行子进程对应的命令。最后开始跑新的事件循环（`while()`），直到新的事件循环执行结束（没有任何新事件，即子进程也结束了，定时器也没有了）。



所以是跟我之前的 `sync-runner` 类似。都是在 `while()` 中阻塞直到进程执行完毕。只不过我是手写的 `while()`，而 `execSync()` 则是直接新建了一个新的事件循环。

### `cluser` 模块：Node.js 官方包工头
前面我们聊的 `child_process`，更像是给了你一套工具，让你自己去当那个“包工头”老包。你想 `fork` 几个进程，进程之间怎么通信，怎么管理，都得亲力亲为。这对于想快速搭建一个能利用多核能力的网络服务来说，还是有点麻烦。



像开一家连锁店，`child_process` 给了你砖头、水泥、钢筋，让你自己从零开始盖每一家分店。虽然自由度高，但对大多数只想安安静静拉花的店长来说，还是太复杂了。



Node.js 的设计者们也想到了这一点，于是他们提供了一个更方便的工具——`cluster` 模块。你可以把它理解成一个专业的“连锁店加盟品牌方”。它帮你处理好了所有开分店的繁琐事务，比如选址、装修、员工培训等等。你只需要专注于拉好你的花就行了。



`cluster` 模块的本质，其实就是对 `child_process.fork()` 的一层封装和管理。它帮你创建了一堆 Worker 进程，并且用一种巧妙的方式，让这些 Worker 进程都能响应同一个端口的网络请求。



当我们使用 `cluster` 模块时，代码通常会有一个入口文件，但这段代码在 Master 进程和 Worker 进程中执行的“角色”是不同的。通过 `cluster.isMaster` (在较新版本中是 `cluster.isPrimary`) 和 `cluster.isWorker`，我们可以区分当前进程的角色，然后让它们各司其职。



- **Master（包工头）**：不亲自下场处理具体的业务请求。它的主要工作是根据你电脑的 CPU 核心数，`fork` 出适量的 Worker 进程。然后像个包工头一样监督着手下的工人们，如果哪个工人（Worker）不小心崩溃，它会立刻再招一个新的工人顶上；

- **Worker（AAA 建材朱哥）**：这些才是真正干活的。每个 Worker 都是一个独立的 Node.js 实例，它们共同接收来自同一个端口的请求，然后埋头处理业务逻辑，最后把结果返回给客户。



这种模式下，一个请求进来，Master 进程会以一种负载均衡的策略，把它交给一个当前比较“闲”的 Worker 去处理。这样一来，原本只能由一个进程处理的所有请求，现在被分摊到了多个进程上，CPU 的每个核心都得到了利用，服务的整体吞吐能力自然就上去了。



所以，`cluster` 模块就是 Node.js 官方提供的一个标准化的、开箱即用的多进程解决方案，专门用来解决单进程无法利用多核 CPU 的痛点。



### 端口共享
说到这里，你可能会有一个疑问：我们都知道，在同一个时间，一个端口只能被一个进程监听和绑定。那为什么 `cluster` 模式下，多个 Worker 进程可以同时“监听”同一个端口呢？难道它们练了什么奇怪的技能？


![image.png](https://p3-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/e426f1246c624605aa8effdbf2d15f49~tplv-k3u1fbpfcp-jj-mark:0:0:0:0:q75.image#?w=600&h=556&s=136835&e=png&b=fefdfd)

这其实并不是真正的“同时监听”，而是 Master 进程在背后耍了一点小心机。这个心机主要有两种流派。



#### 轮询（Round-Robin）
这是最简单、也是 Node.js 在非 Windows 平台下的默认策略。



你开的咖啡店门口只有一个点餐台，但是后面有四个咖啡师在待命。Master 进程就像是站在台后的小姐姐。



当第一个顾客来了，小姐姐会把他的单子给 1 号咖啡师。第二个顾客来了，引导给 2 号咖啡师。第三个顾客来了，引导给 3 号咖啡师。第四个顾客来了，单子给 4 号 AAA 建材朱哥（乱入）。第五个顾客来了，再回头引导给 1 号咖啡师。



……如此循环往复，像发扑克牌一样，一个一个挨着来。



在这种模式下，实际上只有 Master 进程真正在监听那个端口。它接收到所有连接请求，然后通过轮询，把这些请求公平地分发给手下的每一个 Worker 进程。Worker 进程们并没有直接接触到端口，它们只是被动地接收 Master “投喂”过来的任务。



这种方式简单粗暴，能保证负载大致均衡，但可能不够智能。比如 1 号咖啡师手脚麻利，很快就做好闲下来了；而 AAA 建材朱哥接了个拉坏了 10 杯花，正忙得不可开交。但前台小姐姐不管这些，下一个顾客还是会按顺序分配给他。

#### 句柄传递
这是 Windows 平台上的默认策略，也是一种更高效、更底层的玩法。



下面就比较猎奇了。前台小姐姐不再自己接单，而是把自己接单的能力接了个管子给其他四个咖啡师。这些管子就是前台自己监听的句柄。这样一来，所有的咖啡师都拥有了接单的能力。当一个新订单打进来时，操作系统内核会介入，直接把这个单子转接给当前最空闲的那个 Worker。谁先抢到就是谁的。



这种方式叫“TCP 连接的负载均衡”，它发生在更底层的操作系统层面，效率比上层应用自己搞轮询要高得多。因为内核最清楚哪个进程当前有空。



总结一下，无论是哪种方式，最终都实现了一个效果：**由一个主进程统一对外接收流量，然后通过某种策略将流量分发给多个子进程进行处理**。这便是 `cluster` 模块实现多核利用的核心魔法。它并不是让多个进程真的绑定了同一个端口，而是通过一个统一的入口和巧妙的分发机制，创造出了这种“共享端口”的假象。

### 进程间通信（IPC）：Master 与 Worker 隔空喊话
既然是 Master 带着一群 Worker 干活，那它们之间肯定不能是孤立的，总得有个沟通的法子。不然，Master 怎么给 Worker 下达新的指示？Worker 遇到问题了又怎么向 Master 报告呢？



这种跨进程的沟通机制，就叫做**进程间通信（Inter-Process Communication，简称 IPC）**。



在 Node.js 的 `cluster` 模式下，Master 和 Worker 之间的沟通渠道在 `fork` 的时候就已经建立好了。你可以把它想象成，Master 在招募每一个 Worker 的时候，都给它配发了一部内部对讲机。

![](https://p3-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/b3c3430cf0a445028af728a6239420a6~tplv-k3u1fbpfcp-jj-mark:0:0:0:0:q75.image#?w=300&h=218&s=20861&e=png&a=1&b=f9f9f9)

通过这部对讲机，它们可以互相发送消息：

- **Master 对 Worker 喊话**：Master 进程可以通过遍历 `cluster.workers` 对象，拿到每一个 Worker 的实例，然后调用其 `.send()` 方法来发送消息。比如，数据库配置变了，Master 就可以广播一条消息：“各位注意，准备下班！”

- **Worker 向 Master 报告**：Worker 进程则可以通过 `process.send()` 方法向 Master 发送消息。比如，一个 Worker 发现自己处理的某个任务特别耗时，可能会报告给 Master：“老板，我这边接到一个超大订单，可能要忙一会儿，我要下班。”



这些消息可以是字符串，也可以是 JSON 对象，非常灵活。底层实现上，Node.js 会把这些消息序列化（打包成一种可传输的格式），通过已经建立好的 IPC 通道（在 Linux 和 macOS 上是 Unix Domain Socket，在 Windows 上是命名管道）进行传输，接收方再反序列化（解包）还原成原始消息。



这个过程就像前台小姐姐通过对讲机下达指令，咖啡师们收到后执行；咖啡师也可以通过对讲机向前台小姐姐汇报自己的工作状态或遇到的问题。有了这个沟通机制，整个“团队”才能协同作战，而不是一盘散沙。

### 优雅退出与进程守护
一个健壮的系统，不能想当然地认为所有进程都会永远健康地运行下去。程序有 Bug、内存泄漏、或者处理某个特定请求时发生了意料之外的错误，都可能导致 Worker 进程突然“牺牲”。



如果一个 Worker 挂了，我们又没做任何处理，那么这个服务团队就减员一人，处理能力下降。如果所有 Worker 都相继挂掉，那整个服务就瘫痪了。这显然是不能接受的。



`cluster` 模块对此也早有准备，它提供了一套“进程守护”的机制。



Master 进程作为包工头，它会时刻关注着手下 Worker 们的状态。它通过监听 `cluster` 对象的 `exit` 事件，就能第一时间知道哪个 Worker “离职”了。



```javascript
cluster.on('exit', (worker, code, signal) => {
  // 一个 Worker 进程退出了...
});
```



一旦这个事件被触发，Master 进程就可以在回调函数里采取行动。最常见的做法就是：**拉起一个新的 Worker 进程来替代挂掉的那个**。



这个过程就好像餐厅里一个厨师突然肚子疼请病假回家了，经理并不会让厨房停摆，而是立刻打电话给后备厨师：“AAA 建材朱哥，赶紧过来顶班！”

通过这种方式，无论 Worker 进程因为什么原因挂掉，Master 进程总能保证服务团队的“编制”是满的，始终有足够数量的 Worker 在岗提供服务。这就大大提高了整个应用的健壮性和可用性。这种一个挂了再拉起一个新进程的模式，也叫“守护进程”。



## 本章小结
这一章，我们从一个看似简单的问题“为什么要利用多核 CPU”出发，进行了一次探索。



我们首先理解了家用 CPU 和服务器 CPU 在设计哲学上的不同，点明了在服务端编程中，充分利用多核处理并行任务的重要性。



接着，我们介绍了 Master-Worker（或称 Cluster-Worker）这一经典的多进程架构模式，解释了它的核心思想——**一个管理者，多个工作者**。



一头扎进了 Node.js 的 `child_process` 模块的源码世界，揭示了 `spawn`、`exec` 等我们常用 API 的底层关系，最终都殊途同归于 `uv_spawn()`。我们还用“影分身”和“夺舍”拆解了 `fork()` 和 `execvp()` 这两个核心的系统调用，说明了 Node.js 启动一个子进程的完整流程。



为了更方便地构建多进程网络服务，我们引出了 Node.js 的官方“包工头”——`cluster` 模块。我们探讨了它是如何通过“轮询”或“句柄传递”这两种魔法，巧妙地解决了多个进程监听同一个端口的难题。



Node.js 虽然核心是单线程的，但它通过 `child_process` 和 `cluster`，为我们提供了充分利用多核 CPU、构建高性能、高健壮性应用的强大能力。
