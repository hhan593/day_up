**可视化是通过图形、图像、图表等`视觉元素`来呈现`数据`、`信息`或`概念`的过程**。它将抽象的、复杂的或大量的数据转化为可视化形式，以便人们能够更直观、更容易地理解和分析。

在我们平常的开发中，对于可视化的需求最常见的莫过于图表、动画效果；在各大活动节日中，我们也常常有抽奖需求、H5 小游戏的需求，等等。这都需要身为前端工程师的我们去做对应的技术选型、技术实践和沉淀。如果没有可视化的体系认知，对于显示层我们看到的只是 UI 效果，更何况大部分人对三维的认知也并没有。

因此，我想梳理出一条清晰的`学习路径`，来帮助大家从繁杂的技术中破开迷雾，找到属于自己的捷径。

## HTML + CSS

**`HTML`（HyperText Markup Language）是用于创建网页的标记语言**。HTML 文档由一系列标签组成，这些标签定义了网页的结构和内容。HTML 标签可以用来创建文本、图像、表格、链接等各种元素。

**`CSS`（Cascading Style Sheets）是用于描述网页样式的语言**。它可以用来定义网页的字体、颜色、布局、背景等。CSS 是 HTML 的补充，它可以让网页更加美观和易于阅读。

直到 HTML5 标准的发展，让我们拥有了包括视频、音频、图形、动画等高级特性，而 CSS 需要适应这些新的标签，为它们提供新的样式特性，比如过渡、变形、动画、函数、Houdini、过滤器等。

## SVG

**`SVG`（Scalable Vector Graphics）是一种基于 XML 的图形格式，用于描述二维矢量图形**。它是一种开放标准，由 W3C（World Wide Web Consortium）制定和维护。

与位图图形（如 JPEG、PNG）不同，SVG 使用数学公式和几何描述来定义图形，因此**它是矢量图形，可以无损地缩放和变换而不失真**。这使得 SVG 在各种分辨率和屏幕尺寸下都能保持清晰和精确。

使用 SVG，我们可以定义各种图形元素，如路径、线条、矩形、圆形、椭圆、多边形等，并对这些元素应用样式、动画和交互效果。SVG 还支持滤镜、渐变、剪切、遮罩等高级特性。

接下来我们看看用 SVG 标签定义的图表案例：

```js

    <rect x="190" y="60" width="50" height="240" style="fill: rgb(134, 226, 134);"></rect>

    <!-- 添加坐标轴和标签 -->
    <line x1="30" y1="300" x2="350" y2="300" style="stroke: black;"></line>

    <text x="0" y="30" style="font-size: 12px;">200</text>
    <text x="10" y="300" style="font-size: 12px;">0</text>

```

但是同时也伴随了一个新的问题，它的渲染过程跟 DOM 一样也要渲染树的生成、布局合成、绘制，这也就意味着大批量的图形对于 SVG 来说很容易造成卡顿。

## Canvas

**`Canvas` 是 HTML5 中的一个元素，它提供了一个用于绘制图形、动画和图像的空白画布**。Canvas 可以通过 JavaScript 来操作，使开发者可以在网页上实时绘制和渲染图形。

Canvas 在游戏开发、数据可视化、图表绘制等领域具有广泛的应用。它提供了灵活、实时的图形绘制能力，使开发者能够创建交互性强、视觉效果丰富的网页内容。

Canvas 相比 SVG 在性能上有一些优势的原因：

1. **绘图方式**：Canvas 使用基于像素的绘图方式，而 SVG 使用基于矢量的绘图方式。在 Canvas 中，图形被转换为像素，并直接绘制在画布上。而在 SVG 中，图形是通过数学公式和几何描述来定义的，需要在每次渲染时重新计算和绘制。这使得 Canvas 在处理大量图形元素时更高效。

2. **渲染方式**：Canvas 绘制后的图形是静态的位图，不会保留图形的结构信息。而 SVG 保留了图形的结构和属性信息，因此在每次渲染时都需要解析和计算这些信息。这使得 Canvas 在动态更新和频繁重绘时更快。

3. **复杂性**：由于 SVG 保留了图形的结构信息，它可以更容易地进行交互和修改。但这也增加了渲染的复杂性和计算量。相比之下，Canvas 简单地将图形绘制在画布上，不需要处理图形的结构信息，因此更高效。


Canvas 相比 SVG 在性能上更好，主要是因为绘图方式和渲染方式的差异。

此外，在 H5 游戏应用上我们绝对会采用 Canvas 做游戏方面的开发，而不是 SVG，主要原因在于性能和游戏引擎的支持，比如 Phaser、Pixi.js、CreateJS、Cocos2d-js 等，本小册也会从 Phaser 讲解游戏开发。

Canvas 虽好但是也有问题，比如我们无法像 DOM 一样快速绑定事件、绘制方式是实时绘制模式不好上手、位图一样存在失真问题。

但总的来说，**大批量的图形绘制我们更推荐用 Canvas，图形数量少的时候我们可以用 SVG**。

说了这么多，我们在实际项目中图表大概率会用现成的库来实现，比如大家熟知的 ECharts 图表库，它的底层用了 ZRender 做图形渲染，支持 Canvas 和 SVG 的渲染引擎切换。

## WebGL

<p align=center><img src="https://p1-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/274ed6f99f8d499f8971227c95e6c32e~tplv-k3u1fbpfcp-jj-mark:0:0:0:0:q75.image#?w=780&h=212&s=30744&e=png&b=7a1000" alt="image.png"  /></p>

**`WebGL`（Web Graphics Library） 的诞生是为了在 Web 浏览器中实现高性能的 3D 图形渲染**。它是基于 OpenGL ES（OpenGL for Embedded Systems）标准的一个 JavaScript API。

WebGL 的出现有以下原因：

1. **提供更丰富的图形效果**：传统的 HTML 和 CSS 对于复杂的 3D 图形渲染和动画效果支持有限，WebGL 通过利用计算机的图形处理单元（GPU）来实现高性能的 3D 图形渲染，可以在 Web 浏览器中呈现更逼真的图形效果。

2. **跨平台和跨设备**：WebGL 可以在各种设备和平台上运行，包括桌面电脑、移动设备和嵌入式系统。它不依赖于特定的操作系统或硬件，使得开发者可以创建一次性的 Web 应用程序，无需为不同平台编写不同的代码。

3. **提供更好的性能**：WebGL 利用 GPU 的并行处理能力，能够在 Web 浏览器中实现更高效的图形渲染。这使得开发者可以创建更复杂、更流畅的交互式 3D 应用程序和游戏。

4. **开放标准和广泛支持**：WebGL 是一个开放标准，由 Khronos Group 开发和维护。它得到了各大浏览器厂商的支持，包括 Chrome、Firefox、Safari 和 Edge 等。这意味着开发者可以在各种现代 Web 浏览器上使用 WebGL，而无需担心兼容性问题。

WebGL 的诞生是为了在 Web 浏览器中实现高性能的 3D 图形渲染，提供更丰富的图形效果和交互性。它的出现使得开发者可以在 Web 平台上创建更具吸引力和创新性的应用程序和游戏。

而操作 WebGL 的三维图形 API 本质上其实就是处理着色器 Shader。

`Shader（着色器）`是计算机图形学中用于描述光照、材质和渲染效果的程序。它是一种在图形渲染管线中执行的小型程序，用于控制图形的绘制和渲染过程。

Shader 着色器通常由两个主要部分组成：顶点着色器（Vertex Shader）和片段着色器（Fragment Shader）。

- **顶点着色器**是在图形渲染管线的顶点处理阶段执行的程序。它接收输入的顶点数据，并对每个顶点进行一系列的计算和变换，如位置变换、法线变换、纹理坐标变换等并输出。

- **片段着色器**是在图形渲染管线的片段处理阶段执行的程序。它接收顶点着色器传递过来的数据，并对每个片段（像素）进行计算。片段着色器通常用于计算光照、纹理采样、颜色插值等操作，并最终确定片段的最终颜色。也就是屏幕的显示效果是它来决定的。

Shader 的上手门槛很高，但是 Shader 的优势是可以让我们用几 kb 来实现通过几百 M 搭建的建模场景。在 ShaderToy 中，我们可以看到这些令人惊叹的效果：

在游戏领域比如 unity、unreal 主流游戏引擎，我们也能看到非常多的 Shader 应用，比如游戏角色的技能特效、材质、游戏场景、转场动画等。

总而言之，学好 Shader 我们可以实现任意效果。本小册也会带领大家深入学习 Shader 的编写。

## Three.js


<p align=center><img src="https://p3-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/4fff3e4ec5a64f2994473e75bbea3acf~tplv-k3u1fbpfcp-jj-mark:0:0:0:0:q75.image#?w=461&h=229&s=53947&e=png&b=fefefe" alt="image.png"  /></p>

那么对于三维应用我们可以直接用基于 WebGL 封装的 Three.js，而 Three 却在内部慢慢支持 WebGPU 渲染，WebGL 转变为 WebGPU 这是为什么呢？因为 WebGL 在时代的发展中以后会慢慢被淘汰。

WebGL 是一个较为古老的技术，它的标准是基于 OpenGL 继承而来的。OpenGL 的历史可以追溯到 1992 年，而这些早期的设计理念与现代 GPU 的工作原理并不完全匹配。对于浏览器开发者来说，不同的 GPU 新特性的适配是很麻烦的事情。

2014 年，苹果发布了 Metal 图形框架。随后苹果放弃了 OpenGL ES。

微软在 2015 年发布了自己的 D3D12（Direct3D 12）图形框架。紧随其后的是 Khronos Group，这是图形界的国际组织，类似于前端的 W3C、TC39。而 WebGL 是该组织的标准。然而，随着时间的推移，WebGL 逐渐淡化，取而代之的是对现代图形框架 Vulkan 的支持。

迄今为止，Metal、D3D12 和 Vulkan 被视为现代的三大图形框架。这些框架充分发挥了 GPU 的可编程能力，使开发者能够更自由地控制 GPU。

同样重要的是要注意，当今的主流操作系统不再将 OpenGL 作为主要图形支持。这意味着编写的每一行 WebGL 代码都有 90% 的机会不会由 OpenGL 进行绘制。在 Windows 上使用 DirectX 进行绘制，在 Mac 上使用 Metal 进行绘制。

所以，WebGPU 诞生了……



## WebGPU

<p align=center><img src="https://p9-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/cee02fe07fb846cbb4ca03d20969260b~tplv-k3u1fbpfcp-jj-mark:0:0:0:0:q75.image#?w=986&h=271&s=77413&e=png&b=ffffff" alt="image.png"  /></p>

**`WebGPU` 是一种新的 Web 标准，用于在 Web 浏览器中进行高性能图形渲染和计算**。它旨在提供比 WebGL 更底层、更直接的硬件访问，并且更好地与现代图形 API（如 Vulkan、Metal 和 Direct3D 12）对接。

WebGPU 的设计目标是提供一个**跨平台、高性能**的图形和计算编程接口，使开发者能够更好地利用 GPU 的计算能力。它为开发者提供了更多的控制权和灵活性，以实现更高效、更复杂的图形渲染和计算任务。

WebGPU 的特点包括：

1. **低级别的硬件访问**：WebGPU 提供了更底层的硬件访问接口，使开发者能够更直接地控制 GPU 的功能和性能。

2. **线程安全**：WebGPU 支持多线程操作，使开发者能够更好地利用多核 CPU 和多个 GPU。

3. **强大的计算能力**：WebGPU 提供了强大的计算能力，使开发者能够在浏览器中进行复杂的计算任务，如图像处理、物理模拟等。

4. **跨平台支持**：WebGPU 的设计目标是跨平台的，可以在不同的操作系统和设备上运行，包括桌面、移动设备和虚拟/增强现实设备。

WebGPU 的发展是由 Khronos Group（图形界的国际组织）主导的，该组织也负责制定其他图形 API 标准，如 OpenGL、Vulkan 等。目前，WebGPU 正在积极开发中，并已经在一些浏览器中进行了初步实现和测试。预计在未来几年内，WebGPU 将成为 Web 开发中重要的图形和计算编程接口之一。
