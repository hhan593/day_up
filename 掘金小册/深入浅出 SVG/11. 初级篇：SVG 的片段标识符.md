在现代 Web 开发中，SVG 已经成为设计和实现各种图形和图标的首选格式之一。而在利用 SVG 创建 Web 图形时，掌握 SVG 片段标识符（SVG Fragment Identifiers）是至关重要的。SVG 片段标识符是一种特殊的选择器，它允许我们在 SVG 图像中定位和提取特定的片段，而无需加载整个图像文件。这为我们在 Web 开发中使用 SVG 图形提供了更多的可能性，无论是创建复杂的动画效果还是实现交互式图形元素，都可以借助 SVG 片段标识符轻松实现。

  


在这节课中，我们将学习如何使用 SVG 片段标识符来裁剪、定位和呈现 SVG 图像的特定部分。我们将探讨 SVG 片段标识符的语法和用法，并通过实际示例演示如何在 Web 中应用这些技术。

  


无论你是初学者还是有一定经验的 Web 开发者，这节课都将为你提供全面的指导和实用的技巧，帮助你更好地利用 SVG 片段标识符来实现你的 Web 设计和开发目标。让我们一起深入了解SVG片段标识符，并探索其在现代Web开发中的无限可能性！

  


## SVG 片段标识符概述

  


SVG 片段标识符（SVG Fragment Identifiers）是 [SVG 2.0 规范](https://www.w3.org/TR/SVG2/linking.html#SVGFragmentIdentifiers)中引入的一个重要概念，它是一种在 SVG 图像中定位和提取特定部分的方法。它们是一种特殊的选择器，允许 Web 开发者通过 URL 的哈希部分来提取 SVG 文档的特定部分，而无需加载整个 SVG 文件。使用 SVG 片段标识符，Web 开发者可以精确地定位和提取 SVG 图像中的特定元素、路径或视图，并将其应用于 Web 设计和开发中的各种场景。

  


在 SVG 文档中，每个元素、路径或视图都可以通过唯一的 `ID` 进行标识。通过在 SVG 文件的 URL 后面添加 `#` 符号和元素的 `ID` ，就可以使用 SVG 片段标识符来指定并提取该元素。例如，可以使用以下语法来指定 SVG 文档中 `ID` 为 `example` 的元素：

  


```XML
 <svg xmlns="http://www.w3.org/2000/svg">
    <circle id="example" cx="50" cy="50" r="40" fill="red" />
</svg>
```

  


然后，可以通过将 `#example` 添加到 SVG 文件的 URL 末尾来引用此元素：

  


```HTML
<img src="svgfile.svg#example" alt="" />
```

  


这样就可以单独引用 SVG 文件中的特定元素，而不需要加载整个SVG文档。这为 Web 设计和开发提供了更灵活和高效的方式来处理 SVG 图像，并支持动态加载、交互和模块化。

  


SVG 片段标识符具有多种作用和用途，主要体现在以下几个方面：

  


-   **定位和提取特定元素或路径**：SVG 片段标识符允许 Web 开发者通过 URL 指定 SVG 文档中的特定元素或路径，并在页面中使用这些元素或路径。这对于在网页设计和开发中需要使用 SVG 图像的特定部分时非常有用。
-   **创建 SVG 图像的精细控制**：通过使用 SVG 片段标识符，开发者可以更精确地控制 SVG 图像的呈现方式。例如，可以针对不同的 SVG 元素应用不同的样式或动画效果，从而实现更加丰富和多样化的视觉效果。
-   **减少网络请求和提高性能**：使用 SVG 片段标识符可以避免加载整个 SVG 文件，而仅加载所需的部分。这有助于减少页面加载时间并节省带宽，提高网站的性能和响应速度。
-   **实现模块化和重用**：将 SVG 图像分割为多个可重用的部分，并使用片段标识符将它们组合在一起，可以实现 SVG 图像的模块化和重用。这样一来，开发者可以更轻松地管理和维护 SVG 图像，并在不同的项目中重复使用它们。
-   **支持动态加载和交互**：通过 JavaScript 等脚本语言，可以动态地修改 SVG 片段标识符指定的元素或路径，实现图像的动态加载和交互效果。这为创建交互式的 SVG 图像提供了强大的支持。

  


总的来说，SVG 片段标识符是一种强大而灵活的工具，可用于在网页设计和开发中实现更加精细和优化的 SVG 图像处理，提高用户体验和页面性能。

注意，`icons.svg` 文件中应用了一个我们不太熟悉的 SVG 元素，即 `<view>` 元素。它用于指定 SVG 图形的可视区域，允许你定义 SVG 文件中的特定区域，并将其作为视口进行显示。其语法如下：

  


```XML
<view id="identifier" viewBox="min-x min-y width height" />
```

  


-   `id` 属性：定义视图的唯一标识符
-   `viewBox` 属性：指定了视图框的位置和尺寸。它是一个以空格分隔的四个值的列表，分别代表了视图框的左上角的 `x` 坐标和 `y` 坐标，以及视图框的宽度（`width`）和高度（`height`）

  


使用 `<view>` 元素，你可以将 SVG 图形分割成多个部分，并在需要时选择性地显示这些部分。这为 SVG 图形的重用和定位提供了灵活性。

  


```XML
<view id="codepen-view" viewBox="0 0 32 32"/>
<view id="twitter-view" viewBox="0 32 32 32"/>
<view id="github-view" viewBox="0 64 32 32"/>
<view id="dribbble-view" viewBox="0 96 32 32"/>
<view id="vimeo-view" viewBox="0 128 32 32"/>
```

## 案例：SVG 雪碧图

  


![](https://p3-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/680e82660d5d4f7fbbb96532627b25aa~tplv-k3u1fbpfcp-jj-mark:0:0:0:0:q75.image#?w=1332&h=750&s=207903&e=jpg&b=003156)

  


SVG 雪碧图（又称 SVG 精灵）技术在 Web 开发领域并不陌生，通常被视为图标和其他矢量图形的首选技术。然而，今天我们将有机会重新审视 SVG 雪碧图技术，探索其中一些不太为人所知的选项，并探讨它们可能的用途。

  


众所周知，雪碧图技术出现以及得到广泛使用，主要有几个原因：

  


-   **减少 HTTP 请求**：将多个图像合并到一个雪碧图中，可以减少页面加载时的 HTTP 请求次数。这对于提高网页加载速度和性能非常重要，尤其是在移动设备和低带宽环境下
-   **提高性能**：减少 HTTP 请求次数有助于降低网站的加载时间，从而提高用户体验。一次加载一个大的雪碧图比单独加载多个小图像要快得多
-   **优化带宽利用**：合并多个图像到一个雪碧图中可以减少网站的带宽使用量，从而降低服务器负载和运营成本
-   **缓存利用率高**：由于雪碧图是单个文件，浏览器可以更有效地缓存它，从而提高了网页的加载速度
-   **适应性强**：雪碧图可以包含不同大小的图像，从小图标到背景图像都可以合并在同一个文件中
-   **方便管理和维护**：将所有图像合并到一个雪碧图中可以使管理和维护变得更加简单。通过这种方式，可以轻松地添加、删除或更新图像，而不必修改大量的 HTML 或 CSS 代码

  


先来看一个未使用雪碧图技术的小例子：

  


![](https://p3-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/1de308ec523247409b49d510767fb872~tplv-k3u1fbpfcp-jj-mark:0:0:0:0:q75.image#?w=2000&h=1060&s=258853&e=jpg&b=f5f5f5)

  


假设 Web 应用上有一个“设置”的图标，默认下是黑色的，用户鼠标悬浮到该图标上时，图标会变成一个紫色：

  


```CSS
/* Default */
.icon-setting {
    background-image: url("./path/link.svg");
    
    &:hover,
    &:focus {
        background-image: url("./path/hover.svg");
    }
```

![](https://p3-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/fd2edd09445d4923845c4865fc5ed772~tplv-k3u1fbpfcp-jj-mark:0:0:0:0:q75.image#?w=1166&h=558&s=221686&e=gif&f=129&b=c1e97d)

  


不难发现，用户鼠标第一次悬浮到图标上时，它会闪烁一下，尤其是在网络较慢的情况之下。而且，在这种方式之下，浏览器是不会预加载悬浮状态下需要的图标资源（`hover.svg`）。你可以打开浏览器开发者调试工具验证这一点：

  


![](https://p3-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/83969aa5f4014640bd0659c97308ea9f~tplv-k3u1fbpfcp-jj-mark:0:0:0:0:q75.image#?w=1228&h=512&s=801854&e=gif&f=104&b=1d2325)

  


如果我们使用雪碧图技术，就可以很好的避免这个现象。

  


众所周知，随着 Web 的发展，Web 开发相关的技术也在不断演进，其中雪碧图技术也不例外。为了让大家更全面地了解这项技术，让我们从雪碧图技术的最初开始讨论。

  


最初的雪碧图通常是将多张位图拼接在一张图中，后面演变出用于矢量图的 SVG 雪碧图。例如，上面示例中链接的两个状态应用了单独的 SVG 文件（默认是 `link.svg` ，黑色的“设置”图标；悬浮状态是 `hover.svg` ，紫色的“设置”图标），它们仅仅是颜色上的差异。现在，我们可以在同一个 SVG 文件中将这两个状态的图标放在一起：

  



```CSS
.icon--setting {
    background: url("./path/srpites.svg") no-repeat 0 0 / cover;
    
    &:hover,
    &:focus {
        background-size: -200px 0;
    }
}
```

  


与位图雪碧图相比，SVG 雪碧图更为灵活，因为你可以通过缩放背景来显示任何大小的图标。不过，也有令人感到困惑的部分，那就是必须根据生成的图标大小和雪碧图的尺寸来计算背景大小值（`background-size`）。例如，一个 `72 x 24` 的雪碧图用于一个 `200 x 200` 的图标，那么 `background-size` 将是 `600 x 200` 。这只是简单的数学计算，但一旦雪碧图的尺寸发生变化，你可能需要重新计算，并且更新 CSS 代码。

  


不过，如果雪碧图中的图标尺寸大小一致，你只需要将 `background-size` 设置为 `cover` 即可。一旦图标被缩放，你需要调整 `background-position` 来移动背景图片的位置。

  


> 注意，上面这个示例中所演示的 SVG 雪碧图不是一个很好的方案，这里只是为了让大家能更全面的认识 SVG 雪碧图技术发展过程，才在这里向大家展示的。在实际生产中，个人强烈建议，不要这样使用！

  


另外还存在一个现象就是，大部分 Web 开发者都认为，雪碧图技术是 CSS 的 `background-image` 专有特性。事实上并非如此，[现代 CSS 已经有很多新的特性出现了](https://s.juejin.cn/ds/iFgxyhqN/)，雪碧图技术不再仅局限于此，CSS 中的其他属性也可以利用雪碧图技术，例如 [CSS 的遮罩（mask-image）](https://juejin.cn/book/7223230325122400288/section/7259668885224456252)。

  


更有趣的是，雪碧图技术也不再局限于背景图，还可以用于 `<img>` 元素：

  


```HTML

<img src="https://static.fedev.cn/icons.svg" alt="codepen" class="icon icon--vimeo"  style="--index: 4;"/>
```

  


```CSS
.icon {
    --positionBase: -200px;
    display: block;
    width: 200px;
    aspect-ratio: 1;
    object-fit: cover;
    object-position: 0 calc(var(--index) * var(--positionBase));
```

与传统的 SVG 雪碧图技术相比，使用 `<symbol>` 要容易得多。你只需要将绘制图标的元素嵌套在 `<symbol>` 元素中，并指定唯一名称的 `ID` 即可。但这种方案也存在一些缺点。例如，如果想通过 CSS 来修改图标的颜色，那么在使用 `<symbol>` 定义图标时，就需要使用 CSS 自定义属性作为 `fill`、`stroke` 等属性的值。只有这样，才能通过 CSS 来穿透 `<use>` 元素，实现高度定制化的图标，从而使 SVG 雪碧图变得更加灵活、更加强大。你可以返回阅读上一节课的内容，以进一步了解这方面的内容。

  


另一个缺点是，`<symbol>` 无法将这样的图标应用于 `background-image` 或 `<img>`，而只能用于内联 SVG 的 `<use>`。

## 你真的需要 SVG 片段标识符？

  


SVG 雪碧图的应用案例再次提醒我们：SVG 片段标识符是众多 SVG 雪碧图方案中的一种，甚至可以说是一种聪明有效的解决方案。事实上，SVG 片段标识符的存在，大多数时候都是为了服务于 SVG 雪碧图。

  


雪碧图技术是 Web 性能优化的古老技巧之一。与服务多个单独的图标相比，服务一个大的雪碧图有许多好处，对吧！但在今天，它是否实际上表现得更好呢？我期望如此，而在某些情况下确实如此。这也是 SVG 片段标识符继续服务于 SVG 雪碧图的原因之一，也是雪碧图技巧继续为性能优化服务的原因之一。

  


然而，在实践中，对于大多数用例来说，雪碧图的好处是有限的。以下是一些原因：

  


-   HTTP/2 的多路复用允许在单个 TCP 连接上请求多个资源，这意味着最多六个并发 HTTP 请求的限制不再是问题。尽管如此，如果文件被捆绑，压缩效果会更好。但大多数应用程序和网站在网页上不会加载那么多的图标。
-   减少几千字节和请求在整体方案中并不会产生可衡量的影响。例如，我刚访问了 nytimes.com，即使是移动版本也下载了超过 7MB 的资源。在大多数情况下，通过 HTTP 加载几个 SVG 图标对性能的影响微乎其微。这不仅限于 nytimes.com，对于大多数应用程序来说，性能的决定因素是重资源，比如嵌入、JavaScript 和大图像，以及页面加载的整体方法，而不是小图标。
-   请求的数量并不像阻塞请求那样重要。并非所有请求都是平等的。例如，在头部加载的 CSS 文件将阻塞渲染，直到所有文件都加载完成，并且会对网站的性能产生可测量的影响。但对于异步加载的不阻塞渲染的微小请求，这个说法就不成立了。
-   有效的缓存头减少了后续页面加载的许多开销。如果资源以有效的缓存头提供，浏览器可以缓存大部分请求以供后续浏览使用。

  


这也意味着请求的数量并没有我们想象的那样影响用户体验。例如，即使加载了 10 多个图标，几乎所有图标都同时出现。即使在较慢的连接上，对性能的影响也不太明显。

实际上，SVG 片段标识符可能会对性能产生影响，因为需要下载整个捆绑包，而不仅仅是一个小图标。对于大多数情况来说，它们似乎是一个不必要的选择。当然，它可能有其存在的价值，但我希望避免开发人员在默认情况下就可以解决大多数问题时不得不做出不必要的选择。
