# 微信小程序中 ECharts 图表不展示问题总结

> 本文聚焦 **微信小程序环境** 下使用 **ECharts / l-echart** 时，常见的「**数据已更新但图表不展示 / 不刷新**」问题，**不涉及具体业务逻辑**，仅讨论平台机制、根因与通用解决方案。

---

## 一、问题背景

在实际开发中，经常会遇到如下场景：

- 接口数据已成功返回
    
- Vue 的 `watch / computed` 正常触发
    
- `series / xAxis` 数据已经发生变化
    

但 **图表却：**

- 完全不更新
    
- 仍显示旧数据
    
- 或直接空白

控制台可能出现类似报错：

```
chart update skipped
TypeError: Cannot read property 'width' of undefined
```

在 **H5 正常、小程序异常** 的情况下，该问题尤为常见。

---

## 二、典型错误现象

### 1. 图表无任何变化

- 数据已更新
    
- `setOption` 被调用
    
- 但画面无变化
    

### 2. 控制台关键报错

```
TypeError: Cannot read property 'width' of undefined
```

该报错通常出现在：

- `chart.setOption()`
    
- `chart.resize()`
    
- ECharts 内部重绘阶段
    

---

## 三、问题本质（核心原因）

### 1️⃣ 小程序 Canvas 的生命周期特性

微信小程序中，ECharts 是基于 **Canvas** 渲染的，而 Canvas 具有以下特性：

- **Canvas 不是稳定节点**
    
- 会因组件结构变化被销毁并重建
    

常见触发条件包括：

- `v-if / v-else` 条件切换
    
- 图表容器高度变化（legend、unit、dataZoom 显隐）
    
- `xAxis` 数量变化
    
- `series` 数量变化
    
- 数据短暂变为空数组再恢复
    

> ⚠️ Canvas 一旦重建，**原来的 ECharts 实例就失效了**。

---

### 2️⃣ 失效实例仍被调用（关键问题）

常见逻辑如下：

```ts
chartInstance = echarts.init(canvas)
```

当 Canvas 被销毁后：

- `chartInstance` 仍然存在
    
- 但其内部 `canvas` 已是 `undefined`
    

于是再调用：

```ts
chartInstance.setOption(...)
```

就会导致：

```
Cannot read property 'width' of undefined
```

这正是 `chart update skipped` 的真实含义。

---

## 四、为什么 setOption 在小程序中不可靠？

|场景|H5|小程序|
|---|---|---|
|DOM 稳定性|高|低|
|Canvas 生命周期|稳定|易被重建|
|setOption 容错|高|低|

**结论：**

> 同一套代码，在 H5 可行，但在小程序中会因 Canvas 重建而失效。

---

## 五、什么是「结构级变化」？

以下变化，均属于 **结构级变化**：

- 年 / 月 / 日等维度切换
    
- xAxis 类目数量明显变化
    
- bar / line / mix 类型切换
    
- legend 显示状态变化
    
- dataZoom 显示 / 隐藏
    
- yAxis 数量变化
    

⚠️ **结构级变化 = 不能只靠 setOption**

---

## 六、唯一稳定的解决方案

### ✅ 核心原则

> **当图表结构发生变化时，必须销毁并重建 ECharts 实例。**

---

### ✅ 标准流程

```
dispose 旧实例
↓
等待 nextTick（确保新 canvas 已渲染）
↓
重新 init ECharts
↓
setOption
```

---

### ✅ 通用示例代码

```ts
const rebuildChart = async () => {
  await nextTick();

  if (chartInstance) {
    chartInstance.dispose();
    chartInstance = null;
  }

  initChart(); // 重新绑定新的 canvas
};
```

```ts
watch(
  [() => seriesData, () => xAxisData],
  () => {
    if (!seriesData.length || !xAxisData.length) return;
    rebuildChart();
  },
  { deep: true }
);
```

---

## 七、什么时候可以只用 setOption？

以下场景 **setOption 是安全的**：

- 数据值变化，但结构不变
    
- 类目数量不变
    
- 仅数值刷新（如定时轮询）
    
- H5 环境
    

---

## 八、快速判断清单（实战口诀）

满足任意一条，**直接重建图表**：

- 图表维度切换（年 / 月 / 日）
    
- xAxis / series 数量变化
    
- legend / dataZoom 显隐变化
    
- 报错包含：
    
    ```
    width of undefined
    chart update skipped
    ```
    

口诀总结：

> **小程序里，结构一变，实例必废；**  
> **图不更新，先别 setOption，直接重建。**

---

## 九、结语

该问题并非 Vue 响应式失效，也不是 ECharts 使用错误，而是：

> **微信小程序 Canvas 生命周期 + ECharts 实例管理不匹配导致的系统级问题。**

一旦理解这一点，类似问题都可以快速定位并解决。

---

> 📌 建议：在小程序项目中，为图表封装统一的「**安全重建机制**」，避免在业务层反复踩坑。
