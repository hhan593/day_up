# IR20251107008 - 掌上信泰配合双录环节"确认投保"流程改造

## 一、需求概要

**核心目标**：将掌上信泰的投保流程与双录流程拆分，提高容错率。

- **原流程**：投保和双录合并进行 → 任何错误都导致投保作废，需重新投保+双录
- **新流程**：先独立完成双录 → 确认无误后再提交投保 → 查看结果

---

## 二、编写思路

### 1. 新增"已录制"菜单（保单中心-待双录）

**改动点**：

- 在"待双录"菜单下新增"已录制"Tab/子菜单
- 查询接口：获取当前业务员下，双录已完成但尚未成功投保的保单列表
- 列表展示保单基本信息

### 2. "提交投保"功能

**改动点**：

- 在"已录制"列表中的保单卡片增加"提交投保"按钮
- **按钮展示条件**：仅在双录录制**当天**可展示和点击，过期则隐藏
- 点击后进入**只读签字页面**：
  - 展示该保单的空白单证
  - 回显最新双录时的签名（样式与现有签名提交页一致）
  - 页面不可编辑，仅可获取验证码 → 点击"确认投保"
- 调用掌上信泰承保API接口（ST000022），提交投保请求

### 3. 投保失败处理 — "一键重投"

**改动点**：

- 当投保被规则校验拒绝时，在保单信息中展示**投保失败原因**（ST000022接口返回结果）
- 增加"一键重投"按钮，跳转至"保单中心-待提交-一键重投"列表
- "待双录-已录制"与"待提交-一键重投"**数据双向同步**：一边操作完，另一边同步更新

### 4. "保单中心-待提交"页面适配

**改动点**：

- 待提交页面中，已录制结束但未成功投保的保单同样可点击"提交投保"
- 进入只读签字页面，逻辑与上述一致

### 5. 关键接口调用

| 接口 | 用途 |
|------|------|
| 保单列表查询接口 | 查询"已录制"状态的保单 |
| ST000022（承保API） | 提交投保、获取投保结果/失败原因 |
| 签名回显接口 | 获取最新双录时的签名数据 |
| 单证展示接口 | 获取保单所需空白单证 |

### 6. 前端页面结构

```
保单中心
├── 待双录
│   ├── 未推送
│   ├── 已推送
│   └── 已录制（新增）
│       ├── 保单列表
│       ├── 提交投保按钮（当天有效）
│       └── 一键重投按钮（规则校验失败时）
├── 待提交
│   └── 提交投保 / 一键重投
└── ...
```

## 四、核心关注点

1. **当天时效性** — "提交投保"按钮仅当天有效，零点边界测试很关键
2. **数据双向同步** — "已录制"和"待提交"两个入口操作需保持数据一致
3. **ST000022接口** — 承保API的成功/失败/超时各种场景都需覆盖
4. **只读页面** — 确保签字页面严格不可编辑，防止数据篡改
5. **签名回显** — 必须是最新一次双录时的签名，非历史签名
